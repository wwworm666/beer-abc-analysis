## 🔑 КЛЮЧЕВЫЕ НАБЛЮДЕНИЯ

### Наблюдение 1: Разные коды ошибок
| Код ошибки | Сообщение | Значение |
|------------|-----------|----------|
| 403 + код 4 | "Подпись невалидна" | Формат подписи неверный |
| 403 + код 2 | "Проверка подписи не пройдена" | **Подпись валидна, но не подходит для данных** |

**Вывод:** Подпись создаётся технически правильно, но ЧЗ API ожидает **другие данные для подписи**.

### Наблюдение 2: Требования документации v3
Из docs.crp333t.ru.html (строка 445):
> **data** — Подписанные УКЭП зарегистрированного УОТ случайные данные в base64 (**присоединённая электронная подпись**)

**НО!** Есть уточнение:
> "При `unitedToken` = `true` передаётся случайный набор данных, **либо ИНН организации для авторизации с МЧД**"

### Наблюдение 3: Формат запроса v3
Из документации (docs.crp333t.ru.html, строки 435-438):
```json
{
   "data": "string",
   "unitedToken": true
}
```

**Обрати внимание:** В формате v3 с `unitedToken=true` параметр `uuid` **НЕ ТРЕБУЕТСЯ**!

## 🎯 ВЕРОЯТНАЯ ПРИЧИНА ПРОБЛЕМЫ

### Гипотеза: Данные для подписи должны быть в формате API

Из документации следует что для v3 API с `unitedToken=true`:
- **Либо** подписывается строка `data` из `/auth/key` (как делаем сейчас)
- **Либо** подписывается **ИНН организации** (для авторизации с МЧД)

**Проблема:** Мы подписываем строку `data` (30 символов), но возможно ЧЗ ожидает что при `unitedToken=true` нужно подписывать **ИНН**.

## ✅ ПЛАН ПРОВЕРКИ (3 ВАРИАНТА)

### Вариант A: Подписать INN вместо data
```json
{
   "data": "<подписанный ИНН: 7801630649>",
   "unitedToken": true
}
```

### Вариант B: Использовать формат v2 (с uuid)
```json
{
   "uuid": "<uuid из /auth/key>",
   "data": "<подписанная строка data>",
   "inn": "7801630649"
}
```

### Вариант C: Проверить кодировку данных
Возможно данные нужно подписывать в кодировке **Windows-1251** вместо UTF-8.
