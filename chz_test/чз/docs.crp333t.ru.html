1. Назначение и условия применения API Системы прослеживаемости табачного сырья и НМПТ

Настоящий документ содержит описание API Системы прослеживаемости табачного сырья и НМПТ (далее – Система) и предназначен для разработчиков программного обеспечения, выполняющего интеграцию с Системой.

1.1. Функции, реализуемые API участника оборота табачного сырья

API участника оборота табачного сырья содержит методы и типы данных, которые обеспечивают следующую функциональность взаимодействия:

  • Аутентификация пользователя.

  • Создание документов для подачи сведений в Системе:

    • постановка на учёт остатков;

    • постановка на учёт из производства;

    • постановка на учёт товара, полученного из стран ЕАЭС;

    • постановка на учёт товара, полученного из стран, не входящих в ЕАЭС;

    • постановка на учёт при переходе на новый классификатор;

    • списание с учёта в производство;

    • списание с учёта при экспорте в ЕАЭС;

    • списание с учёта при экспорте в другие страны, не входящие в ЕАЭС;

    • списание с учёта при продаже товара в Магазине Беспошлинной Торговли (далее – МБТ);

    • списание с учёта по разным причинам: Брак; Истек срок годности; Отзыв с рынка; Недостача; Порча; Другие причины;

    • списание с учёта при переходе на новый классификатор;

    • отмена документа (Аннулирование);

    • распаковка;

    • уведомление об остатках сырья на конец месяца.

  • Получение информации о документах.

  • Получение информации о состоянии виртуального склада:

    • Получение информации о содержимом склада.

    • Получение информации о детализации по конкретной позиции на складе.

    • Получение сведений о движении на складе.

1.2. Электронное взаимодействие участника оборота товаров с Системой

Взаимодействие участников оборота товаров с Системой осуществляется в электронном виде. Обмен документами на бумажном носителе не предусмотрен.

Электронное взаимодействие осуществляется по одному из двух вариантов:

  1. Предоставление сведений, то есть обмен электронными документами, на основании которых могут вноситься изменения в реестры ГИС МТ –Системы. Предоставление сведений является обязательством участников оборота товаров.

  2. Запрос сведений, то есть обмен, при котором сведения из реестров ГИС МТ – Системы предоставляются участниками оборота товаров в соответствии с установленными разрешениями на получение сведений. При этом изменения в реестры не вносятся. Запрос и последующее представление сведений из ГИС МТ – Системы инициируется участником оборота товаров по своему усмотрению и применяется, например, для проверки статусов КИ или статуса регистрации участника оборота товаров.

Электронное взаимодействие осуществляется путем обмена электронными документами в соответствии с определенным порядком. Для юридически значимых электронных документов применяется УКЭП участника оборота товаров, выпущенная в соответствии с законодательством РФ.

Допускается использование УКЭП, применяемой для автоматического создания и/или автоматической проверки электронных подписей, в сертификате ключа проверки, которой не указано в качестве владельца сертификата ключа проверки электронной подписи физическое лицо, действующее от имени участника оборота товаров (обезличенная УКЭП).

Обезличенная УКЭП применяется для электронных документов, автоматически формируемых и представляемых участником оборота товаров в ГИС МТ, за исключением заявления о регистрации участника оборота товаров, заявки на тестирование информационного взаимодействия и заявки на доступ к устройству регистрации эмиссии.

Распорядительным актом участника оборота товаров должно быть определено физическое лицо, ответственное за автоматическое создание и (или) автоматическую проверку электронной подписи в информационной системе, и лицо, ответственное за содержание информации, подписываемой данной электронной подписью.

В случае отсутствия указанного распорядительного акта лицом, ответственным за автоматическое создание и (или) автоматическую проверку электронной подписи в информационной системе, является руководитель юридического лица.

Важно
 Доступ к функциональности ГИС МТ осуществляется в соответствии с ролевой моделью. Информацию об ограничениях прав и условия применения обезличенной УКЭП см. в документе Памятка по ролевой модели доступа Системы маркировки.

Электронное взаимодействие осуществляется с применением программных интерфейсов взаимодействия (API) Системы и информационных систем участника оборота товаров.

При взаимодействии через API выполняются следующие требования:

  1. Аутентификация и авторизация участника оборота товаров (определение полномочий на осуществление электронного взаимодействия) осуществляется при помощи УКЭП.

  2. При предоставлении и запросе сведений защита передаваемых данных выполняется TLS/HTTPS шифрованием (в соответствии с ГОСТ 34.12-2018 и ГОСТ 34.13-2018) канала передачи данных в адрес Системы сертифицированными РФ СКЗИ (в соответствии с ГОСТ Р 34.10-2012).

  3. Количество запросов в секунду не должно превышать 50 от одного участника оборота товаров. В случае невыполнения данного требования участник оборота товаров может быть заблокирован.

Примечание
 При выводе данных о нерезидентах вместо ИНН будет возвращаться его аналог в стране ЕАЭС.

1.5. Требования к файлам обмена

API Системы при обмене использует файлы формата JSON. Файл обмена должен быть подписан УКЭП лица, имеющего право действовать от имени УОТ, и соответствовать следующим требованиям:

  • для обмена Системы использовать JSON стандарта RFC 7159;

  • применять кодировку UTF-8;

  • в результирующем JSON может содержаться набор полей, специфичных для конкретного товара.

1.6. Экранирование

Экранирование – замена в тексте управляющих символов на соответствующие текстовые подстановки. Экранирование используется для отделения текста от кода и управляющих символов. Экранирование применяется при передаче параметров запросов API методов согласно следующим стандартам:

  • при передаче параметров в едином указателе ресурса (Uniform Resource Locator, URL), т.е. в URL запроса, должно применяться экранирование согласно стандарту RFC 3986;

  • при передаче параметров в JSON-строке тела запроса URL должно применяться экранирование согласно стандарту RFC 8259.

1.7. Тип контента, возвращаемого методами API

Методы API Системы возвращают ответы в формате JSON, т.е. MIME-type application/json.

2.3. Запрос авторизации при единой аутентификации

Метод используется для получения UUID (идентификатора текущей аутентификации) и сгенерированных случайных данных, которые в дальнейшем подписываются Участником и передаются в метод [_получение_ключа_сессии_при_единой_аутентификации]. Это происходит следующим образом:

  1. Получение пары uuid - data, где:

3. Метод подачи электронных документов в Систему (создания документов)

3.1. Описание метода подачи электронных документов

Описанный в настоящем разделе метод подачи электронных документов относится только к документам прямой подачи от участников обмена товарами в Систему.

Для всех документов метод имеет следующие общие элементы:

  • формат и параметры строки вызова;

  • параметры заголовка;

  • код и параметры положительного ответа;

  • параметры ответов с ошибкой.

Документы могут иметь следующие отличающиеся элементы:

  • параметры тела запроса;

  • коды ответов с ошибкой.

В настоящем подразделе описаны элементы метода подачи электронных документов, одинаковые для всех документов ТГ Табачное сырьё и НМТП.

3.2. Описание документов прямой подачи

3.2.1. Постановка на учёт: Производство

Документ содержит коды товаров (в т.ч. сырья), которые создают в процессе производства. Переданные в документе коды товаров будут обработаны с положительным знаком на виртуальном складе Системы.

(т.е. пришел документ в Систему 2024-04-03, если в нем указана дата "2024-03" (прошлый календарный месяц), то Система его принимает, а если в нем указана дата "2024-04" (текущий календарный месяц), то Система сформирует ошибку. Пример: при получении даты в таком формате "2024-03" считаем по месяцу крайний день и проводим операцию "2024-03-31" в 23:59:59:999 по UTC+0

7. Метод получения сведений по операциям на виртуальном складе

Важно

На методы API работы с виртуальным складом накладывается следующее ограничение по использованию:

  • Вызовы метода API возможно использовать не чаще чем 1 раз в 60 секунд

Метод возвращает список операций УОТ на складе за указанный промежуток времени.

URL: /api/v1/wh/operations

Метод: GET

Пример запроса

GET http://url_stand/api/v1/wh/operations?pgId=150
                        &participantInn=1111111111
                        &docDateFrom=2024-04-01T00:00:00.000Z
                        &docDateTo=2024-04-04T23:59:59.999Z
                        &docType=WRITE_OFF
                        &operationType=OST
                        &operationId=aea6a762-2428-49b3-bb66-aa6f99f53b84-150-1
                        &productCode=RMT00000000008
                        &productExtInfo=0500
                        &unit=112
                        &productType=5
                        &serial=4235325-235235-sdgsg23
                        &sid=1234ABCD
                        &modId= bca3efa6-664e-47e8-903e-76de57afbb33
                        &direction=ALL
                        &order=DESC
                        &limit=20
                        &pageDir=NEXT
Authorization: Bearer <ТОКЕН>
Примечание
 О токене авторизации см. [_получение_ключа_сессии_при_единой_аутентификации].

JWT

JSON Web Token – это JSON объект, который определен в открытом стандарте RFC 7519. Он считается одним из безопасных способов передачи информации между двумя участниками

URL

Uniform Resource Locator (англ.) – унифицированный указатель ресурса

UTF-8

Unicode Transformation Format 8-bit (англ.) – универсальный стандарт кодирования символов, который позволяет представлять символы из всех письменностей мира, включая латиницу, кириллицу, китайские иероглифы, и другие, используя 8 битов

UUID

Universally unique identifier (англ.) – Универсальный уникальный идентификатор участника оборота товаров. Записывается в виде последовательности шестнадцатеричных цифр в нижнем регистре, разделенных знаками минуса на несколько групп, в следующем порядке: группа из 8 цифр, три группы из 4 цифр и группа из 12 цифр, что в сумме составляет 32 цифры и представляет 128 бит. Пример UUID: a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11
