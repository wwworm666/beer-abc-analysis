# Аналитика гостей (CRM) — раздел «Гости»

## Что это

Раздел `/guests`: аналитика гостей программы лояльности по ТЗ «Модуль аналитики
гостей (CRM)». 14 отчётов: рост базы, когорты жизненного цикла, активность,
частота, retention, когортные доходы, RFM, LTV, товары, сочетаемость, точки,
карточка гостя, динамика базы, сводка.

Ключевое отличие от остальных разделов: метрики считаются не только за выбранный
период, но и за всю жизнь гостя (Lifetime). Для этого чеки гостей зеркалируются
в локальную SQLite-витрину — при просмотре обращений к iiko нет вообще.

**Временная логика (обязательная, из ТЗ):**
- **Primary Period** — неделя (ISO пн–вс) / календарный месяц (дефолт) / квартал / год;
- **YTD** — с 1 января года конца периода по конец периода;
- **Lifetime** — вся загруженная история (с 2017-12, первый чек с картой 2017-12-18).

## Как работает

### Источник данных и витрина

OLAP SALES отдаёт гостевые поля `Delivery.*` только у чеков с проведённой картой
лояльности (~43% чеков). Отдельного справочника гостей в серверном API iiko нет —
гость существует для нас, только если хоть раз купил (принято владельцем
2026-07-15: «новая карта по первой покупке»).

Помесячный ETL двумя запросами: уровень чека (7 разрезов, суммы) и уровень
позиций (блюдо в чеке). Ресинк месяца идемпотентен: `DELETE` месяца + `INSERT`
одной транзакцией, затем пересчёт денормализации гостей. Ночью пересинкается
текущий месяц; предыдущий — только в первые 2 дня месяца (стык учётных дней;
ретро-правок чеков в барах не делают). Остальные месяцы `frozen`; полная
перезаливка — `POST /api/guests/sync?force=1` (админ).

Таблицы: `guests` (гость + денормализация), `receipts` (чеки), `receipt_items`
(позиции), `guest_aliases` (карта → телефон), `sync_state` (журнал месяцев).

### Идентичность гостя (проверено на живых данных, Фаза 0 2026-07-15)

В 100% гостевых строк заполнены И номер карты, И телефон. Две популяции:
регистрация по телефону (карта = телефон с лишней ведущей 7: карта
`79XXXXXXXXX`, телефон `779XXXXXXXXX`) и пластиковая карта (карта `100XXXXX`,
телефон настоящий).

**Канонизация:** только цифры; если цифр ≥ 10 → `'7' + последние 10 цифр`;
меньше 10 (код пластика) — цифры как есть; цифр нет — строка без пробелов.
**guest_id = канонический телефон**; карта пишется псевдонимом в
`guest_aliases`. «Дата создания клиента» (`Delivery.CustomerCreatedDateTyped`)
заполнена в 100% строк — это дата регистрации; у кого пусто, регистрация
приравнивается к первой покупке (`registration_source='first_order'`, доля
отдаётся в meta каждого ответа).

### Определения (единые с /discounts и месячной витриной)

- **Заказ (чек)** = ключ `(учётный день, бар, номер чека, гость)`;
- **Визит** = уникальная пара `(гость, учётный день)`;
- **Выручка** = `DishDiscountSumInt` (что фактически заплатил гость);
- **Округления:** деньги — целые ₽, проценты — 1 знак, дни/частота — 1 знак.

### Формулы разделов (зеркало: `static/js/guests/formulas.js` и тултипы UI)

| Раздел | Формула |
|--------|---------|
| §1 Рост | Регистрации(P) = гости с датой создания в P; первые заказы(P) = гости с первым чеком в P; конверсия = зарегистрированные в P с заказом ≤ конца P / регистрации P (показывается справочно — источник видит только купивших); дни до заказа = AVG(MAX(0, первый чек − регистрация)) — MAX(0,·) гасит сдвиг учётного дня за полночь |
| §2 Когорты | Когорта = месяц регистрации, только когорты ≤ даты среза. Lifetime-доли: ≥1 / ≥2 / ≥5 чеков за жизнь; % активных = визит за 30 дн на срез |
| §3 Активность | На дату конца периода, по дням с последнего визита: Active ≤30, Sleeping 31–90, At Risk 91–180, Lost >180; население = первый заказ ≤ срез; дельта — тот же расчёт на конец предыдущего периода |
| §4 Частота | Визиты гостя за P; сегменты 1 / 2–3 / 4–8 / 9+; средняя = визиты / гости с ≥1 визитом |
| §5 Retention | Когорта = месяц первой покупки; возврат = визит в (первая покупка, +N дней], N ∈ {30,60,90,180,365}; ячейка зрелая, если конец месяца когорты + N ≤ срез, иначе «—» |
| §6 Доходы | Базис когорты переключаемый (регистрация/первая покупка); накопительно за жизнь: выручка, чеки, LTV = выручка/гостей |
| §7 RFM | Окно 365 дней на срез; R = дни с последнего визита (пороги 7/14/30/60), F = визиты окна (пороги 156/104/42/16 — ПРЕДВАРИТЕЛЬНЫЕ, утвердить по распределению), M = выручка окна; сегменты как в /discounts (CHAMPIONS/LOYAL/NEW/AT_RISK/CHURNED/POTENTIAL); CSV-экспорт |
| §8 LTV | Средний = вся выручка / все гости; YTD-вариант = выручка YTD / гости с визитом YTD; по точкам — гость закреплён за точкой первого заказа |
| §9 Товары | Топ по выручке за P; первые покупки = позиции в день первого заказа гостей с первым заказом в P (ранжир по гостям); повторные = позиции после дня первого заказа; динамика — топ-10 по месяцам за 12 мес |
| §10 Сочетаемость | Пары разных позиций одного чека; поддержка = чеки с парой / чеки с 2+ позициями; уверенность A→B = чеки с парой / чеки с A |
| §11 Точки | Первая = бар первого чека (tie-break — порядок PHYSICAL_VENUES); любимая = максимум дней-визитов; миграции = гости с 2+ барами + матрица первая→любимая |
| §12 Карточка | Все три среза: заказы/визиты/выручка/средний чек; LTV; топ-5 товаров; RFM и статус на срез; последние чеки |
| §13 Динамика | active_start = визит за 30 дн до начала месяца; new = первый заказ в месяце; reactivated = визит в месяце после паузы >30 дн; active_end = визит за последние 30 дн месяца; churned = баланс (start + new + react − end) |
| §14 Сводка | Композиция готовых функций; кэш `cached_olap` с ключом от `MAX(sync_state.synced_at)` — инвалидация после синка |

### API

`GET /guests` — страница. Отчёты (все с `?period_type=&anchor=`):
`/api/guests/summary`, `base-growth`, `base-dynamics?months=`, `activity`,
`frequency`, `cohorts/lifecycle`, `retention?basis=`, `cohorts/revenue?basis=`,
`rfm[?export=csv]`, `ltv`, `products?mode=top|first|repeat|trend`,
`product-pairs?scope=`, `venues`, `search?q=`, `guest/<id>`.
Сервис: `POST /api/guests/sync[?force=1]` (админ), `GET /api/guests/sync-status`.

### Ограничения и принятые решения (владелец, 2026-07-15)

- Гости без покупок невидимы; «регистрации» = регистрации купивших; свежие
  периоды дорастают задним числом — принято, пометка в «Как считается»;
- карты сотрудников НЕ исключаются (сотрудник = такой же гость);
- ретро-правки чеков не делаются → пересинк истории не нужен (только вручную);
- конверсия регистрация→заказ на практике ≈100% (регистрация происходит при
  первой покупке) — показывается справочно;
- фильтра по точке в v1 нет (сеть целиком), точки раскрыты в §8/§11/§14.

### Что сломается, если менять неправильно

- Канонизация номера (`normalize_guest_id`) — менять только с полным ресинком
  (`?force=1`): guest_id хранится в БД, старые и новые записи разъедутся.
- Правая граница OLAP-запросов ЭКСКЛЮЗИВНАЯ — сдвиг на «включительно» задвоит
  чеки на стыках месяцев.
- Порядок запуска: `sync_month` сначала грузит чеки, потом позиции — связка
  позиция→гость строится по чекам того же месяца.
- `frozen`-месяцы пропускаются ночным синком — сброс флага без надобности
  устроит полный прогон по iiko каждую ночь.
