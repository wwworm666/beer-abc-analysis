# Месячный отчёт (отдельная страница)

## Что это

Отдельная страница **«Месячный отчёт»** (`/monthly-report`, ссылка в сайдбаре и в нав-баре
дашборда) — детальная аналитика в **помесячной динамике за год** (12 точек по оси X),
целиком на графиках Chart.js. 16 графиков в трёх блоках: Выручка, Продуктовый разрез,
Лояльность. Раньше была вкладкой внутри дашборда; вынесена в самостоятельную страницу
2026-06-05 (свой роут + шаблон, переиспользует тот же партиал и JS-модуль).

**Легенда баров** (сверху, цвет → бар; выбора нет — показываются ВСЕ бары всегда):
Общая (оранж) / ВО (розовый) / Лиг (зелёный) / Крем (жёлтый) / Варш (синий).
- **Линейные** (одна метрика) **накладывают 4 бара** — по линии на бар, цвет = цвет бара
  (сравнение баров между собой). **«Общая» на линейных НЕ показывается** — её сумма кратно
  выше отдельных баров и сплющивала их линии у нижней границы (масштаб нечитаем). Легенда
  сверху описывает именно эти 4 линии;
- **С разбивкой по категориям** (наценка по категориям — 3 линии Розлив/Фасовка/Кухня; расход
  розлив/фасовка — 2 линии) показывают **один бар**;
- **Столбчатые** (стопкой за месяц) нельзя наложить — показывают **один бар**; внутри каждого
  сегмента вписана его **доля от суммы столбца** (%), тонкие сегменты (< 6% или < 14px) без подписи;
- У всех графиков «одного бара» и у таблицы ТОП-гостей — свой **выпадающий выбор бара**
  (`.mr-bar-select`, опции = ВСЕ 5 баров, по умолчанию первый);
- **Год** выбирается дропдауном (текущий и 2 прошлых).

Данные по каждому бару берутся из OLAP per-venue (фильтр по `Store.Name`); «Общая» — запрос
без фильтра (iiko агрегирует все точки).

Принцип переиспользования: почти все метрики берутся из уже существующих расчётов
дашборда и страниц проливов/лояльности — новый код только связывает их помесячно и
добавляет 4 недостающих разреза.

## Как работает

### Эндпоинты

```
GET /api/monthly-report?venue=<key>&years=2026,2025[&force=1]
    -> core-блок помесячно за указанные годы (YoY)

GET /api/monthly-report/loyalty?venue=<key>&year=2026[&force=1]
    -> новые гости + выручка по картам/без карт помесячно

GET /api/monthly-report/draft-liters?venue=<key>&year=2026[&force=1]
    -> литры розлива по стилям помесячно (top-8 стилей + "Прочее")

GET /api/monthly-report/top-guests?venue=<key>&year=2026&month=6[&limit=10]
    -> ТОП гостей по тратам за конкретный месяц
```

`venue` — ключ бара (`bolshoy`/`ligovskiy`/`kremenchugskaya`/`varshavskaya`) или `all`.

### Графики и формулы

**Блок 1. Выручка** (core, из `get_all_sales_report` → `calculate_metrics`):

| График | Тип | Формула |
|--------|-----|---------|
| Выручка | линия (наложение баров) | Σ `DishDiscountSumInt` |
| Выручка К/Ф/Р | stacked bar | выручка Кухня + Фасовка + Розлив |
| Наценка % | линия (наложение баров) | Σ(MarkUp×Cost) / Σ Cost, ×100 |
| Наценка по категориям | 3 линии (выбор бара) | взвешенная наценка Розлив / Фасовка / Кухня для выбранного бара |
| Наценка ₽ (маржа) | линия (наложение баров) | Выручка − Себестоимость |
| Чеки | линия (наложение баров) | count(DISTINCT `UniqOrderId.Id`) |
| Средний чек | линия (наложение баров) | Выручка / Чеки |

**Блок 2. Продуктовый разрез:**

| График | Тип | Источник / формула |
|--------|-----|--------------------|
| Доли К/Ф/Р | 100% stacked | доли категорий (нормировано к 100%) |
| Расход розлива/фасовки | 2 линии (выбор бара) | Розлив / Фасовка помесячно. Деньги (`DishDiscountSumInt`), либо кол-во: розлив в **литрах** (сумма блока литров по стилям — он продаётся объёмом), фасовка в **штуках** (`units_packaged` = `DishAmountInt`). У линий разные единицы (`_fmt` на датасете) |
| Проливы в литрах по стилям | stacked bar | `DishAmountInt` × объём порции (оценка из названия), группировка `ThirdParent` |
| Локал/импорт фасовка | stacked bar | по `DishForeignName`: Россия = локал, прочие страны = импорт |
| Локал/импорт розлив | stacked bar | то же для `Напитки Розлив` |

**Блок 3. Лояльность:**

| График | Тип | Источник / формула |
|--------|-----|--------------------|
| Новые гости | линия | `get_new_guests_count` — уник. телефоны с `CustomerCreatedDateTyped` в месяце |
| Выручка карты/без карт | stacked bar | по наличию `Delivery.CustomerCardNumber` (из `get_rfm_report`) |
| Списанные баллы | линия | Σ `DiscountSum` (вся сумма скидок, как на главном дашборде) |
| ТОП-10 гостей | таблица | агрегация по карте за месяц: траты = Σ `DishDiscountSumInt`, визиты = уникальные даты `OpenDate.Typed` (как в RFM-разделе скидок), свой селектор месяца |

### Архитектура: витрина данных (ETL / материализованное представление)

Исторические месячные агрегаты **иммутабельны** — закрытый месяц не меняется.
Поэтому просмотр и пересчёт разделены:

- **Просмотр (serve)** — эндпоинты читают **только с диска**, без обращения к iiko →
  мгновенно. Это ключевой момент: при открытии вкладки iiko не дёргается вообще
  (раньше всплеск авторизаций ронял живой дашборд — `ConnectionReset`).
- **Пересчёт (refresh)** — фоновый, по расписанию
  ([core/monthly_report_scheduler.py](../core/monthly_report_scheduler.py)):
  - **ночью** (env `MONTHLY_REPORT_HOUR`/`MINUTE`, default 04:30) досчитывает
    **недостающие закрытые месяцы** по всем заведениям и годам глубины — на практике
    реальная работа только когда закрылся новый месяц, т.е. **1-го числа**;
  - **закрытый** месяц считается один раз и «замораживается» навсегда; текущий
    неполный месяц не считается (в отчёте не нужен);
  - ежедневная проверка — страховка: если 1-го сервер был недоступен, досчитает
    следующей ночью;
  - **при старте** приложения — разовый бэкфилл за `BACKFILL_YEARS` (3) лет, чтобы
    свежий деплой наполнил витрину (на тёплом кэше — почти no-op).

Хранилище:
- Каталог: `<persistent>/monthly_cache/<block>__<venue>__<year>.json`
  (на проде — постоянный диск `/kultura`, в dev — `data/`). Объём — килобайты
  (только готовые агрегаты, не сырые OLAP-строки).
- Запись — `atomic_write_json` + `file_lock` ([core/json_store.py](../core/json_store.py)),
  безопасно под `gunicorn --workers 2`; шедулер защищён atomic lock-файлом на день.
- Поле `_refreshed_at` в каждом файле → на фронте подпись «данные на …».

Обновление — автоматическое:
- Кнопки «Обновить» в интерфейсе **нет** (сотрудники не должны вручную дёргать пересчёт —
  это только нагружало iiko и ничего не меняло). Данные обновляются сами 1-го числа ночным
  шедулером.
- Админ-эндпоинты (только вручную через URL): `?force=1` — досчитать недостающие закрытые
  месяцы текущего года; `?full=1` — глубокий пересчёт всего текущего года (для редких
  ретро-правок в iiko, пересчитывает и замороженные месяцы). Прошлые годы заморожены.

**Обычный дашборд** (`/api/dashboard-analytics`) этой схемой **не затронут** — он
остаётся живым и онлайн, всегда актуальные данные.

### Допущения и ограничения

- **Литры — оценка.** Объём порции парсится из названия позиции; если не распознан —
  по умолчанию 0.5 л (см. `DraftAnalysis._estimate_volume`).
- **Локал/импорт по стране.** Позиции с пустым `DishForeignName` попадают в «импорт».
- **Новые гости** определяются по дате регистрации карты; гости без карты не считаются.
- **Списанные баллы** = вся сумма скидок (`DiscountSum`), не только баллы лояльности.
