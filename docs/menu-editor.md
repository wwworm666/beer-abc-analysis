# Редактор меню (`/menu`)

## Что это

Серверная страница для ведения пивного меню и печати A4-карточек. Перенесена из
локального `menu_tool/` (Flask на :5050) в основное приложение: данные хранятся на
постоянном диске `/kultura`, правки идут через дашборд (раздел «Управление» →
«Редактор меню»), цены тянутся из iiko по кнопке.

Интерфейс — две страницы (в стиле дизайн-системы приложения: IBM Plex Mono,
терракотовый акцент, темы):
- `/menu` — **библиотека-таблица** с поиском (№/Название/Страна·Пивоварня/Стиль/ABV/
  дескрипторы), клик по строке открывает карточку; тулбар: Обновить цены, Печать,
  Скачать PDF, + Новая.
- `/menu/edit?id=N` — **редактор одной карточки**: слева форма, справа живое превью
  A4-карточки (PT Serif, бежевая) с авто-масштабом.

Ключевые свойства:
- **Единый каталог** `menu_cards.json` (~261 карточка): историческая библиотека + 21
  текущий кран + весь разливной ассортимент за 2 года (`data/_draft_sorts_2y.csv`),
  дедуплицированы. Поле `tap` = № крана, если сорт сейчас на кране.
- **Один вариант разлива** — 0,25 / 0,4 / 0,5 (прежний переключатель A/B 0,3/0,5
  убран; поля `p03` в данных нет).
- **Кнопка «Обновить цены из iiko»** — актуальные цены прайса из продаж разливного.
- **Вкусы/теги/шкалы** (горечь·плотность·цвет) заполнены веб-исследованием по сортам.
- **Печать / PDF** — все карточки (или только краны) одним файлом, Chromium-рендер
  на стороне сервера (в прод-образе `mcr.microsoft.com/playwright/python` есть).

## Как работает

### Модель данных (одна карточка)

```json
{
  "id": 101, "n": 1, "tap": 1,
  "name": "ФестХаус Хеллес", "latin": "FestHaus Helles",
  "brewery": "ФестХаус", "country": "Россия",
  "style": "Хеллес", "abv": "5,2",
  "tags": ["солодовый", "медовый", "освежающий"],
  "ratings": { "gor": 2, "plot": 2, "cvet": 1 },
  "p025": 230, "p04": 390, "p05": 450
}
```

- `tap` — номер крана (1..24) или `null`. На карточке № = `tap` (если на кране), иначе `n`.
- Десятичный разделитель — запятая; `null` в цене рендерится как «—».
- Шкалы 1–5: **горечь** (по IBU), **плотность** (по OG/крепости), **цвет** (по SRM).

### Хранение (постоянный диск)

`routes/menu_editor.py` читает/пишет через `core.storage_paths.get_data_path(
"menu_cards.json", seed_from_local=True)`:
- на проде файл живёт на `/kultura`; при первом обращении сидируется из repo-копии,
  далее диск — единственный источник правды (правки из UI сохраняются прямо туда);
- в dev (диск `/kultura` отсутствует) используется repo-копия `data/menu_cards.json`.

### Сборка каталога (детерминированно)

`py -3 scripts/build_menu.py` (цены из кэша) — три шага, порядок важен:
1. **`build_menu_catalog.py`** — дедуп библиотеки по имени (5 групп) + слияние 21
   крана; плюс ручной список семантических дублей `SEMANTIC_DUPES` (одно и то же
   пиво по-разному записано — «Коммерцинрат»↔«Коммерциенрат Приват» и т.п., см.
   `find_menu_dupes.py`). Победитель кранов = текущая правда.
2. **`fill_menu_prices.py`** — текущие цены из iiko (`--live`) или из кэша.
3. **`apply_research.py`** — теги/шкалы из `menu_research_output.json` по `id`.

Источники (`menu_tool/data/menu_items.json`, `menu_tool/data/menu_taps.json`,
`data/menu_research_output.json`) — в репозитории, поэтому сборка воспроизводима.

### Цены: формула (core/menu_pricing.py)

Те же продажи разливного, что у `scripts/menu_prices.py`, но обобщённо по любому
имени карточки:
- запрос OLAP `SALES`, фильтр `DishGroup.TopParent="Напитки Розлив"`, поля
  `DishAmountInt`, `DishSumInt` («сумма без скидки» = цена прайса);
- **цена порции = взвешенная мода** unit-цены `round(DishSumInt/DishAmountInt)` по
  строкам (бар, дата), вес = проданные порции; тай-брейк — более свежая дата;
- **сопоставление карточка↔сорт iiko** высокоточное: токены имени (включая стиль,
  с транслитерацией кир/лат) должны быть над/подмножеством токенов сорта и делить
  хотя бы один «бренд»-токен (не слово стиля). Карточки без бренд-токена
  («Стаут», «Хеллес», «ИПА») и неоднозначные совпадения НЕ обновляются —
  лучше не тронуть, чем поставить чужую цену. Не найденные в текущих продажах
  сорта сохраняют прежнюю цену.

Кнопка «Обновить цены» (`POST /menu/api/refresh-prices`) гоняет это по живым данным
iiko (последние ~105 дней), пишет результат на диск и возвращает сводку (сколько
совпало/изменилось).

### Вкусы/теги/шкалы: происхождение

Заполнены веб-исследованием (workflow `menu-taste-research`, 86 сортов): по каждому
сорту 1–3 веб-поиска (untappd / сайт пивоварни / profibeer), затем маппинг
IBU→горечь, OG/крепость→плотность, SRM→цвет и 3 русских слова-дескриптора вкуса.
Результат и источники — `data/menu_research_output.json` (71 high / 11 medium / 4 low
confidence). 21 кран уже были заполнены ранее, их ресёрч не трогал.

### Рендер PDF

`menu_card.html` / `menu_print.html` ссылаются на `/static/menu/*.css`. Для серверного
рендера через Playwright (`page.set_content`, без серверного контекста) абсолютные
`/static`-ссылки не резолвятся — `_inline_assets()` подставляет содержимое CSS/шрифта
как `<style>` перед рендером. Иначе PDF выходит без стилей. `page.pdf(format=A4,
margin=0, print_background=True, prefer_css_page_size=True)` — каждая карточка на своей
A4-странице (`page-break-after: always` в `@media print`). Заполненные деления шкал —
`background-color`, поэтому `print-color-adjust: exact` обязателен (иначе пустые).

### Маршруты

| Endpoint | Метод | Что |
|---|---|---|
| `/menu` | GET | Страница-редактор |
| `/menu/card?id=N` | GET | Одиночное превью карточки |
| `/menu/print?filter=tap\|all` | GET | Все карточки на печать (Ctrl+P) |
| `/menu/api/items` | GET/POST | Список / создание |
| `/menu/api/items/<id>` | PUT/DELETE | Обновление (автосейв) / удаление |
| `/menu/api/refresh-prices` | POST | Обновить цены из iiko |
| `/menu/api/render-pdf` | POST | PDF одной карточки (Playwright) |
| `/menu/api/export-pdf?filter=tap\|all` | GET | PDF всех карточек одним файлом |

## Деплой (важно)

`docker-compose.yml` монтирует `/srv/beer/data:/app/data`, поэтому baked-in repo-копия
`/app/data/menu_cards.json` **затеняется** хост-маунтом. Чтобы каталог засидировался на
`/kultura`, файл нужно положить в источник сидирования (`/app/data` = хост
`/srv/beer/data`) **или** сразу на диск:

```bash
ssh root@139.100.200.92
cd /opt/beer && git pull --ff-only
cp data/menu_cards.json /srv/beer/data/menu_cards.json      # источник сидирования
#   ИЛИ сразу на постоянный диск (минуя сидирование):
# cp data/menu_cards.json /srv/kultura/menu_cards.json
docker compose up -d --build app
```

Первое обращение к `/menu` засидирует файл на `/kultura/menu_cards.json`; дальше диск —
источник правды (как у планов/остатков). Playwright/Chromium в образе уже есть —
отдельных шагов для PDF не нужно. См. [[project_prod_deploy_selectel]],
[[project_plans_editing]].
