# chz-stock-integration.md — интеграция iiko + Честный Знак

Полное описание системы стыковки остатков iiko со сроками годности
из Честного Знака. По состоянию на 2026-04-26.

## Что это

Бар-менеджер ИНВЕСТАГРО (4 точки в СПб) видит на одной странице:
- что физически на полке каждого бара (из iiko)
- какие конкретно партии этих товаров есть в ЧЗ с привязкой по
  КПП каждого бара
- ближайший и дальний срок годности по каждой партии
- сигналы «касса не RETIRE'ит» когда CHZ-кодов больше чем iiko-сток

Доступ: `GET /stocks` → вкладка «Сроки годности».

## Главные концепты

### 1. Стыковка iiko ↔ ЧЗ происходит по GTIN (баркоду)

iiko хранит баркоды только в **XML**-номенклатуре, не в OLAP. У товара
может быть 0..N баркодов. EAN-13 (13 цифр) нормализуем в GTIN-14 (14
цифр с лидирующим нулём) через `zfill(14)`.

```python
# core/iiko_barcodes.py
def _normalize_gtin(raw: str) -> str | None:
    s = raw.strip()
    if not s.isdigit() or len(s) > 14:
        return None
    return s.zfill(14)
```

Если в iiko-карточке баркод вписан криво (или вообще не вписан) —
матча не будет. Это самая частая причина «нет в ЧЗ» **после** того
как все остальные fixes сделаны.

### 2. Привязка партии к бару — по КПП

CSV-выгрузка `FILTERED_CIS_REPORT` для каждой строки CIS содержит
поля **`kpp` и `fiasId`** — КПП и ФИАС-id места осуществления
деятельности (МОД), на котором числится этот код. У ИНВЕСТАГРО 4
МОД = 4 КПП:

| Бар | КПП | fiasId | Адрес |
|---|---|---|---|
| Большой пр. В.О | 780145001 | 02d5531c-c228-4fd5-9f39-831cfd4bd07c | БОЛЬШОЙ В.О. ПР-КТ, 18 |
| Лиговский | 781645001 | 44b7f9ba-edcc-4385-b165-bc5625f88582 | пр-кт Лиговский, 271 |
| Кременчугская | 784201001 | 44b7f9ba-edcc-4385-b165-bc5625f88582 | КРЕМЕНЧУГСКАЯ УЛ, 11 |
| Варшавская | 781045001 | f111ab59-a4aa-4939-bf25-08ca83b72b3d | ВАРШАВСКАЯ УЛ, 6 |

Маппинг получен через `chz.py mods` (запрос к `GET /mods/list`),
сохранён в `chz_test/debug/mods.json`. Жёстко прописан в
[routes/stocks.py:756](../../routes/stocks.py#L756) (`bar_kpp_map`).

Так что для бара X фильтруем `chz_item.by_kpp[bar_kpp_map[X]]` — точные
партии именно этого бара.

### 3. Партии — это `(production_date, expiration_date)` пары

В CSV каждая строка — один CIS-код с собственными датами. Парсер
агрегирует в `chz_stock.json` так:

```json
{
  "gtin": "04610093628430",
  "name": "Пиво светлое...",
  "count": 130,
  "expiration_dates": ["2025-05-28", ..., "2026-08-31"],
  "production_dates": ["2025-04-17", ..., "2026-03-04"],
  "batches": [
    {"production_date": "2026-03-04", "expiration_date": "2026-08-31", "count": 4},
    {"production_date": "2026-03-03", "expiration_date": "2026-08-30", "count": 8},
    ...
  ],
  "by_kpp": [
    {
      "kpp": "784201001",
      "fiasId": "44b7f9ba-...",
      "count": 93,
      "batches": [{"production_date": ..., "expiration_date": ..., "count": 4}, ...]
    },
    {"kpp": "781045001", ...},
    ...
  ]
}
```

`batches` глобально по юрлицу + `by_kpp[].batches` по конкретному бару.
Сортировка batches по `production_date desc` — newest first.

### 4. КПП-фильтр — точные данные, не эвристика

Старый алгоритм брал «N самых свежих партий по дате производства до
iiko-стока» — врал для одного SKU размазанного по 4 барам (все 4 бара
видели одни и те же партии).

Новый алгоритм: **берём `chz_item.by_kpp[bar_kpp]` напрямую**. Для
бара «Лиговский» = только КПП 781645001 = только партии физически
числящиеся за Лиговским. Никакой эвристики.

### 5. Категории «нет в ЧЗ»

Из 366 фасовка-позиций 241 матчатся с ЧЗ, 125 — нет. Причины:

| Категория | Кол-во | Можно ли починить |
|---|---|---|
| Старые остатки до 01.12.2025 (до поэкземплярного учёта) | ~80 | Нет — это легитимно. По мере распродажи матч-рейт сам поднимется |
| Импорт где поставщик не подписал УПД через ЭДО | ~30 | Звонок поставщику, чтобы передавал ЭДО |
| Иikoшный баркод не совпадает с DataMatrix на бутылке | ~10 | Бармен сверяет → исправляет карточку |
| Товар маркируется в группе для которой нет доступа (alcohol 11 = ЕГАИС) | <5 | Нерелевантно (мы не на ЕГАИС) |

Подробности в разделе «История и инсайты» ниже.

## API endpoints (Flask)

### `GET /api/stocks/expiry?bar=<bar>`

**Главный эндпоинт интеграции.** Стыкует iiko-остатки с ЧЗ-партиями
по выбранному бару.

Параметр `bar`: `Большой пр. В.О` / `Лиговский` / `Кременчугская` /
`Варшавская` / `Общая`.

Поток исполнения:
1. Маппит bar → `target_store_id` (iiko UUID склада) и `target_kpp` (КПП ЧЗ)
2. Дёргает iiko: nomenclature (OLAP-кэш) + балансы (live)
3. Фильтрует по группе фасовки (`FASOVKA_GROUP_ID = '6103ecbf-...'`)
4. Получает `barcode_map = {gtin: [iiko_pid]}` через `core/iiko_barcodes`
5. Читает `chz_test/debug/chz_stock.json`
6. Для каждой iiko-позиции: получает её GTIN'ы → ищет в `chz_by_gtin` →
   если выбран конкретный бар, фильтрует `by_kpp[target_kpp].batches`,
   иначе берёт все `batches` юрлица
7. Считает `nearest_expiry`, `latest_expiry`, `days_to_expiry`
8. Сортировка: срочные (<30д) сверху, дальние, в конце без CHZ-данных

Ответ:
```json
{
  "bar": "Лиговский",
  "updated_at": "2026-04-26T17:30:00",
  "chz_updated_at": "2026-04-26T17:13:48",
  "total_items": 115,
  "matched_items": 76,
  "near_expiry_count": 0,
  "items": [
    {
      "name": "...",
      "category": "ООО МаркетБир",
      "stock": 10.0,
      "avg_sales": 0.5,
      "stock_level": "high",
      "gtins": ["04610093628430"],
      "chz_total_count": 130,
      "bar_chz_count": 9,
      "expiration_dates": [...],
      "production_dates": [...],
      "inferred_batches": [{"production_date": "...", "expiration_date": "...", "count": 9}],
      "nearest_expiry": "2026-09-15",
      "latest_expiry": "2026-09-15",
      "days_to_expiry": 142,
      "has_chz_data": true
    }
  ]
}
```

`bar_chz_count` vs `chz_total_count`:
- `bar_chz_count` — кодов на КПП этого бара
- `chz_total_count` — кодов по всему юрлицу
- Если `bar_chz_count > stock + 1` → касса не RETIRE'ит, в UI красная плашка

Реализация: [routes/stocks.py:735-974](../../routes/stocks.py#L735).

### `POST /api/chz/refresh`

Запускает обновление кэша ЧЗ через бар-ПК. Идёт в фоне (subprocess.Popen).
Возвращает мгновенно.

Поток: запускает `remote_exec.py run csv-auto` →
1. SSH к бар-ПК
2. `chz.py token` (refresh, ~10 секунд)
3. `chz.py csv-auto` (3 dispenser-задания × ~30 сек = ~2 минуты)
4. SFTP pull `chz_stock.json` локально

Логи: `chz_test/debug/refresh.log` (truncated перед каждым запуском).

Требует env var `REMOTE_PASS`. Без неё → 503.

Защита от параллельных запусков: `_refresh_lock` + проверка
`_refresh_proc.poll()`. Один параллельный refresh.

### `GET /api/chz/refresh/status`

Возвращает текущее состояние:
```json
{
  "running": false,
  "exit_code": 0,
  "cache_updated_at": "2026-04-26T17:13:48",
  "log_tail": "...последние 3000 байт лога..."
}
```

Используется UI для polling раз в 5 секунд после нажатия кнопки.

### `GET /api/chz/stock`

Прямое чтение `chz_stock.json` без всякой обработки. Используется
старым кодом и для отладки.

### `GET /api/stocks/bottles?bar=X`

Старый endpoint без ЧЗ-данных, остался ради обратной совместимости со
старой вкладкой «Фасовка». Не трогать — вкладка ему важна.

### `GET /api/stocks/chz`

Прямой синхронный вызов `chz.get_chz_stock()` через бар-ПК. Очень медленный
(минуты), используется только для ручного разового вызова. Не нужно
вызывать на проде.

## Регулярные процессы

### Авторефреш в 03:00 (scheduler)

[core/chz_scheduler.py](../../core/chz_scheduler.py) — daemon-thread.

Запускается при старте Flask **только если `REMOTE_PASS` в env**:
```python
# app.py
from core.chz_scheduler import start_scheduler
start_scheduler()
```

Логика: считает время до следующих 03:00 → `time.sleep` → POST на
свой же `/api/chz/refresh` (через urllib).

Время настраивается env:
- `CHZ_REFRESH_HOUR` (default 3)
- `CHZ_REFRESH_MINUTE` (default 0)

Цикл: после refresh ждём ещё 60с защиты от мгновенного повтора, потом
снова считаем интервал до следующих 03:00 и спим.

При перезапуске Flask scheduler стартует заново, время пересчитается.

### Лимиты ЧЗ dispenser API

- 10 заданий/сутки на одну product-группу
- Хранение результатов в их системе: 30 дней
- Таймаут выполнения задания: ~3600 сек (мы ждём максимум 600)

У нас 3 группы × 1 раз/сутки = с большим запасом. Ручные нажатия
кнопки тоже идут в этот лимит.

## UI

`templates/stocks.html` → вкладка «Сроки годности».

Структура:
- Селектор бара (общий для всех вкладок)
- Кнопка «Обновить данные ЧЗ» с polling-индикатором
- Мета-строка: дата кэша + поясняющая подсказка про legacy-остатки
- Stat-блок: всего/с CHZ/срочных <30д
- Группы по поставщикам, в каждой таблица:
  - Название
  - Остаток (шт)
  - Ср. продажи
  - Срок (по последним партиям) — `nearest_expiry` + `latest_expiry`
  - Партии на полке — список `inferred_batches` + предупреждения

Подсветка строк:
- Красный фон — `days_to_expiry < 30` или просрочено
- Жёлтый — `days_to_expiry < 90`
- Серый — `has_chz_data == false`

Предупреждения внутри ячейки «Партии на полке»:
- ⚠ когда `bar_chz_count > stock + 1` (касса не RETIRE'ит)
- Серая подпись когда товар есть и в других барах юрлица

JS: [templates/stocks.html:786-1050](../../templates/stocks.html#L786) —
функции `loadExpiry`, `formatBatches`, `formatHeadlineExpiry`,
`refreshChzData`, `pollChzRefreshStatus`.

```bash
py chz_test/suggest_barcode_fixes.py
```

Для каждой iiko-позиции «нет в ЧЗ» ищет в `chz_stock.json` товар по
схожести бренда/названия (минимум 2 общих слова длиной ≥3). Output:

Score:
- 4 — точный матч
- 3 — уверенный (Кулинар, Дальняя Дача, Бакунин и т.п.)
- 2 — требует ручной проверки (часто ложные совпадения)

```bash
py chz_test/export_bartender_list.py
```

Фильтрует несматченные позиции, исключает поставщиков из
`EXCLUDE_SUPPLIERS = {"СПБ-Премиум"}` (старый импорт без поштучного
учёта). Output: `chz_test/debug/для_барменов_проверка_баркодов.csv`.

Колонки: Бар, Поставщик, Товар, Остаток, Текущий баркод, Предлагаемый
GTIN, Уверенность, **пустая колонка для реального GTIN с DataMatrix**,
Заметки.

Список группируется по барам — каждый бармен берёт свою часть.

## Эксплуатация

### Первый запуск

1. Установить переменную окружения с паролем SSH к бар-ПК:
   ```bash
   export REMOTE_PASS='пароль'
   ```
2. Запустить Flask:
   ```bash
   py app.py
   # или: REMOTE_PASS=... py app.py
   ```
3. Убедиться в логе:
   ```
   [CHZ-SCHED] стартовал, refresh ежедневно в 03:00
   ```

### Нажать «Обновить данные ЧЗ» вручную

`http://127.0.0.1:5000/stocks` → вкладка «Сроки годности» → кнопка.
Цикл занимает 1-3 минуты. После завершения таблица перезагружается.

### Если поменялись баркоды в iiko

В iiko XML обновляется автоматически только при ручном вызове —
`/products` API дёргается **только** через `iiko_barcodes.get_barcode_map`
который читает с диска. Если баркод изменился в iiko, нужно:

```python
import requests
from core.olap_reports import OlapReports
olap = OlapReports(); olap.connect()
r = requests.get(f'{olap.api.base_url}/products', params={'key': olap.token}, timeout=300)
open('data/cache/nomenclature__products.xml', 'wb').write(r.content)
```

Затем `core.iiko_barcodes.get_barcode_map(force_refresh=True)`. На
практике это делается раз в 1-2 месяца.

**TODO на будущее:** добавить эндпоинт `POST /api/iiko/refresh-xml` или
автоматический refresh раз в неделю в scheduler.

### Если бары добавили новый КПП / новую точку

Обновить:
1. `routes/stocks.py:bar_id_map`, `store_id_map`, `bar_kpp_map`
2. `chz_test/debug/mods.json` через `chz.py mods`

## Известные ограничения

1. **Бутылки розлива до 01.12.2025** — поштучный учёт не велся.
   Из 125 несматченных ~80 — это легитимные старые остатки. Не баг.

2. **Касса не делает RETIRE при продажах** — `bar_chz_count` со
   временем накапливается, становится больше iiko-стока. Видно в UI
   как ⚠. Решается на стороне кассы (другая задача).

3. **iiko XML устаревает** — раз в несколько месяцев нужно вручную
   обновлять `data/cache/nomenclature__products.xml`. Когда устарел —
   часть товаров «не в XML» → нет баркодов → нет матча.

4. **Ошибочные баркоды в iiko** — бариста при заведении товара мог
   вписать левый EAN-13 (как в случае Кулинара). Решается через
   `export_bartender_list.py` + ручная сверка.

5. **Импорт без ЭДО** — поставщик не передал УПД через электронный
   документооборот → коды у него на балансе, не у нас. Решается
   звонком поставщику.

6. **СПБ-Премиум и подобные** — возят импорт в основном до 01.12.2025,
   у них всё легитимно без КИ. Не отвлекаемся, исключены из
   bartender-списка.

## История разработки и инсайты

Хронологический обзор того что мы делали и какие ошибки совершали.
Полный текст в [docs/CHANGELOG.md](CHANGELOG.md), здесь — компактный
рассказ.

### Шаг 1 — попытка использовать `/cises/search` API ❌

Сначала пытались тянуть остатки через синхронный API `cises/search`.
**Не сработало:**
- API режет страницы до **100 записей**, наш `limit=1000` игнорировался
- Пагинация рвалась из-за условия `len(items) < limit`
- Без фильтра по дате возвращалось **79,000+ INTRODUCED-кодов** —
  историческое накопление за все годы (касса не RETIRE'ит)

### Шаг 2 — переход на CSV-выгрузку из ЛК (вручную) ✅

Выгружать CSV из ЛК ЧЗ → класть в `chz_test/debug/pg*_csv/` →
парсить через `chz.py csv`. CSV из ЛК уже применяет правильные
фильтры (`participantInn + packageType=UNIT`). Получили
**541 GTIN, 2878 кодов** — реальный склад вместо 30 GTIN от API.

### Шаг 3 — стыковка с iiko по GTIN ✅

Создан `core/iiko_barcodes.py` — парсит XML iiko, строит
`{gtin14: [iiko_pid]}`. Эндпоинт `/api/stocks/expiry` соединяет
iiko-балансы с ЧЗ-партиями.

### Шаг 4 — попытка эвристики «свежие партии под iiko-сток» ❌

Когда один SKU размазан по 4 барам, эвристика «брать N последних
партий по дате производства до iiko-стока» давала бред — все 4 бара
видели одни и те же партии. Юзер заметил.

### Шаг 5 — открытие kpp/fiasId в CSV ✅

В CSV-выгрузке ЛК **уже были колонки `kpp` и `fiasId`**, но мы их
игнорировали. Каждый CIS привязан к одному из 4 КПП = 4 МОД =
4 баров. Парсер пересобран чтобы агрегировать `by_kpp`. Теперь —
точные данные, не эвристика.

### Шаг 6 — обновление iiko XML кэша ✅

Файл `nomenclature__products.xml` был от 26.12.2025 (не обновлялся
4 месяца). За это время добавилось 496 товаров и 349 баркодов.
Match rate скакнул со 82 до 202 позиций (Общая) после refresh.

### Шаг 7 — полная автоматизация через dispenser API ✅

Самое большое улучшение в сессии. Раньше юзер вручную выгружал CSV
из ЛК. Теперь:
- `chz.py csv-auto` дёргает `dispenser/tasks` для каждой группы
- API возвращает **ZIP с CSV внутри** (не чистый CSV — пришлось
  добавить unzip)
- После poll → COMPLETED → download → перестройка `chz_stock.json`
- В Flask: `POST /api/chz/refresh` запускает SSH-цепочку
- Daemon-thread в `core/chz_scheduler.py` — авторефреш в 03:00
- Кнопка в UI с polling

### Шаг 8 — расширение групп ✅, проба `alcohol` ❌, проба `APPLIED` ❌

Добавили `nabeer` (group 22) и `softdrinks` (group 23). Match rate
скакнул на 13 позиций.

Попробовали `alcohol` (group 11) — **0 кодов**. Алкоголь маркируется
**ЕГАИС-маркой**, не ЧЗ. Удалили.

Попробовали `status=APPLIED` для импорта — **0 кодов**. APPLIED-коды
принадлежат **producer-у**, не нам. С `participantInn=наш` они никогда
не вернутся. Удалили цикл, оставили только INTRODUCED.

### Шаг 9 — расследование «нет в ЧЗ» ✅

Гипотеза юзера про поэтапные даты маркировки подтвердилась. Найдены
постановления:
- ПП РФ № 2173 — пиво (этапы 01.04.2023 → 15.01.2024)
- ПП РФ № 678 — безалк. пиво (с 01.10.2024)
- ПП РФ № 887 — безалк. напитки (с 01.12.2023)
- ПП РФ № 1415 — поэкземплярный учёт бутылок с **01.12.2025**

Вывод: ~80 из 125 несматченных — это легитимные остатки до даты
поэкземплярного учёта своей категории.

### Шаг 10 — открытие баркод-несоответствия (Кулинар) ✅

Юзер отсканировал бутылку «Кулинар Смородиновый Крамбл» приложением
ЧЗ. На бутылке GTIN `04610424291500`, в iiko-карточке `04631169328016`.
**Это разные товары в системе ЧЗ.** Реальный GTIN ЕСТЬ в нашем
`chz_stock.json` (3 шт на Большом В.О, срок до 2027-04-05). Просто
iiko не знает правильный баркод.

### Шаг 11 — авто-подбор настоящих GTIN ✅

`suggest_barcode_fixes.py` — для каждой несматченной iiko-позиции
ищет в `chz_stock.json` товар по схожести бренда. Score 4 = точный,
3 = уверенный, 2 = требует проверки.

`export_bartender_list.py` — генерит CSV для барменов с колонкой
для заполнения реального GTIN. После заполнения можно будет
автоматически обновить карточки iiko через iiko API
(не реализовано — пока ручной труд).

### Шаг 12 — проверка новости про 01.12.2025 ✅

Юзер прислал новость. Подтвердили через web-research:
- 01.12.2025 — обязательный поэкземплярный учёт бутылок/банок пива
- ОСУ (объёмно-сортовой) отменён
- Принимать УПД с КИ через ЭДО, иначе ответственность на покупателе

Это объясняет почему старый импорт «не в ЧЗ» — он ещё в режиме ОСУ.

### Шаг 13 — фильтр СПБ-Премиум из bartender-списка ✅

СПБ-Премиум возит только импорт со сроком розлива до 01.12.2025.
Поштучного учёта не ведётся, ЧЗ-приложение не пропускает сканирование.
Бесполезно проверять баркоды — добавили в `EXCLUDE_SUPPLIERS`.
