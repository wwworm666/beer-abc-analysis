# График смен

## Что это

Управление графиком смен сети из 4 баров. Две страницы с разным назначением:

- **`/schedule` — просмотр** (для барменов): сетка «дни x бары», кто где работает,
  ввод факта отработанных часов. Финансовых данных на странице нет.
- **`/schedule/edit` — рабочий стол владельца**: кисть для назначения смен с
  пресетами времени, предупреждения (незакрытые точки, конфликты с выходными),
  план/факт выручки по дням, нагрузка по сотрудникам, сводка месяца, пожелания,
  реестр сотрудников.

Эталоном функционала была гугл-таблица владельца (дни x сотрудники, часы, итоги);
веб-версия хранит те же данные в SQLite и считает агрегаты сама.

## Как работает

### Схема БД (v5)

SQLite `shifts.db`, WAL + busy_timeout (gunicorn 2 workers). Миграции **только
additive** (`PRAGMA table_info` -> `ALTER TABLE ADD COLUMN`, `CREATE TABLE IF NOT
EXISTS`); перед миграцией менеджер снимает консистентный бэкап через SQLite
backup API в `shifts.db.backup_v{старая_версия}`. DROP таблиц запрещён — в проде
живые данные.

- `locations` — 4 бара; `venue_key` — мост к планам
  (`varshavskaya`, `bolshoy`, `kremenchugskaya`, `ligovskiy`).
- `roles` — бармен / второй бармен (роль «стажёр» убрана 2026-06-24, владелец:
  только две роли); `color` — цвет **левой полоски чипа** смены (не заливка, см.
  «Внешний вид сетки»); `rate_per_hour` (v5) — ставка за час для ЗП (DEFAULT 300).
- `shifts` — смена: `date`, `employee_id` (стабильный id из iiko, v6 — ключ
  сотрудника), `employee_name` (снимок имени для показа/фоллбэка, канонизируется
  при синке), `location_id`, `role_id`, `start_time`, `fact_minutes`.
- `day_off_requests`, `wishes`, `daily_revenue` — как раньше (+ `employee_id`
  впрок, v6).
- `schedule_employees` — реестр: `name` (PK, текущее имя из iiko), `iiko_id`
  (v6, стабильный ключ — UNIQUE), `short_label` («РЮ»), `active`, `sort_order`.
  Идентичность — по `iiko_id`; см. «Идентичность сотрудника».
- `schedule_audit` (v4) — журнал изменений графика (append-only): `ts`,
  `actor_login`, `actor_name`, `action`, `entity_date`, `employee_name`,
  `summary`. См. раздел «Журнал изменений».

### Идентичность сотрудника — стабильный id из iiko (v6)

**Зачем.** Раньше сотрудник в смене и реестре хранился строкой-именем. iiko-синк
делал `INSERT OR IGNORE` по имени и ничего не удалял, поэтому переименование
сотрудника в iiko плодило дубли: новое имя приходило новой записью, а прошлые
смены оставались под старым. Ручная чистка не масштабируется.

**Как стало.** Сотрудник идентифицируется стабильным `id` из справочника iiko
(`/employees`, GUID; те же id, что в `responsibleUserId` кассовых смен). Смена
хранит `employee_id`; `employee_name` — только снимок для показа/фоллбэка.
Реестр (`schedule_employees`) ключуется `iiko_id`. Имя везде резолвится из
реестра по id (фронт: `Schedule.shiftDisplayName` / `shiftLabel`).

**Синк** (`ShiftsManager.sync_employees(pairs, overrides)`, эндпоинт
`POST /api/schedule/employees/sync`) за одну транзакцию, идемпотентно:
1. распространяет переименования: `shifts.employee_name` по `employee_id`;
2. бэкофиллит `employee_id` у смен без него: override → точное имя → однозначная
   перестановка слов (`«Васильев Никита» == «Никита Васильев»`);
3. upsert реестра по `iiko_id` (имя обновляется; legacy-строка с тем же именем
   «усыновляется» — получает id, сохраняя метку/порядок/активность); новые люди
   из справочника **без смен** добавляются неактивными (скрыты из кисти, видны в
   админке — владелец включает барменов), со сменами — активными;
4. удаляет осиротевшие legacy-строки реестра.

`overrides` (`{старое_имя_смены: iiko_id_или_текущее_имя}`) — разовая ручная
привязка наследия, где имя изменилось так, что авто-сопоставление по строке
невозможно (напр. «Артемий»→«Артем»). После бэкофилла переименования
подхватываются автоматически — **дубли больше не возникают**. Тесты:
`tests/test_employee_ids.py`.

### Два слоя времени смены

- **План**: `start_time` `'HH:MM'`. Смены двух типов: **день** (с 14:00,
  норма — хранится как `NULL`, бейдж времени на чипе не показывается, роль
  по умолчанию бармен) и **вечер** (`'18:00'`, бейдж «с 18:00», роль по
  умолчанию второй бармен; меняется в модалке). Другое время — только
  вручную через модалку для нестандартных случаев.
- **Факт**: `fact_minutes` — вводится **барменом руками в конце смены**
  (страница просмотра, клик по своей смене). Смена может длиться дольше
  кассовой, поэтому по API iiko факт часов вытащить нельзя — ручной ввод
  является единственным источником факта часов для зарплатных расчётов.
  Прошедшие смены без факта попадают в «Без часов».
### Виджет рейла «Нагрузка» (редактор И просмотр, money-free)

«Нагрузка» общий для `/schedule/edit` и `/schedule` — рендер один
(`Schedule.renderLoad` в `common.js`), данные — **money-free** эндпоинт
`GET /api/schedule/widgets/<y>/<m>` (без iiko, без рублей). На просмотре read-only —
у барменов финансов на странице нет.

**Нагрузка по сотрудникам** (группировка по стабильному `employee_id`):
- **Смен / 15** — число смен против нормы `SHIFT_NORM=15` (`core/schedule_plans.py`,
  то же значение, что в KPI-коэффициенте). Цвет: ≥нормы — зелёный, <60% нормы —
  янтарный (недобор), иначе обычный.
- **Часов** — сумма факта часов (ручной ввод бармена; единственный источник часов оплаты).
- **Подряд** — макс. серия смен подряд по календарным дням (`_max_consecutive_days`);
  5+ подряд — красный флаг переработки.
- **Без** — прошедшие смены без введённого факта.

### Виджет «План / Факт по дням» (деньги, обе страницы)

Таблица под сеткой: за каждый день месяца — **план выручки** (веса дней),
**факт** (живой iiko OLAP, тот же источник, что «Факт» на дашборде), **% выполнения**
и **кто стоял на смене** (чипы по стабильному `employee_id`). Клик по строке дня
раскрывает разбивку по барам (план/факт/% + кто на каждом баре).

- **Общий рендер** `Schedule.renderPlanFact(host)` (`common.js`) — на `/schedule/edit`
  И `/schedule`. Доступ к обеим страницам одинаковый (равные права аккаунтов), поэтому
  отдельной «money-free» страницы для барменов больше нет — владелец решил видеть
  выручку и в просмотре.
- **Без отдельного эндпоинта и без второго вызова iiko**: рендерится на фронте из
  уже загруженного `S.state.plans` (`GET /api/schedule/plans` →
  `build_month_plans(..., olap_fact=...)`) и `S.state.shifts`. View подтягивает `/plans`
  в `loadPlanFact` (раньше не тянул).
- **% бейдж**: ≥100 зелёный, 85..99 янтарный, <85 красный; нет факта — «—».
- **Итоговый %** = `Σфакт / Σплан тех дней, где факт уже есть` (как completion на
  дашборде): иначе будущие дни с планом без факта занижали бы выполнение. Σплан в
  футере — полный план показанных дней.
- **Честность атрибуции**: в строке дня показаны **все**, кто работал (день+вечер
  могут делить бар), без ложного «вся выручка дня — заслуга одного». Точную
  привязку человек→бар→результат даёт разворот по барам.

> Примечание: `/api/schedule/widgets` («Нагрузка») остаётся money-free — рублей в нём
> нет; деньги приходят отдельным `/plans`.

Финансовой «Сводки месяца» на странице графика нет — агрегаты живут на дашборде
(убрана 2026-06-25). Старый роут `/api/schedule/summary` фронтом не используется.

### Чип смены (сетка, обе страницы)

`buildChip` (`common.js`) рендерит смену как карточку, заполняющую ячейку: **имя
целиком** (`employeeShortName`, а не двухбуквенная метка) + строка-мета — капсула
типа смены (**«День»** при `start_time = NULL` / **«с HH:MM»** для вечера) и **факт
часов**, если проставлен. Роль читается цветом левой полоски (`--role-color`). Раньше
дневная смена показывала только инициалы — ячейка выглядела пустой.

### Пожелания на странице просмотра (read-only)

Пожелания сотрудников видны и на `/schedule` (read-only, `Schedule.renderWishesReadonly`),
не только в редакторе — бармены видят, что просили учесть. Правка пожеланий —
по-прежнему только в редакторе (textarea с автосейвом).

### Внешний вид сетки (визуальная дисциплина)

- **Чип смены** — спокойная карточка на нейтральном фоне (`--bg-secondary`),
  тёмный текст, тонкая **цветная полоска слева** = роль (`roles.color`,
  прокидывается в CSS-переменную `--role-color` в `common.js: buildChip`).
  Заливкой роль НЕ кодируется — иначе сетка превращается в кислотный heatmap на
  тёплой палитре бренда (так было до 2026-06-24).
- **Цвет ячейки = статус** (дыра/выходной/сегодня/конфликт), цвет полоски = роль —
  два независимых канала, не путать.
- **Радиусы и тени — из дизайн-системы** (`variables.css`): `--sch-radius` маплется
  на `--border-radius-sm` (16px), мелкие элементы 10px, кнопки/пиллы — pill-форма,
  карточки с `--shadow-sm`. Не заводить локальные «свои» радиусы — страница должна
  быть неотличима по языку от дашборда.
- Чипы прижаты к верху ячейки с зазором; пустое место под одиночной сменой —
  нейтральный фон ячейки, а не растянутый цветной блок.

### Раскладка редактора

- **Липкая панель управления** (`.sched-controls`): месяц-нав + кисть +
  предупреждения едут за прокруткой (`position: sticky`, z 50 — ниже модалки), чтобы
  кисть оставалась под рукой на длинной сетке.
- **Две колонки** (`.sched-layout`) на экране > 1180px: сетка слева, справочный рейл
  (Нагрузка + Сводка) справа — read-only блоки не растягивают вертикальный скролл.
  Уже — одна колонка, рейл уезжает под сетку. Пожелания / Сотрудники / История —
  на всю ширину под раскладкой.

### План выручки дня — read-only

Ручной ввод плана на странице графика **заморожен** (2026-06). Источник —
конвейер планов: `plansdashboard.json` -> `core/day_weights.py`
(пт/сб вес 2.0, override праздников) -> `data/daily_plans.json`.

Правило отображения плана точки на дату (`core/schedule_plans.py`):

1. есть `venue_key` в `daily_plans` за дату и значение > 0 -> план «из весов»;
2. иначе есть старый ручной `daily_revenue.plan_revenue` > 0 -> «ручной (устар.)»;
3. иначе — **«нет плана»** (None, никогда не 0: отсутствие плана не равно нулю).

Менять план — через «Планы по дням» в дашборде (override веса), не здесь.

### Факт выручки — автоматически из iiko OLAP (как на дашборде)

Факт подтягивается живьём при загрузке `plans`/`summary` — тем же способом
и метрикой, что «Факт» на дашборде:

- OLAP `SALES`, группировка `Store.Name + OpenDate.Typed`, агрегат
  `DishDiscountSumInt` (`core/olap_reports.py: get_store_daily_revenue`);
- общий кэш дашборда `cached_olap` (`extensions.py`): TTL 10 минут +
  single-flight, ключ `schedule_fact_{YYYY-MM}`;
- маппинг `Store.Name -> venue_key` через `venues_manager`
  (имена точек в iiko совпадают с нашими).

Если iiko недоступен — фоллбэк на сохранённый `daily_revenue.fact_revenue`
(наполняется legacy-эндпоинтами `POST /api/schedule/revenue/sync[-month]`,
из UI они больше не вызываются). Отсутствие продаж за день = «нет данных»,
не ноль.

### Формулы сводки месяца (тултипы в UI показывают эти же тексты)

| Метрика | Формула |
|---------|---------|
| План месяца | сумма планов всех дней, где план есть |
| Факт | сумма синкнутого факта |
| Средняя в день | факт / число дней с фактом |
| Ожидаемая | сумма по дням: факт, если есть, иначе план (веса пт/сб учтены в плане) |
| Выполнение % | факт / план тех же дней, где факт уже есть, x 100 |

«Ожидаемая» здесь — сознательно ДРУГАЯ метрика, чем `expected` дашборда
(`core/revenue_metrics.py` — линейная экстраполяция среднего): формула графика
weight-aware — повышенные планы пт/сб и праздничные override в остатке месяца
учитываются автоматически.

### Предупреждения (детерминированные, считаются на клиенте)

- **Не закрыто**: будущая дата (включая сегодня), точка без единой смены.
  Прошедшие пустые дни нейтральны — дырой не считаются.
- **Конфликт с выходным**: смена в день, попадающий в интервал day-off заявки
  того же сотрудника. Назначить можно (подтверждение), маркер остаётся.

### Журнал изменений (кто что менял)

Появился вместе с авторизацией: раньше график правили анонимно, теперь у каждого
действия есть автор (текущий пользователь сессии, `core/auth_guard: current_user`).

- **Что фиксируется**: создание / изменение / удаление смены, простановка и
  очистка факта часов, добавление / удаление пожелания выходного. По одной
  append-only записи на действие в `schedule_audit`. Выручка, пожелания (текст) и
  правки реестра пока не журналируются (выручка тянется из iiko автоматически,
  остальное — не «изменения графика» в смысле задачи).
- **Где пишется**: на уровне роутов (`routes/schedule.py`, хелпер `_audit`), а не
  менеджера — там доступен `current_user`, а сам `ShiftsManager` остаётся чистым
  хранилищем без зависимости от веб-слоя. Для удаления и изменения снимок «до»
  берётся через `get_shift()` / `get_day_off_request()` **перед** мутацией, чтобы
  в записи было читаемое описание (кого/где/когда).
- **`actor_name` — снимок** `display_name` автора на момент действия: журнал
  читается, даже если аккаунт потом переименуют или удалят (не FK на `users`).
- **Best-effort**: `_audit` обёрнут в try/except — сбой журналирования никогда не
  валит саму операцию со сменой (логируется предупреждением в stderr).
- **`entity_date`** — дата, к которой относится изменение; по ней история
  фильтруется на месяц, который открыт в редакторе.
- **Где видно**: две точки показа одной и той же ленты (`GET /api/schedule/audit`),
  обе за открытый месяц, новые сверху:
  - `/schedule/edit` — панель «История изменений» (сворачиваемая, ленивая,
    обновляется на каждый reload), формат «ДД.ММ ЧЧ:ММ — Автор — что сделал».
  - `/schedule` (просмотр, для барменов) — компактная лента «Последние изменения»
    под сеткой: цветная точка по типу действия (создание/удаление/факт/выходной),
    `limit=8`. Факт часов бармены вводят здесь же, и запись пишет тот же серверный
    роут — поэтому видно, кто проставил себе часы.

`ts` пишется в локальном серверном времени (как `created_at`/`updated_at`),
фронт показывает его как есть, без пересчёта таймзоны.

### Оплата часов по ролям (источник часов для ЗП)

С 2026-06-24 **график — единственный источник часов оплаты** (раньше страница ЗП
брала часы из кассовых смен iiko). Причина: на смену могут выходить двое — один
на полную (день, с 14:00), другой с вечера (18:00), и ставка за час у них разная;
а смена может длиться дольше кассовой, поэтому iiko-часы для оплаты не подходят.

Модель: **ставка — у роли** (`roles.rate_per_hour`), **часы — факт по каждой
смене** (`shifts.fact_minutes`). Оплата сотрудника за период =
`Σ по сменам: часы смены × ставку её роли`. Полная смена = роль «бармен»,
вечерняя = «второй бармен» (другая ставка). Ролей две (стажёр убран). До правки
владельцем ставка у всех ролей = 300 (старая единая), поэтому миграция оплату
не меняет.

- **Расчёт** — `ShiftsManager.get_hours_by_role_for_period(date_from, date_to)`:
  группирует смены периода по сотруднику и роли, суммирует `fact_minutes`
  (NULL не суммируется), считает `pay = часы × rate_per_hour`. Возвращает разбивку
  по ролям + `total_pay`, `total_hours`, `shifts_without_fact` (смены без факта —
  чтобы на ЗП было видно пробелы). Детерминированно, тестируется
  (`tests/test_salary_hours.py`).
- **API** — `GET /api/schedule/hours-by-role?date_from=&date_to=` (даты
  включительно) -> `{rates: роли со ставками, employees: [...]}`. Страница ЗП
  (`templates/bonus.html`) вызывает его рядом с `bonus`/`kpi` и подмешивает часы
  по имени сотрудника. Часы на ЗП **read-only** (правка — в графике, изменения
  пишутся в журнал); ставки правятся там же блоком «Ставки за час по ролям»
  (`PUT /api/schedule/role/<id>/rate`).
- **Ставка не снапшотится в смену**: меняешь ставку роли — пересчитываются все
  смены периода по новой ставке (ЗП считается помесячно, ставки меняются редко).
- iiko-часы (`total_hours`) остаются метрикой Employee Dashboard (выручка/час),
  но в оплату ЗП больше не входят. См. `docs/employee.md`, «Расчёт часов работы».

### API

| Метод | Путь | Что делает |
|-------|------|-----------|
| GET | `/api/schedule/employees[?all=1]` | реестр (+имена из истории смен) |
| POST | `/api/schedule/employees/sync` | синк реестра с iiko по id (бэкофилл/дедуп/переименования); body опц. `{overrides}` |
| PUT | `/api/schedule/employee/<iiko_id>` | short_label / active / sort_order (по id) |
| GET | `/api/schedule/roles`, `/api/schedule/locations` | справочники (roles c rate_per_hour) |
| PUT | `/api/schedule/role/<id>/rate` | ставка за час роли (для ЗП) |
| GET | `/api/schedule/hours-by-role?date_from=&date_to=` | часы по ролям + оплата за период (страница ЗП) |
| GET | `/api/schedule/<year>/<month>` | смены месяца |
| POST/PUT/DELETE | `/api/schedule/shift[/<id>]` | CRUD смены (+`start_time` HH:MM) |
| PUT | `/api/schedule/shift/<id>/fact` | факт минут (0..1440 или null) |
| GET/POST/DELETE | `/api/schedule/dayoff` | заявки на выходные |
| GET | `/api/schedule/audit/<year>/<month>[?limit=]` | журнал изменений за месяц (новые сверху) |
| GET | `/api/schedule/plans/<year>/<month>` | план/факт по дням (план read-only) |
| GET | `/api/schedule/widgets/<year>/<month>` | данные виджета «Нагрузка», money-free, без iiko — для редактора и просмотра |
| GET | `/api/schedule/summary/<year>/<month>` | сводка месяца + нагрузка (legacy, фронтом не используется) |
| GET | `/api/schedule/revenue/<date>` | план/факт дня по точкам |
| PUT | `/api/schedule/revenue/<date>/<loc>` | факт дня (`plan_revenue` игнорируется) |
| POST | `/api/schedule/revenue/sync/<date>` | legacy: наполнить фоллбэк-кэш факта |
| POST | `/api/schedule/revenue/sync-month/<y>/<m>` | legacy: то же за месяц |
| GET/POST | `/api/meeting-notes[...]` | заметки совещаний (без изменений) |
