# Мониторинг температуры по барам (`/temperature`)

## Что это

Страница `/temperature` показывает текущую температуру и влажность воздуха в каждом
из 4 баров: в зале стоит умный термо-гигрометр (Tuya / Smart Life), приложение
опрашивает Tuya Cloud и привязывает показания к каноническим барам
(`bolshoy`/`ligovskiy`/`kremenchugskaya`/`varshavskaya`).

Зачем: видеть на одном экране, не жарко/холодно ли в зале, не отвалился ли датчик,
и динамику за сутки (спарклайн). Данные обновляются автоматически.

Источник прототипа — рабочий Jupyter-ноутбук владельца (Tuya OpenAPI v2.0). При
переносе в проект: секреты вынесены в `.env`, добавлены кэш токена, корректный
масштаб из спецификации устройства, история в SQLite и фоновый опрос.

## Как работает

### 1. Авторизация и подпись (Tuya OpenAPI v2.0)

Каждый запрос подписывается HMAC-SHA256:

```
stringToSign = METHOD + "\n" + sha256(body) + "\n\n" + path
sign         = HMAC_SHA256(secret, accessId + [accessToken] + t + stringToSign).hex().upper()
```

- `body` для GET пустой, `sha256("")` константа; `t` — текущее время в миллисекундах.
- Для запроса токена `accessToken` пустой; для остальных — подставляется токен.
- Токен (`POST /v1.0/token?grant_type=1`) кэшируется в памяти процесса до
  `expire_time - 60 c`. При ответе «token invalid» (код 1010-1012) токен один раз
  принудительно обновляется и запрос повторяется.

### 2. Маппинг устройство -> бар

Константа `DEVICE_TO_VENUE` в `core/tuya_temperature.py`. Три бара однозначны
(`varsh`/`lig`/`krem`), четвёртый по элиминации: `Andr` (Андреевский, Васильевский
остров) = `bolshoy` (Большой пр. В.О). Переопределяется через env `TUYA_DEVICE_MAP`
(JSON `{"<device_id>": "<venue_key>"}`) без правок кода.

### 3. Масштаб показаний (важно для корректности)

Датчик отдаёт значение целым числом; реальное значение = `raw / 10**scale`, где
`scale` берётся **из спецификации устройства** (`GET /v1.0/devices/{id}/specifications`,
поле `status[].values.scale`), а не угадывается по величине.

- Температура: `va_temperature`, `scale=1` => `236 -> 23.6 C`.
- Влажность: `va_humidity`, `scale=0` => `62 -> 62 %` (не делим).

Почему не эвристика «если raw > 100, делить на 10»: на холоде `95` при `scale=1` —
это `9.5 C`, а эвристика показала бы `95 C`. Спецификация кэшируется на весь процесс
(не меняется). Фоллбэк, если спецификация недоступна: температура `scale=1`,
влажность/батарея `scale=0`.

### 4. Диапазоны окраски (`TEMP_BANDS`)

Единый источник правды — константа `TEMP_BANDS`; UI берёт её из API
(`meta.bands`), поэтому цвет карточки и легенда всегда совпадают с кодом. Это
температура **воздуха в зале**, границы — ориентир комфорта, не жёсткая норма:

| Диапазон | °C | Цвет |
|---|---|---|
| Холодно | < 16 | синий |
| Норма | 16–25 | зелёный |
| Тепло | 25–28 | жёлтый |
| Жарко | > 28 | красный |

### 5. История и дедуп под 2 воркера

Показания пишутся в `temperature.db` (SQLite, WAL). Первичный ключ
`(bar_key, bucket_ts)`, где `bucket_ts = (ts // interval_s) * interval_s` —
время показания, округлённое вниз до интервала опроса.

Под `gunicorn --workers 2` оба воркера опрашивают независимо, но `INSERT OR IGNORE`
на один бакет оставит ровно одну строку. Это проще и надёжнее lock-файлов (ср.
`core/open_check_scheduler.py`): если у одного воркера запрос к Tuya упал, значение
запишет другой. `ts` — это `update_time` устройства (когда датчик реально присылал
данные), поэтому история отражает фактические апдейты, без дубликатов плоской линии.

Путь к БД: `/kultura/temperature.db` на проде (persistent volume), `data/temperature.db`
локально (в `.gitignore`). Старше 90 дней подчищается (`prune`).

### 6. Фоновый опрос

`core/temperature_scheduler.py` — daemon-поток. Стартует из `app.py` только если
заданы `TUYA_ACCESS_ID/SECRET` и `TUYA_POLL_MINUTES > 0` (иначе фича тихо выключена).
Делает стартовый опрос сразу, далее каждые `TUYA_POLL_MINUTES` минут; раз в сутки
чистит историю.

### 7. API

| Endpoint | Описание |
|---|---|
| `GET /api/temperature/current` | Текущие показания. Живой опрос Tuya с кэшем 60 с; `?force=1` обходит кэш. Живой опрос пополняет историю. Если Tuya недоступна/не настроена — фоллбэк на последнее из БД (`source: store`). Ответ: `{bars:[...], meta:{bands, configured, poll_minutes}, updated_at, source}` |
| `GET /api/temperature/history?hours=24` | Точки за период (1–168 ч) для спарклайна: `{bars:{venue_key:[{ts,temperature,humidity}]}}` |

### 8. UI

Карточки 4 баров: крупная температура с окраской по диапазону, метка диапазона,
влажность, батарея (если датчик отдаёт; подсветка при ≤ 20 %), спарклайн за 24 ч,
время последнего показания и бейдж связи (`на связи` / `не на связи` / `давно` —
если показанию больше 2.5 интервалов опроса). Кнопка «Обновить» (force) и
автообновление раз в 60 с. Легенда с диапазонами.

### Переменные окружения

```
TUYA_ACCESS_ID=...        # Cloud-проект на iot.tuya.com
TUYA_ACCESS_SECRET=...
TUYA_ENDPOINT=https://openapi.tuyaeu.com   # EU/РФ; US — tuyaus.com, CN — tuyacn.com
TUYA_POLL_MINUTES=15      # 0 = фоновый опрос выключен (живой опрос на странице остаётся)
TUYA_DEVICE_MAP=          # (опц.) JSON-переопределение маппинга устройство->бар
```

На проде (Selectel VPS) задать те же переменные в окружении контейнера; секреты —
в рунбуке `C:\KULT\ctr\selectel-migration-credentials.md`.
