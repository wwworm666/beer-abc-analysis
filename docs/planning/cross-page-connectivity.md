# Аудит связности страниц: межстраничные переходы и UX-связки

> Анализ от 2026-06-29. Цель — найти полезные взаимосвязи между страницами сервиса
> (типа «Мой график → Расчёт ЗП с уже подставленным сотрудником и периодом»),
> улучшить удобство и убрать повторный ввод контекста. Каждая идея проверена против
> реального кода (routes/templates/static/js/core): реализуемость, не-дубль, реальная польза.
> Бесполезное/нереализуемое отброшено и вынесено в раздел «Не делать».
>
> Методика: 79 агентов, 6 фаз (профили 17 страниц → граф общих сущностей → генерация
> связок по хабам и персонам → дедуп → адверсариальная проверка против кода → синтез).
> Сводные данные: 30 связок прошли проверку, 20 отброшены, 29 недостающих элементов.

## Что это

Карта того, как страницы сервиса могли бы передавать контекст друг другу (бар, период,
сотрудник, продукт/кег, гость) вместо того, чтобы заставлять пользователя на каждой
странице заново выбирать бар и даты. Источник истины по переходам — этот файл; реализация
ведётся волнами (см. раздел 6).

## 1. Краткий вывод

Сервис состоит из 15+ страниц-силосов. Каждая страница самодостаточна, но межстраничный
контекст (бар, период, сотрудник, продукт, гость) почти нигде не передаётся: единственная
контентная навигация — это `/ -> /taps`, `/taps -> /taps/<id>`, `/menu -> /menu/edit` и
серверные редиректы `/bonus,/kpi -> /salary`. Всё остальное движение идёт через глобальный
сайдбар голыми `<a href="/draft">` без параметров, поэтому при каждом переходе пользователь
заново выбирает бар и даты. Корневых причин три:

- **(а) URL-контекст читают только 3 страницы из ~15** (`explorer.html` — полностью,
  `discounts.html` — лишь `?mode=`, `menu_editor.html` — `?id=`); все аналитические страницы
  стартуют анализ только по ручному клику «Анализировать». Проверено: `URLSearchParams`
  в шаблонах встречается только в этих трёх.
- **(б) Идентификаторы бара рассинхронизированы** между тремя подсистемами:
  `bolshoy/ligovskiy/...` на дашборде и explorer; `bar1..bar4` на taps/stocks/expiration
  (подтверждено: `bar1..bar4` в `routes/stocks.py`, `pages.py`, `dashboard.py`,
  `expiration.py`); русские имена `Большой пр. В.О` на packaging/draft/waiters/employee;
  `Варш/Крем/ВО/Лиг` в shifts.db.
- **(в) Сотрудник идентифицируется по имени** (`WaiterName` из OLAP), а стабильный
  `iiko_id` живёт только в schedule/auth и с OLAP-именами не связан.

Поэтому большинство ценных дрилл-даунов сегодня — мёртвый текст.

## 4. Сквозные пользовательские маршруты

### Маршрут 1 — Владелец утром (обход сети)
1. `/dashboard/widget` (PWA): видит % плана по 4 барам → **рвётся**: ни одного href наружу.
2. Хочет в проседающий бар → `/` вкладка Выручка → **рвётся**: дашборд не читает `?bar`.
3. От недобора к смене → `/waiters` или `/schedule` → **рвётся**: обе сбрасывают на «Все бары»/now().
4. Проверить акции на проседающем баре → `/discounts` → **рвётся**: читается только `?mode=`.
5. Проверить горящие сроки → `/expiration` → **рвётся**: `loadBoard()` читает только `bar-select.value`.

Развязывает: механизмы A + B, далее агрегатный `/api/morning-summary`
(revenue_completion + empty_taps + expiry_risk_rub на бар).

### Маршрут 2 — Бармен про себя (перед сменой)
1. `/schedule` борд «Что сейчас»: видит точку сегодня → **рвётся**: борд показывает ВСЕ 4 точки.
2. Краны своего бара → `/taps/<bar_id>` → **рвётся**: нет словаря `venue_key → bar1..bar4`.
3. Кран пустой → остаток/заказ `/stocks` → **рвётся**: `?bar`/`?tab` не парсятся.
4. Свои показатели/ЗП → `/employee`, `/salary` → **рвётся**: name-based без чтения URL.

Развязывает: словарь `venue_key→bar_id` (часть механизма B) + чтение `?bar/?tab`
на /stocks и /expiration (механизм A).

## 5. Недостающие элементы

Чего нет в коде, но требуется для связок (с указанием, с чем именно):

- **Чтение URL-параметров + автозапуск** на `/packaging`, `/draft`, `/waiters`, `/discounts`
  (есть только `?mode=`), `/employee` — нужно для ВСЕХ входящих deep-link.
- **Канонический ключ бара** (`bar_id` в `venues_config.py` + функции `key<->bar_id`;
  `short_name` через конвертер из shifts.db).
- **Резолвер `WaiterName(OLAP) -> employee_iiko_id`** (поверх `schedule_employees.iiko_id`
  + нормализация имён).
- **`renderEmployeeList`/`/api/employees`/`/api/waiter-analyze` отдают id рядом с именем.**
- **Чтение `?year/?month` в `static/js/schedule/common.js`** (сейчас `state.year/month` =
  `new Date()`, жёстко).
- **`get_hours_by_role_for_period` кладёт `emp_key/iiko_id` в результат** (`shifts_manager.py`
  — сейчас только `employee_name`).
- **Кликабельность статичных элементов-источников**: ячейки `/explorer`, строки ТОП-5
  гостей `/monthly-report`, ячейки «План/Факт по дням» `/schedule` (`td.pf-tt`), строки
  «Сроки годности» `/stocks`, текст рекомендаций `/expiration`, href наружу в
  `revenue_widget.html`.
- **`Chart.js onClick`** на графиках `/monthly-report`.
- **Общий резолвер `beer name -> keg_id/sku`** (нормализация из `stocks.py` + `taps.py
  find_beer_info` в `core/`).
- **`is_packaged` в `/api/discount-analyze`** (по `DishGroup.TopParent`) — чтобы
  скидка→/packaging не вела в пустоту для еды/розлива.
- **Агрегатный `/api/morning-summary`** (revenue + empty_taps + expiry_risk на бар).
- **Профиль/карточка гостя как страница** — её НЕТ; ТОП-5 гостей и модалка гостя — тупик.
  Минимальный суррогат: `/discounts?mode=rfm&card=`.

## 6. Дорожная карта (ценность/усилие)

### Волна 1 — фундамент (делать первым, разблокирует всё)
1. **Механизм A** (URL-контекст + автозапуск): вынести `period_url.js` из explorer, подключить
   на 5 страницах. Эффорт M, ценность максимальная.
2. **Механизм B** (канонический ключ бара): `venue_config.py` + замена хардкодов. Эффорт M,
   предусловие всех bar-связок.

### Волна 2 — быстрые победы (S/M, высокая ценность, опираются на волну 1)
3. `/waiters -> /employee` по имени (S).
4. `/packaging <-> /draft` переключатель «то же по разливному» (S).
5. `/draft -> /waiters` «кто наливал» (`/api/waiter-analyze` уже есть) (S–M).
6. `/draft -> /stocks?tab=order-board` остаток/заказ (S–M).

8. `/taps -> /stocks,/expiration` остатки/сроки своего бара (S).
9. `/salary -> /schedule?year=&month=` по предупреждению «N смен без часов» (M) — закрывает
   главный тупик ЗП.
10. `/dashboard/widget -> /?bar=&tab=analytics` — оживить виджет (M).

### Волна 3 — крупное (L)
11. **Механизм D** (хаб сотрудника): резолвер `WaiterName->iiko_id`, id в `renderEmployeeList`.
12. **Механизм C** (контекстное меню «Открыть в…» на пиве).

14. `/explorer` clickable-ячейки → packaging/draft с границами бакета.

16. Утренняя сводка-мост: `/api/morning-summary` + deep-links по 4 барам.

### Не делать (отброшено по коду, не по приоритету)
- Любые связки с `focus_iiko_id` — механизма в репо нет (0 совпадений).
- `/employee -> /schedule` на посылке «часы employee = факт минут графика» — это
  `total_hours` из кассовых смен iiko, другая семантика.
- `/menu № крана -> /taps` — меню глобальное, номера кранов коллидируют между барами.
