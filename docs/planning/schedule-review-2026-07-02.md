# Ревью страницы "График смен" — 2026-07-02

Многоагентное ревью (5 направлений: логика бэкенда, конкурентность, фронтенд,
производительность, интеграции .ics/выручка). Каждая находка адверсариально
проверена отдельным агентом по коду; ниже только подтверждённые (33 уникальных
из 38, 1 опровергнута, 4 — дубли между направлениями).

Файлы модуля: routes/schedule.py, core/shifts_manager.py, core/schedule_plans.py,
core/calendar_ics.py, static/js/schedule/{common,screens,view}.js, templates/schedule.html.

## Деньги и целостность данных

### 1. [СРЕДНИЙ] Модалка редактирования смены при сохранении безусловно шлёт PUT /fact и затирает введённый барменом факт часов null-ом

**Где:** `static/js/schedule/view.js:446`

onShiftFormSubmit всегда цепочкой отправляет PUT /api/schedule/shift/<id> и затем PUT /api/schedule/shift/<id>/fact с содержимым поля «Факт» из модалки; пустое поле -> fact_minutes: null, а сервер (set_shift_fact) трактует null как «очистить». Сценарий: менеджер открывает модалку смены (поле факта пустое) и держит её открытой; в это время бармен в конце смены вводит факт 10:30 (единственный источник часов оплаты, по API не восстановить); менеджер меняет роль/время и жмёт «Сохранить» -> его stale null уходит в /fact и стирает 630 минут бармена. ЗП за эту смену молча теряется (смена лишь попадёт в shifts_without_fact, если кто-то заметит). Ни optimistic-lock, ни сравнения «изменилось ли поле» нет ни на клиенте, ни на сервере; окно гонки — всё время, пока открыта модалка.

### 2. [СРЕДНИЙ] Маппинг кассовой смены на точку — подстрока без break: одна смена может засчитаться нескольким точкам (двойной счёт выручки)

**Где:** `routes/schedule.py:636`

В _aggregate_cash_shifts_by_day цикл по locations не прерывается после первого совпадения: если pointOfSale кассовой смены содержит и name/short_name точки A, и short_name точки B, полная выручка смены прибавится ОБЕИМ точкам, и sync запишет задвоенный факт в daily_revenue. Матчинг подстрокой по короткому short_name особенно хрупок: двухбуквенное «ВО» встречается внутри обычных слов (напр. терминал/точка с «Ново...» в названии или переименование кассы в iiko), и выручка чужой кассы засчитается точке ВО. Сценарий провала: в iiko добавляют новую точку/переименовывают кассу так, что в pointOfSale попадает подстрока чужого short_name -> sync-month молча пишет неверный факт по всем дням месяца, updated_rows это никак не выдаёт. Нужен break после первого матча и матчинг по стабильному departmentId вместо подстрок.

### 3. [СРЕДНИЙ] revenue/sync и sync-month не проверяют результат authenticate() — при недоступном iiko отдают 200 «updated: 0» вместо ошибки

**Где:** `routes/schedule.py:652`

В schedule_sync_revenue (строка 652) и schedule_sync_revenue_month (строка 697) `iiko.authenticate()` вызывается без проверки возврата (сравни с employees/sync, где False -> 503). При неудачной авторизации token=None, get_cash_shifts возвращает [] (core/iiko_api.py:285-287), by_day пустой -> ответ 200 {"updated": 0} и запись в журнал «обновлено точек 0». Сценарий: iiko лежит, владелец жмёт синк, чтобы наполнить фоллбэк-кэш (единственный момент, когда этот кэш и нужен) — получает «успех», думает, что выручки просто не было, кэш остаётся пустым/устаревшим без какого-либо сигнала об ошибке.

### 4. [СРЕДНИЙ] Синк сотрудников при сбое iiko-справочника (пустой ответ) запускает деструктивную чистку реестра

**Где:** `routes/schedule.py:141`

IikoAPI.get_employees() при любой не-auth ошибке (HTTP != 200, таймаут, битый XML) возвращает [] (core/iiko_api.py:573-595), а не None. Роут делает `emps = iiko.get_employees() or []` и без проверки на пустоту зовёт sync_employees(pairs=[]). В sync_employees шаг 4 (shifts_manager.py:860-864) выполняется всегда: DELETE FROM schedule_employees WHERE iiko_id IS NULL AND name NOT IN (смены без employee_id). Сценарий: iiko вернул 500/таймаут на /employees при живой авторизации -> pairs пустой -> все ещё не привязанные к id строки реестра (legacy-имена без смен) удаляются вместе с short_label/active/sort_order, хотя данные из iiko вообще не пришли. Нужен guard `if not pairs: return 503` перед sync_employees.

### 5. [СРЕДНИЙ] Очистить факт выручки через API невозможно: COALESCE сохраняет старое значение, а журнал пишет «очищен»

**Где:** `core/shifts_manager.py:684`

update_revenue делает `fact_revenue = COALESCE(?, fact_revenue)` — передача None сознательно означает «оставить прежнее». Но роут PUT /api/schedule/revenue/<date>/<loc> (routes/schedule.py:604-616) при fact_revenue=null всё равно пишет в журнал «Факт выручки ... очищен». Сценарий: пользователь ввёл ошибочный факт (например 999999), затем отправляет null чтобы стереть -> ответ 200, аудит фиксирует «очищен», но в daily_revenue остаётся 999999, и при недоступном iiko это значение продолжает попадать в план/факт и «Выполнение %». Ни одного пути сбросить факт в NULL в API нет — журнал лжёт о выполненном действии.

### 6. [СРЕДНИЙ] PUT /api/schedule/revenue не валидирует fact_revenue как число — нечисловое значение потом роняет /plans и /summary в 500

**Где:** `routes/schedule.py:604`

fact = data.get('fact_revenue') пишется в БД как есть (update_revenue), без проверки типа/знака — в отличие от соседних роутов (rate, fact_minutes). SQLite с REAL-аффинити сохранит неконвертируемую строку как TEXT. Сценарий: PUT {"fact_revenue": "12 000"} (строка с пробелом из поля ввода) -> 200 OK; затем, когда iiko недоступен и build_month_plans берёт фоллбэк из daily_revenue, `float(fact)` в core/schedule_plans.py:102 бросает ValueError -> GET /api/schedule/plans/<y>/<m> и /summary отдают 500 на ВЕСЬ месяц, пока строку не вычистят из БД руками. Отрицательные значения тоже принимаются молча.

### 7. [СРЕДНИЙ] Кисть «Выходной»: смены удаляются ДО постановки выходного, при сбое POST смены потеряны без отката

**Где:** `static/js/schedule/view.js:371`

toggleDayOff строит цепочку: сначала последовательные DELETE всех смен клетки, затем POST /api/schedule/dayoff. Сценарий: владелец кликает «Выходной» по клетке со сменой, DELETE проходит, а POST падает (обрыв сети, 500, перезапуск gunicorn). Итог: смена удалена, выходной НЕ поставлен, показан общий тост «Ошибка: …», отката нет — после reload клетка просто пустая, и если владелец не заметил, сотрудник теряет смену без какой-либо пометки. Введённый факт часов удалённой смены тоже безвозвратно теряется. Правильнее ставить выходной первым (или атомарной серверной операцией), а смены снимать после успеха.

### 8. [НИЗКИЙ] Sync пишет в daily_revenue сумму cash+card из кассовых смен, а «Факт» на странице — из OLAP: при падении iiko фоллбэк показывает цифры другой методологии

**Где:** `routes/schedule.py:633`

Живой факт в /api/schedule/plans берётся из OLAP get_store_daily_revenue (как на дашборде), а sync-эндпоинты сохраняют в daily_revenue выручку по формуле cashOrders + cardOrders из v2/cashshifts (без прочих типов оплат — безнал/бонусы/доставка, и по openDate смены, а не по учётному дню OLAP). Сценарий: день закрыт с оплатами не только нал/карта; пока iiko жив, менеджер видит факт X (OLAP); iiko упал — build_month_plans падает на manual.get('fact_revenue') и та же ячейка показывает Y < X без какой-либо пометки, «Выполнение %» и «Ожидаемая» скачут. Нарушается принцип детерминированности: один и тот же день даёт разные цифры в зависимости от доступности iiko. Правильнее фоллбэк-кэш наполнять тем же OLAP-источником (сохранять olap_fact при удачных запросах), а cash-shift-sync оставить только как явный ручной инструмент с иной подписью.


## Гонки и конкурентность (Gunicorn 2 workers)

### 9. [ВЫСОКИЙ] Гонка fetch при переключении месяцев: устаревший ответ перетирает state, и сетка нового месяца рендерится пустой

**Где:** `static/js/schedule/view.js:131`

reload() никак не сериализуется и не помечает запросы токеном/AbortController. Сценарий: пользователь на июле кликает «След» (стартует загрузка августа: /api/schedule/2026/8 + dayoff + plans), не дождавшись, кликает ещё раз (стартует сентябрь). Если ответ августа приходит ПОСЛЕ ответа сентября (обычная ситуация при медленной сети/тяжёлом OLAP-плане), loadMonthData августа перезаписывает state.shifts/state.dayOffs августовскими данными и вызывает renderAll(). Но state.month уже = 9, поэтому renderEditLanes строит сетку сентября, а фильтр s.date === '2026-09-…' не находит ни одной августовской смены — весь сентябрь выглядит пустым (и «дыры покрытия» горят красным), хотя данные на сервере есть. Владелец может начать заново «закрашивать» месяц, плодя дубли. Дополнительно: Promise.all внутри одного reload пишет plans и shifts независимо, так что возможен и микс «план сентября + смены августа»; первый завершившийся reload вызывает S.showLoading(false), пряча спиннер, пока второй ещё грузится. Тот же класс гонки задевает reload после сохранения факта/смены: навигация по месяцам во время save-reload не блокируется флагом saving. Нужен счётчик поколения запроса (сравнивать перед записью в state) или AbortController.

### 10. [СРЕДНИЙ] PUT смены игнорирует результат update_shift: правка одновременно удалённой смены возвращает ok:true без ошибки и без записи в журнал

**Где:** `routes/schedule.py:302`

schedule_update_shift вызывает shifts_mgr.update_shift(...) и не проверяет возвращаемое значение (rowcount). Сценарий двух менеджеров: A через кисть «перекрашивает» смену (replaceShift шлёт один PUT c location_id/role_id/start_time), B в соседней вкладке в тот же момент её удаляет (DELETE в другом gunicorn-воркере). PUT A попадает на уже удалённый id: UPDATE затрагивает 0 строк, но роут отвечает {ok:true}, фронт показывает тост «Перекрашено», after=None -> запись в schedule_audit не создаётся. A уверен, что изменение сохранено, а журнал (заявленная «защита от саботажа») не содержит следов его действия. Корректно — 404, как это уже сделано в /fact (set_shift_fact проверяет rowcount).

### 11. [СРЕДНИЙ] Модалка смены сохраняет двумя последовательными PUT: при сбое второго (факт) роль/время уже изменены, но UI сообщает об ошибке и не перерисовывается

**Где:** `static/js/schedule/view.js:448`

onShiftFormSubmit шлёт PUT /shift/{id} (роль+время), затем PUT /shift/{id}/fact. Сценарий: владелец меняет в модалке роль и факт часов, первый PUT успешен, второй падает (сеть/ошибка валидации на сервере). Пользователь видит тост «Ошибка: …» и открытую модалку — логично считает, что НИЧЕГО не сохранилось, закрывает её. На деле роль/время уже изменены на сервере, а reload() в ветке ошибки не вызывается, так что сетка продолжает показывать старую роль до следующего действия — рассинхрон UI и сервера плюс частично применённая правка без отката. Нужен либо один атомарный эндпоинт, либо reload() и честное сообщение «время сохранено, факт — нет» в catch.

### 12. [НИЗКИЙ] Снимки для журнала (before/after) читаются вне транзакции изменения — при одновременных действиях двух менеджеров журнал искажается

**Где:** `routes/schedule.py:356`

Во всех мутирующих роутах паттерн: get_shift() -> изменение -> get_shift() -> _audit(), три отдельные транзакции в разных воркерах. Сценарии: (1) два менеджера одновременно жмут «Удалить» на одной смене — оба успевают прочитать снимок до удаления (delete_shift не проверяется на rowcount), в журнале появляются ДВЕ записи «Удалена смена» для одной смены; (2) A и B одновременно PUT-ят одну смену: before A уже содержит правку B (или наоборот), и diff в summary («роль X -> Y», «время ... -> ...») приписывает автору чужое изменение либо пишет «без изменений». Данные в shifts не портятся (last-write-wins на уровне UPDATE), но журнал, на котором держится разбор «кто что менял», для таких пересечений врёт. Лечится чтением снимка и записью аудита в одной транзакции с изменением либо хотя бы проверкой rowcount у delete.

### 13. [НИЗКИЙ] Seed свежей БД не идемпотентен между процессами: два воркера одновременно вставляют locations/roles -> IntegrityError и падение воркера на старте

**Где:** `core/shifts_manager.py:296`

_seed_data делает check-then-insert (SELECT COUNT(*) -> executemany INSERT) без OR IGNORE. На пустой БД (первый деплой на новый том, восстановление, тестовый стенд) оба gunicorn-воркера параллельно проходят _init_database: оба видят COUNT=0, оба вставляют 4 точки; у второго executemany падает на UNIQUE(locations.name) -> необработанный sqlite3.IntegrityError -> ShiftsManager не создаётся (extensions.py импортирует его на уровне модуля) -> воркер не поднимается, gunicorn его перезапускает (со второго раза стартует, т.к. данные уже есть). Гонка ALTER TABLE в _ensure_columns предусмотрительно проглатывается, а точно такая же гонка seed-вставок — нет. Лечится INSERT OR IGNORE либо перехватом IntegrityError по аналогии с duplicate column.


## Производительность

### 14. [ВЫСОКИЙ] Каждое изменение одной смены вызывает полную перезагрузку месяца (5 HTTP-запросов) и полную перерисовку всей страницы

**Где:** `static/js/schedule/view.js:128`

Сценарий: владелец красит график кистью. Один клик по клетке -> POST /shift -> reload(): GET /api/schedule/{y}/{m} (все смены месяца), GET /dayoff, GET /plans (может синхронно ходить в iiko OLAP при промахе 10-мин кэша), затем renderAll() перерисовывает ВСЁ (myShifts, todayBoard, lanes ~30 сотрудников x 31 день с ~930 листенерами, legend, таблицу План/Факт, доску пожеланий, реестр сотрудников — который внутри renderEmployeesAdmin() делает ещё один GET /employees?all=1), затем loadFeed() — GET /audit. Итого 5 запросов и полный ребилд DOM на один клик. Флаг saving=true снимается только после завершения всего reload (строка 313), поэтому следующий клик кистью заблокирован до конца цепочки: раскраска месяца (~60 кликов) превращается в 60 x (латентность 5 запросов + ребилд). На мобильном при вводе факта часов — видимая задержка и мигание всей страницы. Бонус-эффект: renderWishesBoard() пересоздаёт textarea пожеланий — текст, набранный за последние <500 мс (дебаунс сохранения), теряется при любом действии со сменой. Достаточно локально обновить state.shifts (POST уже возвращает id, PUT/DELETE — известны) и перерисовать только lanes + виджеты, а plans/feed/реестр не перезапрашивать вовсе.

### 15. [ВЫСОКИЙ] GET /api/schedule/plans синхронно ходит в iiko OLAP при промахе кэша — сетка и каждый клик редактора блокируются на латентность iiko

**Где:** `routes/schedule.py:500`

Сценарий: пользователь открывает /schedule (или делает любую правку — фронт после неё дергает /plans, см. view.js reload) спустя >10 минут после прошлого обращения. _month_inputs -> _fetch_month_fact_olap -> cached_olap(TTL 10 мин, промах) -> OlapReports.connect() + get_store_daily_revenue за весь месяц — синхронный сетевой запрос к iiko прямо в обработчике GET. Ответ /plans (а с ним спиннер страницы и снятие блокировки saving после клика кистью) ждёт секунды, при недоступном iiko — до таймаута соединения. Особенно бессмысленно для ПРОШЛЫХ месяцев: их факт заморожен и уже лежит в daily_revenue, но листание истории назад всё равно дергает OLAP на каждый месяц (каждый — отдельный ключ кэша). Лечится: для месяцев < текущего брать факт только из сохранённого daily_revenue (фоллбэк уже реализован в build_month_plans), а для текущего — отдавать протухший кэш сразу и освежать фоном, либо хотя бы не дергать /plans после каждой правки смены.

### 16. [СРЕДНИЙ] Пустой/None результат OLAP не кэшируется — при недоступном iiko каждый запрос /plans и /summary заново долбит iiko с таймаутами до 90+ секунд

**Где:** `routes/schedule.py:477`

Сценарий: iiko лёг (или запрошен будущий месяц без выручки). _fetch в _fetch_month_fact_olap возвращает None (или пустой dict), cached_olap falsy-результат не кэширует (extensions.py:126: 'if data:'). Значит КАЖДЫЙ запрос /api/schedule/plans и /api/schedule/summary (оба зовут _month_inputs) заново открывает OLAP-подключение; IikoAPI.authenticate ретраит до 3 попыток с timeout=30 и backoff — открытие страницы графика при лежащем iiko блокируется на минуту-полторы на каждый из двух параллельных запросов, воркеры gunicorn (их всего 2) заняты, весь сайт деградирует. Single-flight в cached_olap держит лок только на время одного fetch — следующий запрос снова идёт в iiko. То же с _fetch_month_iiko_hours при неудачной авторизации (там falsy-обёртка сделана только для успешного пустого месяца, строка 545). Нужен негативный кэш с коротким TTL для ошибок iiko.

### 17. [СРЕДНИЙ] N+1 к SQLite в месячном синке выручки: ~155 открытий соединения и отдельных commit на один POST sync-month

**Где:** `core/shifts_manager.py:694`

Сценарий: POST /api/schedule/revenue/sync-month/{y}/{m} (routes/schedule.py:705-709) в цикле по ~31 дню вызывает sync_revenue_from_iiko(), который на КАЖДЫЙ день заново вызывает self.get_locations() (новое соединение + 3 PRAGMA) и затем update_revenue() на каждую из 4 точек — ещё по одному соединению с PRAGMA journal_mode=WAL/busy_timeout/synchronous/foreign_keys и отдельным commit. Итого на месяц: ~31 чтение locations + ~124 отдельных write-транзакции = ~155 открытий соединения и ~124 fsync-коммита вместо одной транзакции. Пользователь, нажавший синк месяца, ждёт заметно дольше необходимого, а под gunicorn с 2 воркерами каждая из 124 микро-транзакций — отдельная точка конкуренции за writer-lock (busy_timeout ретраи). Лечится: locations читать один раз перед циклом, а все upsert'ы месяца выполнять в одном соединении/одной транзакции (executemany + один commit).

### 18. [СРЕДНИЙ] Waterfall при первой загрузке страницы: 4 последовательных раунда запросов, хотя независимые можно параллелить

**Где:** `static/js/schedule/view.js:102`

Сейчас страница показывается через 4 последовательных RTT вместо 2: при мобильном RTT 200-300 мс это лишние 0.5-1 c до первого отображения сетки. Дополнительно updateEmployee() (строка 638) после PUT делает GET /employees и полный renderAll — который через renderEmployeesAdmin повторно запрашивает тех же сотрудников (двойной запрос) и перерисовывает всю страницу ради правки одной метки в реестре.

### 19. [НИЗКИЙ] Фильтр журнала по месяцу через LIKE не использует индекс entity_date — полный скан растущей append-only таблицы

**Где:** `core/shifts_manager.py:942`

Сценарий: GET /api/schedule/audit/{y}/{m} для прошлого месяца. Запрос WHERE entity_date LIKE 'YYYY-MM-%' — в SQLite LIKE по умолчанию регистронезависим, поэтому idx_audit_entity_date для префиксного LIKE НЕ применяется; план — скан по id DESC с фильтром. Лечится заменой LIKE на диапазон: entity_date >= 'YYYY-MM-01' AND entity_date < 'YYYY-(MM+1)-01' — тогда работает существующий индекс. Сейчас эффект малый (таблица молодая), потому severity low.

### 26. [СРЕДНИЙ] DTSTAMP в /schedule/cal.ics — наивное серверное время (МСК) с суффиксом Z: заявлен UTC, реально +3 часа

**Где:** `routes/schedule.py:440`

Роут вызывает build_calendar(name, shifts, dtstamp=datetime.now()) — наивное локальное время сервера (на проде МСК). build_calendar форматирует его как '%Y%m%dT%H%M%SZ', то есть объявляет UTC. Файл, скачанный в 14:00 МСК, содержит DTSTAMP:...T140000Z (реальный UTC — 11:00). Ирония: дефолт внутри core/calendar_ics.py:102 специально исправлен на datetime.now(timezone.utc) с комментарием ровно про этот баг, но роут передаёт аргумент и тем самым отключает исправление. Достаточно убрать dtstamp=datetime.now() или передавать datetime.now(timezone.utc). Тест test_dtstamp_passthrough_and_count (tests/test_calendar_ics.py:16,107) закрепляет неверное поведение: передаёт наивный datetime и ожидает суффикс Z. Практическое влияние: клиенты, использующие DTSTAMP при разрешении конфликтов повторного импорта по одинаковому UID, сравнивают искажённые времена.

### 27. [НИЗКИЙ] Плавающее локальное время без VTIMEZONE: при импорте в Google Calendar с не-московским часовым поясом календаря смены встают на неверное абсолютное время

**Где:** `core/calendar_ics.py:133`

Решение в докстринге объявлено осознанным («не уезжает при смене настроек телефона»), но для сети, жёстко привязанной к Москве, семантика обратная: правильно фиксировать TZID=Europe/Moscow + VTIMEZONE, тогда время стабильно именно по месту работы. Помечаю low, так как весь персонал фактически в одном поясе.

### 28. [НИЗКИЙ] _escape не обрабатывает символ CR — перевод строки \r\n в данных даёт нелегальный управляющий символ в .ics

**Где:** `core/calendar_ics.py:73`

Экранируется только \n (RFC 5545 3.3.11 требует убрать все CR/LF из значения). Если в location_name, role_name или display_name пользователя (X-WR-CALNAME) попадёт текст с Windows-переводом строки \r\n (копипаста при заведении сотрудника/точки), после замены останется голый \r внутри строки свойства: 'SUMMARY:Первая\r\\nВторая'. ## Идеи для развития (продакт-агент)

Уже отвергнутое из бэклога (копия недели, undo, ТК-варнинги, SPLH, TG) исключено.

### Подсказка «кого поставить» при клике на дыру (трудоёмкость: средняя)

Менеджер видит незакрытую точку, но перебирает сотрудников в голове. По клику на дыру в «Покрытии точек» показать ранжированный список свободных: без смены и без выходного в этот день, отсортированных по недобору к норме 15 смен и без серии 5+ подряд. Все данные уже в state (shifts, dayOffs, нагрузка), сортировка детерминированная, клик по имени — сразу назначить кистью. Ускоряет закрытие дыр с «вспомнить и проскроллить» до двух кликов.

### Прогноз ФОТ месяца по графику в «План / Факт» (трудоёмкость: средняя)

Владелец не видит, сколько зарплаты «обещает» составленный график, пока месяц не закончился. Считать: по прошедшим сменам — факт часов × ставка роли (уже есть в get_hours_by_role_for_period), по будущим — плановая длительность (день 9ч / вечер 5ч, те же константы, что в calendar_ics) × ставка. Показать строкой в шапке «План / Факт»: «ФОТ ~N ₽ · X% от плана выручки» с формулой в тултипе. Менеджер сразу видит, что лишняя вечерняя смена в тихий вторник съедает маржу — до того, как она отработана.

### Проверка правдоподобности факта часов (трудоёмкость: малая)

Факт часов — единственный источник ЗП, а валидация сейчас только 0..1440 минут: опечатка «1:00» вместо «10:00» молча уезжает в зарплату. При сохранении сравнивать с плановой длительностью роли (день 9ч / вечер 5ч): отклонение больше чем на 3 часа в любую сторону — confirm «Вечерняя смена обычно 5 ч, вы ввели 12:00. Точно?». Границы — явные константы, чистая функция на фронте, ничего не блокирует, только страхует деньги.

### Кнопка «Отработал по плану» в модалке факта (трудоёмкость: малая)

Большинство смен отработаны ровно по стандарту, но бармен каждый раз набирает «9:00» руками — отсюда забытые смены в «Без часов». В модалку факта добавить одну кнопку с предзаполнением плановой длительности роли (день 9:00 / вечер 5:00, значение видно на кнопке — бармен подтверждает, а не слепо жмёт). Пишет тот же PUT /fact, попадает в журнал как обычно. Меньше трения — меньше дыр в данных для ЗП.

### Бейдж «твой график изменился» для бармена (трудоёмкость: малая)

Владелец переставил смену — бармен узнаёт об этом, только случайно заметив сетку. Журнал изменений уже пишет всё: фильтровать audit по employee_name/датам смен текущего пользователя, хранить ts последнего визита в localStorage и на экране «Мои смены» показывать плашку «С прошлого визита: смена 14.07 перенесена на Лиг» (записи из summary как есть). ### Учёт пиковых Пт/Сб: счётчик в «Нагрузке» + дыры вечера на пик (трудоёмкость: средняя)

Пт-Сб — пик (вес плана 2.0), но «Нагрузка» считает все смены одинаково, а предупреждение о дыре срабатывает только на точку совсем без смен. Две правки: колонка «Пик» в нагрузке (число смен Пт/Сб — видно, кто пашет все пятницы, а кто ни одной — справедливость ротации); в «Покрытии точек» помечать Пт/Сб без вечерней смены как янтарную полудыру (правило явное: пик = dow 5/6, тот же критерий, что тёплый фон выходных).

### Пожелания под рукой при назначении смены (трудоёмкость: малая)

Пожелания («без понедельников», «2,3 выходной») лежат в свёрнутом блоке внизу — в момент назначения кистью менеджер их не видит и узнаёт о конфликте от обиженного бармена. Показать иконку-маркер у имени сотрудника в полосах (есть непустое пожелание) с текстом в тултипе, и добавлять текст пожелания в confirm при назначении (рядом с уже существующим предупреждением о выходном). wishes уже загружены в view.js — чистый рендер без новых запросов.

### Личный итог месяца на экране «Мои смены» (трудоёмкость: средняя)

Бармен не видит, сколько заработал: считает часы на бумажке или спрашивает владельца. На мобильном экране добавить карточку: «Заработано: Σ(факт часов × ставка роли) · Прогноз до конца месяца: + плановые часы будущих смен × ставка», формула в тултипе (принцип прозрачных расчётов). Данные — свои смены из state + ставки из /roles (уже отдаются с rate_per_hour). Прозрачность ЗП снижает вопросы и мотивирует закрывать факт часов вовремя.
