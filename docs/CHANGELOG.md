### 2026-07-11 — Фикс сроков на /expiration: проданные партии давали ложный «ближайший срок»

Владелец (пример Кременчугской): Вайнштефан Хефе показывал 2026-07-15, на бутылке

Юнит-прогон на продовых партиях: все 3 проблемные позиции дали даты с этикеток,
верные позиции (Konix, Mahou) не изменились. Формула задокументирована в
docs/expiration.md.

### 2026-07-11 — Фикс: авторефреш ЧЗ молча стоял с 26.06 (auth-гейт 401)

Владелец заметил, что «Данные ЧЗ» на /expiration застыли на 26 июня.

**Причина:** планировщик `core/chz_scheduler.py` дёргал refresh через
`POST http://127.0.0.1:10000/api/chz/refresh` без сессии, а глобальный auth-гейт
(добавлен 23.06) отдаёт на `/api/*` без входа 401 → подпроцесс search-stock не
стартовал. Ручная кнопка работала (есть сессия), поэтому последнее обновление 26.06
было ручным. Диагностика: `refresh.log` на volume не писался с 22.06, при этом суточные
локи планировщика за 9–11 июля присутствовали (планировщик срабатывал, но получал 401).

**Что сделано:**
- Тело refresh вынесено в `start_chz_refresh()` в `routes/stocks.py` (возвращает
  `(result, http_code)`); HTTP-эндпоинт `/api/chz/refresh` — тонкая обёртка над ней.
- `core/chz_scheduler.py` зовёт `start_chz_refresh()` НАПРЯМУЮ (ленивый импорт),
  минуя auth-гейт; убран `urllib`-POST и `LOCAL_REFRESH_URL`. Cross-worker file-lock
  общий — кнопка и планировщик не запускаются параллельно.
- Урок записан в `docs/lessons.md`.

**Файлы:** `routes/stocks.py`, `core/chz_scheduler.py`, `docs/lessons.md`.

### 2026-07-04 — График: касса на смене (замена чеклиста кассовой дисциплины)

Бармен закрывает смену вместе с кассой (ручной пересчёт, **без iiko**): в модалке
факта — тумблеры «были траты из кассы» (сумма + заметка «на что») и «была
инкассация» (сумма), плюс «наличные в кассе на конец смены».

- **Схема v7** (аддитивно, автобэкап `backup_v6`): `shifts.cash_expense_kop`,
  `cash_expense_note`, `cash_collection_kop`, `cash_end_kop` — деньги в **копейках**
  (INTEGER, точно). `NULL` = не заполнено, `0` = «не было».
- Кассу сдаёт **дневной бармен** — блок показан только для дневной смены; вечерняя
  — только часы. **Не блокирует** ввод часов, незакрытая касса помечается («не
  сдана»). Панель дня показывает кассу по бару за день.
- API `PUT /api/schedule/shift/<id>/cash` (рубли -> копейки, потолок 1 млн ₽,
  журнал `cash_set`/`cash_clear`). Метод `ShiftsManager.set_shift_cash`.
- Блок **«Касса за месяц»** (`renderCashHistory`): read-only история сдач кассы
  месяца + итог трат/инкассаций. Клик по смене на мобильном открывает окно.
- Файлы: `core/shifts_manager.py`, `routes/schedule.py`, `templates/schedule.html`,

### 2026-07-01 — Месячный отчёт: читаемость графиков (по запросу владельца)

Четыре правки UX на `/monthly-report` (`static/js/dashboard/modules/monthly_report.js`,
`core/monthly_report.py`, `templates/dashboard/monthly_tab.html`):

1. **Линейные графики без «Общей».** На графиках-наложениях (выручка, наценка %, маржа,
   чеки, средний чек, баллы, новые гости) убрана линия «Общая»: её сумма кратно выше
   отдельных баров и сплющивала их линии у нижней границы — масштаб был нечитаем.
   Новый `LINE_BARS` (BARS без `all`), легенда сверху теперь описывает 4 бара. «Общая»
   остаётся выбором в выпадающих списках столбчатых/категорийных графиков.
2. **Проценты внутри накопительных столбцов.** Плагин `percentInStackPlugin` вписывает по
   центру каждого сегмента его долю от суммы столбца (%). Сумму видно по оси Y, пропорцию —
   внутри. Тонкие сегменты (< 6% или < 14px) без подписи, чтобы не было каши.
3. **Расход в штуках** уже работал — переключатель Деньги/Штуки (`units_draft`/`units_packaged`
   в кэше). Дополнений не потребовалось.
4. **ТОП гостей 5 -> 10** (`_compute_topguests` limit=10, заголовок «ТОП-10»). NB: закрытые
   месяцы в кэше `topguests` хранят по 5 — для истории по 10 нужен принудительный пересчёт
   (`?full=1` за текущий год, `refresh_all(force=True)` за прошлые). Новые месяцы — сразу 10.

### 2026-06-24 — Оплата часов по ролям: график как источник часов ЗП (схема v5)

Владелец: на смену могут выходить двое — один на полную (день, с 14:00), другой
с вечера (18:00), и ставка за час у них разная. Плюс решение: **iiko больше не
источник часов оплаты — единственный верный источник это часы из графика.**

**Модель (выбрана владельцем):** ставка — у роли (`roles.rate_per_hour`), часы —
факт по каждой смене из графика (`shifts.fact_minutes`). Оплата сотрудника =
`Σ по сменам: часы смены × ставку её роли`. Полная смена = «бармен», вечер =
«второй бармен» (своя ставка), стажёр — своя.

**Что сделано:**
- `shifts.db` v4 -> **v5**: `roles.rate_per_hour` (additive, DEFAULT 300 = старая
  единая ставка, поэтому миграция оплату не меняет). Методы `set_role_rate`,
  `get_hours_by_role_for_period` (группирует смены периода по сотруднику+роли,
  `fact_minutes`-NULL не суммируется, `pay = часы × ставка`, отдаёт
  `shifts_without_fact` для подсветки пробелов).
- Роуты: `GET /api/schedule/hours-by-role?date_from=&date_to=`,
  `PUT /api/schedule/role/<id>/rate`.
- Страница ЗП (`templates/bonus.html`): убран хардкод `HOURLY_RATE=300` и ручной
  ввод часов; часы теперь read-only из графика, разбивкой по ролям (часы × ставка);
  предупреждение «N смен без часов»; новый блок «Ставки за час по ролям». В список
  ЗП добавляются и те, кто есть в графике, но без данных iiko.
- iiko-часы (`total_hours`) остаются метрикой Employee Dashboard (выручка/час), но

### 2026-06-24 — Журнал изменений графика «кто что менял» (на базе авторизации)

Первая фича, опирающаяся на свежую авторизацию: у каждой правки графика теперь
есть автор. Раньше график правили анонимно — было не видно, кто переставил смену
или проставил часы.

**Что сделано:**
- `shifts.db` схема **v3 -> v4**: таблица `schedule_audit` (append-only) +
  методы `log_audit` / `get_audit` / `get_shift` / `get_day_off_request` в
  `ShiftsManager`. Миграция additive (`CREATE TABLE IF NOT EXISTS`), бэкап
  `shifts.db.backup_v3` снимается автоматически.
- `routes/schedule.py`: хелпер `_audit` берёт автора из `current_user` и пишет
  запись на создание/изменение/удаление смены, простановку/очистку факта часов,
  добавление/удаление выходного. Снимок «до» для delete/update берётся **перед**
  мутацией -> читаемое описание. Журналирование best-effort (try/except: сбой
  журнала не валит операцию). Новый `GET /api/schedule/audit/<y>/<m>`.
- UI: панель **«История изменений»** на `/schedule/edit` (сворачиваемая, ленивая,
  обновляется на каждый reload), формат «ДД.ММ ЧЧ:ММ — Автор — что сделал». Плюс
  компактная лента **«Последние изменения»** на `/schedule` (просмотр, для
  барменов) — цветная точка по типу действия, последние 8.
- `actor_name` — снимок имени автора (журнал читается после переименования/
  удаления аккаунта). `entity_date` — дата изменения, по ней фильтр на месяц.

**Связь с «проставлять себе часы»:** факт часов бармен вводит на `/schedule`
(просмотр), но пишет тот же серверный роут `…/fact` — поэтому действие попадает
в журнал с автором, и в истории редактора видно, кто проставил себе часы.

**Файлы:** `core/shifts_manager.py`, `routes/schedule.py`,

### 2026-06-23 — Датапикеры упирались в 20-е число (maxDate-строка парсилась форматом d.m.Y)

Владелец не мог выбрать на дашборде даты 21/22/23.06.2026 — максимум упирался в 20.06, и так во всех браузерах.

**Причина (настоящая):** Flatpickr-конфиги задавали `dateFormat: 'd.m.Y'` и `maxDate: '2026-12-31'` **строкой**. Flatpickr парсит строковый `maxDate` тем же `dateFormat`: из `'2026-12-31'` токен `d` съедал первые две цифры `20` как день, а месяц/год брались из текущей даты -> максимум = **20-е число текущего месяца**. Поэтому 21–23 были недоступны (и было бы так каждый месяц). До 2026-02-09 стоял `maxDate: 'today'` (спец-кейс Flatpickr, парсился корректно) — баг приехал вместе с заменой на строку «чтобы разрешить будущие даты».

**Исправление:** `maxDate` передаётся **Date-объектом** (`new Date(2027, 11, 31)` на дашборде; `new Date(2026|2027, 11, 31)` на остальных страницах) — Date берётся как есть, без парсинга форматом. Тот же баг был на 6 других страницах с `d.m.Y` + строковым `maxDate`, поправлено везде.

**Файлы:** `static/js/dashboard/modules/datepicker.js`, `templates/{draft,discounts,index,employee,waiters,bonus}.html`.

**Сопутствующее — service worker (`static/service-worker.js`):** до этого подозревали кэш, и воркер действительно отдавал весь JS по Cache-first (риск залипания старого билда). Переведён на **Network-first** для JS/CSS/HTML (Cache-first только для неизменяемой статики — шрифты/иконки/картинки), чужие origin (CDN) не перехватываются, `CACHE_VERSION = 'v2'` чистит старые кэши при активации. Теперь свежий билд приезжает без ручной чистки кэша.

### 2026-06-23 — Авторизация: вход, гейт, аккаунты

Раньше auth-слоя не было вообще — все страницы и API открыты любому. Добавлен

### 2026-06-17 — Open-check бот: ложное «ЗАКРЫТ — Варш» (дробные секунды iiko + Py 3.10)

Бот слал «ЗАКРЫТ — Варш» при открытой смене Варшавской. Локально (Python 3.14)
всё верно, на проде (контейнер, Python 3.10) — «закрыт».

**Root cause:** `openDate` Варшавской = `2026-06-17T13:42:51.45` (2 цифры дробной

поэтому только Варшавская страдала. Лог-след: `[WARN] _parse_iso_datetime failed
for '2026-06-17T13:42:51.45'`. Разбор — `docs/lessons.md`.

**Что:**
- **`core/iiko_api.py`** (фикс корня) — `_parse_iso_datetime` нормализует дробную
  часть секунд до 6 цифр перед `fromisoformat`; работает на всех версиях Python.
  Общий фикс: те же даты парсятся в employee-метриках.
- **`core/open_check_bot.py`** (защита заодно) — резерв `pointOfSaleId -> venue_key`

  выдачи `/corporation/groups`: смена с неизвестным `pos_id` логируется и берётся
  из резерва, а не теряется молча.

**Урок:** dev-Python (3.14) новее прод-Python (3.10) — `fromisoformat` ведёт себя

**Hardening (по адверсариальному ревью изменений):**
- `_read` больше не теряет подписчиков молча при битом JSON — логирует и сохраняет

### 2026-06-07 — «Общее» = сумма баров (автоматический агрегатный план)

План для «Все заведения» показывал 0, если отдельная запись `all_<год-месяц>` не была сохранена вручную (напр. за июнь 2026 планы баров есть, а `all_2026-06` — нет). Проверка данных: за 8 из 9 прошлых месяцев сохранённый `all` точно равен сумме баров (т.е. его и так вели руками), за апрель — устаревший дрейф. Решение (одобрено): «Общее» всегда считается из баров, сохранённые `all_*` больше не читаются.

**Что:**
- **`core/plans_manager.py`** `calculate_plan_for_period`: ключи `all`/`total`/`общая` нормализуются к `''` → суммирование по 4 барам (абсолютные метрики — сумма; относительные — средневзвешенные). Покрывает все вызовы (Аналитика, экспорт и т.д.).
- **`routes/dashboard.py`** `/api/revenue-metrics`: для «Общее» (all/пусто) план берётся как `calculate_plan_for_period('', <1-е>, <последнее число месяца>)`, а не из записи `all_*`. Конкретный бар — `get_monthly_plan`.

### 2026-06-07 — KPI: настраиваемое количество показателей + фонд (пул) вместо фикс-базы

### 2026-06-06 — Авто-расчёт метрик плана: чеки, доля кухни, прибыль, % наценки

В модалке создания/редактирования плана многие метрики вводились руками, хотя выводятся из других, — риск «магических чисел» и рассинхрона с фактом. Сведено к 8 ручным метрикам, 8 считаются автоматически (формулы метрик-двойников факта повторяют расчёт факта).

**Что:**

- Из ручного ввода в авто-расчёт переведены: **чеки** (`= выручка / средний чек`), **доля кухни** (`= 100 − розлив − фасовка`), **прибыль** (`= выручка − Σ себестоимость_кат`, где `себестоимость_кат = выручка_кат / (1 + наценка_кат/100)`), **списания баллов** (`= 5% от выручки`, константа `LOYALTY_WRITEOFF_RATE`). Поля стали `readonly` с бейджем «авто».
- **`% наценки` исправлена:** теперь взвешена по **себестоимости** (`100 × выручка / Σ себестоимость − 100`) — как факт `avg_markup` в [core/dashboard_analysis.py](../core/dashboard_analysis.py). Прежняя формула (средневзвешенная по долям выручки) давала систематическое расхождение с фактом (на синтетическом примере 200% против фактических 180%).
- Связка **выручка/чеки/средний чек**: руками вводятся выручка и средний чек, чеки считаются (по решению пользователя).
- Секция «Основные показатели» переупорядочена (выручка, средний чек, чеки-авто, прибыль-авто); добавлен слушатель ввода среднего чека; `updateSharesSum` теперь сам считает долю кухни и валидирует «розлив + фасовка ≤ 100%».
- Формулы повторяют расчёт факта, поэтому план и факт остаются сравнимыми; все производные поля показываются в подсказках (`title`) — принцип «детерминированность + формула в UI».

### 2026-05-28 — Audit Selectel: надёжность под gunicorn --workers 2, atomic writes, lock-файлы

После миграции на Selectel под Docker (gunicorn `--workers 2`) выявлены и
устранены race-conditions и потенциальные коррупции данных, незаметные на Render
(где раньше шёл один worker). Аудит провёл отдельный подагент, валидацию —
третий. Все правки прошли `py_compile`, lock-папки фактически маунтятся
persistent-volume'ами (`/srv/beer/data → /app/data`,
`/srv/beer/chz_debug → /app/chz_test/debug`).

**Что:**
- `core/chz_scheduler.py` — добавлен ежедневный atomic lock-файл
  (`data/.chz_refresh_lock_YYYY-MM-DD`, `O_CREAT|O_EXCL`). Без него оба worker'а
  тикали в 03:00 и оба POST'или `/api/chz/refresh` → двойной refresh ЧЗ-API.
  Паттерн идентичен `open_check_scheduler.py`.
- `routes/stocks.py:796-858` — `/api/chz/refresh` получил cross-worker lock
  `chz_test/debug/refresh.lock` (atomic `O_EXCL`, stale-detection >30 мин).
  Прежний `_refresh_proc` был in-process переменной → два параллельных POST
  стартовали два `remote_exec.py csv-auto` с записью в один `refresh.log`.
- `core/shifts_manager.py:36-48` — на каждом коннекте `PRAGMA journal_mode=WAL`,
  `busy_timeout=5000`, `synchronous=NORMAL`. Под 2 worker'ами без WAL писатели
  блокировали друг друга и читателей `shifts.db`. SQLite автоматически
  конвертирует существующую БД при первом WAL-pragma.
- `core/taps_manager.py:113-143` — atomic write через tmp + `flush` + `fsync`
  + `os.replace`. Раньше при крэше worker'а в `json.dump` файл
  `/kultura/taps_data.json` оставался полу-записанным (битый JSON, при рестарте
  пустой таплист).
- `core/olap_reports.py:170-173` — `timeout=300` → `100` (gunicorn `--timeout 120`
  убил бы worker раньше; теперь fallback укладывается до SIGKILL).
- `telegram_webhook.py:18`, `telegram_bot.py:18` — убран хардкод-fallback токена
  `8261982160:AAFu...`. Раньше токен утекал в git и публичные форки. Теперь
  при отсутствии env поднимается `RuntimeError`, `extensions.py:74-86` ловит
  через try/except и выставляет `TELEGRAM_BOT_ENABLED=False`.
- `app.py:46-51` — `app.run(debug=True)` → `debug=os.environ.get('FLASK_DEBUG')`.
  Под gunicorn эта ветка не исполняется, но защита от случайного `python app.py`
  на сервере.
- `requirements.txt` — добавлены upper-bounds (`<3` / `<4`) для `portalocker`,
  `openpyxl`, `python-dateutil`, `paramiko`, `Markdown` — защита от breaking
  major-релизов.

**Почему:**
- gunicorn `--workers 2` запускает 2 процесса, каждый импортирует `app.py` →
  стартует свои daemon-потоки → дубли action'ов. `chz_scheduler` дублей не
  замечал, `open_check_scheduler` уже имел свой lock.
- Сетевой/файловый I/O в фоновых тредах + SIGTERM/OOM → polу-записанный JSON
  на `/kultura` (persistent disk).
- iiko OLAP `timeout=300` > gunicorn `timeout=120` — worker рестартился до
  получения ответа, кэш не успевал прогреться.

**Файлы:**
- `core/chz_scheduler.py`, `routes/stocks.py`, `core/shifts_manager.py`,
  `core/taps_manager.py`, `core/olap_reports.py`, `telegram_webhook.py`,
  `telegram_bot.py`, `app.py`, `requirements.txt`.

**Подтверждено валидатором:**
- Все lock-пути маунтятся persistent-volume'ами (переживают рестарт контейнера).
- `os.replace` атомарен и на Windows (Python 3.3+), и на Linux.
- Существующая `shifts.db` корректно мигрирует в WAL при первом подключении.
- Конфликтов с правками URL-аудита нет (разные строки).

**Открытые P2 (не блокеры):**
- `extensions.py:35`, `core/kpi_calculator.py:75`, `core/meeting_notes.py:17`,
  `core/shifts_manager.py:28` — `os.path.exists('/kultura')` напрямую вместо
  `core/storage_paths`. В production работает (mount `/kultura`), но единый
  helper упростит dev-окружение.
- `routes/misc.py:run_async` — `asyncio.get_event_loop()` (deprecated в Py 3.12).
- `core/olap_reports.py` — 25+ `requests.*` без retry/backoff (iiko flaky).
- `print()` повсеместно вместо `logging.*` (под gunicorn `--access-logfile -`
  работает, но без уровней).
- Нет файлового lock на `/kultura/plansdashboard.json`, `meeting_notes.json`,
  `kpi_targets.json` (только in-process `threading.Lock`) — под 2 worker'ами
  потенциальная коррупция. Решается тем же tmp+replace.

### 2026-04-16 — Фикс сохранения планов в дашборде: диск = единственный источник правды

**Что сделано:**
- Убрана авто-перезапись Render Disk значениями из repo при старте сервера
- Удалён дубликат sync-логики (`_merge_json_plans` в `storage_paths.py` + `_sync_local_plans_to_storage` в `plans_manager.py` делали одно и то же)
- Удалён эндпоинт `POST /api/plans/import-from-excel` (импорт через UI больше не нужен)
- Удалён метод `PlansManager.replace_all_plans` (использовался только Excel-импортом)
- `data/plansdashboard.json` (repo-копия) теперь только **seed при первом старте** на чистом диске — если `/kultura/plansdashboard.json` уже есть, repo не влияет ни на что

**Почему:**
Старая логика на каждом рестарте Render безусловно перезаписывала значения на диске значениями из repo, если они отличались. Любые правки, сделанные через дашборд, откатывались после редеплоя. Теперь единственный способ изменить план — вручную через UI, правки живут на диске.

**Файлы:**
- `core/storage_paths.py` — упрощён до одной функции `get_data_path`
- `core/plans_manager.py` — удалён `_sync_local_plans_to_storage`, `_read_json_file_safe`, `replace_all_plans`; убран import `get_local_data_path`
- `routes/dashboard.py` — удалён эндпоинт `/api/plans/import-from-excel`

**Что ломается если изменить неправильно:**
- Если кто-то вернёт sync repo→disk — правки снова будут теряться после рестарта
- CLI-скрипты `scripts/import_export/import_plans_from_excel*.py` оставлены и работают автономно (пишут в `data/plansdashboard.json` локально); использовать только в dev-окружении как одноразовую миграцию
