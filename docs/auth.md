# Авторизация

## Что это

Полноценный вход в приложение. Раньше auth-слоя не было вообще — все страницы и
все API были открыты любому, кто дотянулся до приложения. Теперь:

- **Личный аккаунт на каждого** (бармен / владелец), вход по логину и паролю.
- **«Вход один раз»**: долгая cookie-сессия (180 дней, скользящее продление).
  Плашка входа появляется только когда cookie нет или она протухла.
- После входа — **равные права** на все бизнес-страницы. Флагом `is_admin`
  ограничена только страница управления аккаунтами (`/admin/users`).
- `display_name` совпадает с именем в реестре графика смен (`schedule_employees`)
  — это готовая основа для журнала «кто менял смену» (текущий пользователь
  подставляется автоматически).

Решения владельца (зафиксированы): личные аккаунты, у всех пароль, аккаунты
заводит владелец сам; единственное ограничение прав — управление аккаунтами
только для админа (чтобы бармен случайно не залочил вход).

## Файлы

| Файл | Роль |
|------|------|
| `config.py` | `get_secret_key()` — стабильный ключ подписи сессий (env -> файл -> генерация) |
| `core/auth_manager.py` | SQLite-хранилище аккаунтов (`auth.db`), хэш паролей, CRUD, bootstrap |
| `core/auth_guard.py` | `init_auth(app)`: сессия/cookie, ProxyFix, глобальный гейт, `current_user` |
| `routes/auth.py` | `/login`, `/setup`, `/logout`, `/admin/users` + API `/api/auth/*` |
| `templates/login.html` | страница входа и первичной настройки (один шаблон, два режима) |
| `templates/admin_users.html` | управление аккаунтами (только админ) |
| `templates/shared/nav.html` | имя пользователя + «Выйти» в футере, ссылка «Аккаунты» для админа |
| `app.py` | вызов `init_auth(app)` после `register_blueprints` |
| `tests/test_auth.py` | гейт, логин/logout, admin-only, first-run, защита последнего админа, стабильность ключа |

## Как работает

### SECRET_KEY — «вход один раз»

Сессия живёт в подписанном cookie; стабильность ключа подписи критична — при его
смене все cookie становятся недействительными и всех разлогинивает.

`config.get_secret_key()` (приоритет):
1. env `SECRET_KEY` — для явного управления в проде;
2. persistent-файл `{/kultura|data}/secret_key` — генерируется **один раз** и
   переживает рестарты/редеплои (том `/kultura`). Именно он обеспечивает, что
   плашка входа не появляется на каждый заход;
3. если файла нет — сгенерировать и записать. Создание **эксклюзивное**
   (`O_EXCL`): гонку первого старта (2 gunicorn-воркера) выигрывает один,
   проигравший перечитывает уже записанный ключ.

Оба воркера читают один и тот же файл -> одинаковый ключ -> cookie валиден между
воркерами и переживает редеплой.

### Хранилище аккаунтов

`core/auth_manager.py`, SQLite `{/kultura|data}/auth.db` — отдельная база от
`shifts.db`. Пароли хранятся только как хэш (`werkzeug.security`,
`generate_password_hash`/`check_password_hash`). Схема additive (как в
`ShiftsManager`): миграции только `ALTER TABLE ADD COLUMN`, DROP запрещён.
Таблица (v2) `users(id, login UNIQUE COLLATE NOCASE, display_name, short_label,
password_hash, is_admin, active, created_at, last_login_at)`. WAL + `busy_timeout`
под 2 воркера. Синглтон `get_auth_manager()`.

Поля аккаунта: **логин** (2-32 символа `[A-Za-z0-9_.-]`, регистронезависим),
**имя** (фамилия и имя, `display_name`), **сокращение** (`short_label`, типа «АН»,
до 12 символов — вводится **вручную**, без авто-генерации из имени: первые буквы
часто совпадают). Пароль не короче `MIN_PASSWORD_LEN` (4 — решение владельца).
Защита последнего админа: нельзя выключить / разжаловать / удалить единственного
активного админа.

**Атомарность инвариантов (cross-process).** `threading.Lock` сериализует только
внутри процесса, а в проде gunicorn 2 воркера = 2 процесса. Поэтому проверки
«последний админ» (`set_active`/`set_admin`/`delete_user`) и «первый владелец»
(`create_first_owner`) делают check+write в одной транзакции под `BEGIN IMMEDIATE`
(`_write_txn`): write-лок берётся сразу, второй воркер ждёт на `busy_timeout` —
два одновременных запроса не могут обойти инвариант (ни обнулить админов, ни
создать двух владельцев на first-run).

### Гейт (`init_auth(app)`)

- `SECRET_KEY` из `config`; cookie: `HttpOnly`, `SameSite=Lax`, `Secure` на проде
  (признак — наличие `/kultura`; переопределяется env `SESSION_COOKIE_SECURE`);
  `PERMANENT_SESSION_LIFETIME=180д`, `SESSION_REFRESH_EACH_REQUEST=True`
  (скользящее продление для активных).
- `ProxyFix(x_for=1, x_proto=1, x_host=1)` — Caddy терминирует TLS и проксирует по
  http на localhost; доверяем `X-Forwarded-*` ровно от одного прокси.
- Глобальный `before_request` (`_auth_gate`): **всё закрыто**, кроме
  `PUBLIC_ENDPOINTS`. Неавторизованный: на `/api/...` -> `401 {auth_required:true}`,
  на странице -> `302` на `/login?next=...`.
- `context_processor` отдаёт `current_user` в шаблоны.

Публичные эндпоинты: `auth.login`, `auth.logout`, `auth.setup`, `static`,
`serve_manifest` (PWA-манифест), `misc.telegram_webhook_handler` (Telegram POST'ит
без cookie). Всё остальное — за гейтом, включая PWA-виджет (`/dashboard/widget`),
который работает после входа (PWA делит cookie-jar браузера).

### Потоки входа

- **First-run**: пока в системе 0 аккаунтов, `/login` уводит на `/setup` —
  создание первого владельца (admin). После — `/setup` недоступен.
- **/login** (логин+пароль), **/logout**.
- `next` после входа проходит через `_safe_next` (защита от open-redirect:
  только локальные пути `/...`, не `//`, не схема).

### Управление аккаунтами (`/admin/users`, только админ)

| Метод | Путь | Что делает |
|-------|------|-----------|
| GET | `/admin/users` | страница: список + форма создания |
| GET | `/api/auth/users` | список аккаунтов (без хэшей) |
| POST | `/api/auth/users` | создать (login, display_name, short_label, password, is_admin) |
| POST | `/api/auth/users/<id>/profile` | изменить имя и/или сокращение |
| POST | `/api/auth/users/<id>/password` | сбросить пароль |
| POST | `/api/auth/users/<id>/active` | вкл/выкл аккаунт |
| POST | `/api/auth/users/<id>/admin` | выдать/снять админа |
| DELETE | `/api/auth/users/<id>` | удалить аккаунт |

`display_name` подсказывается из реестра графика (datalist) — чтобы совпадал с
именем в сменах.

### Безопасность (осознанная позиция)

- **CSRF**: отдельных токенов нет; защита — `SameSite=Lax`, при которой браузер не
  шлёт cookie на кросс-сайтовые POST/PUT/DELETE. Для внутреннего инструмента
  достаточно. `/logout` — GET, но единственный его эффект (выход) для CSRF
  безвреден.
- **Пароли** — только хэш; `verify_credentials` не раскрывает, что именно неверно.
- **Неактивный/удалённый юзер с живой сессией**: при следующем запросе гейт видит
  `active=0`/отсутствие и рвёт сессию.
- **Сброс пароля НЕ убивает старые сессии** (сессия живёт до 180 дней). Чтобы
  немедленно выкинуть человека (увольнение, утечка), **выключи его аккаунт**
  (`/admin/users` -> «Выключить») — это рвёт его сессию на следующем запросе; при
  необходимости потом включи обратно с новым паролем. Сброса пароля одного для
  отзыва доступа недостаточно.
- **Нет ограничения попыток входа и мин. длина пароля 4** — осознанный принятый
  риск (внутренний инструмент за Caddy/HTTPS, доверенная команда, логины не
  публичны). При желании усилить — поднять `MIN_PASSWORD_LEN` и добавить
  rate-limit на `/login`.
- Приложение прикрыто на сетевом уровне (Selectel VPS + Caddy); auth — второй слой.

## Деплой

- Том `/kultura` хранит `auth.db` и `secret_key` — переживают редеплой (код
  запекается в образ, данные — на томе).
- **Первый вход**: открыть `/login` -> форма создания владельца. Либо
  автоматизировать через env `AUTH_BOOTSTRAP_LOGIN` / `AUTH_BOOTSTRAP_PASSWORD`
  (+ опционально `AUTH_BOOTSTRAP_NAME`) — создаст admin на первом старте.
- `SECRET_KEY`: можно задать в `.env` для явного контроля; иначе генерируется в
  `/kultura/secret_key` один раз. **Не ротировать** без причины — разлогинит всех.

  бы всех на каждом рестарте.
