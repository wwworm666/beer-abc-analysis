# ИИ-агент «вопрос на человеческом — детерминированный ответ из iiko»

> Статус: концепт / feasibility-исследование (2026-06-12). Код не написан.
> Источники: локальная документация `docs/iiko-api/` (155 статей), кодовая база проекта,
> внешние материалы по NL-аналитике (Cortex Analyst, Databricks Genie, dbt Semantic Layer и др.).

## Что это

Пользователь пишет вопрос на человеческом языке («какое пиво продавалось лучше в марте?»),
LLM переводит его в структурированный запрос к iiko Server API, детерминированный исполнитель
выполняет запрос и считает производные метрики по документированным формулам, ответ показывает
реальные цифры из API — LLM никогда не генерирует числа.

Мотивация: преднастроенных отчётов в сервисе уже 13+ страниц — пользователи теряются;
сырой OLAP в iikoOffice слишком сложен для не-аналитика.

## Главный вывод

Идея реализуема: из 15 типовых вопросов менеджера 9 закрываются API полностью, 5 частично,
и только 1 («прогноз») — нет. Ключевой инсайт индустрии (2024-2026): свободная генерация
запросов LLM-ом проваливается (16-21% точности на корпоративных бенчмарках), а схема
«LLM выбирает метрику/шаблон из семантического слоя, движок компилирует запрос детерминированно»
даёт 90-100%. У нас семантический слой фактически уже есть: 24 выверенных OLAP-тела
(`docs/technical/OLAP_REPORTS_COLLECTION.md`), правила построения
(`docs/technical/OLAP_REPORT_BUILDING_RULES.md`) и машиночитаемый каталог полей
(`GET /resto/api/v2/reports/olap/columns?reportType=...`).

**Решение по архитектуре: не free-form генератор OLAP-тел, а компилятор
«вопрос -> QuerySpec из библиотеки проверенных шаблонов + параметры».**

## Матрица: 15 типовых вопросов менеджера

| # | Вопрос | Вердикт | Как / что мешает |
|---|--------|---------|------------------|
| 1 | Какое пиво продавалось лучше в марте (выручка/литры) | да | Выручка — SALES: DishName x DishDiscountSumInt, фильтр DishGroup.TopParent in (Напитки Розлив/Фасовка). Литры — складской контур, в единицах товара (кега = литры): OLAP TRANSACTIONS Product.Name x Amount.Out по расходу от продаж, или /reports/storeOperations с documentTypes=SALES_DOCUMENT (наш /stocks уже так считает outgoing). Нюанс: это нормативные литры по техкартам; связь блюдо-кега — dish_to_keg_mapping (есть). Парсинг объёма из названий блюд остаётся только фоллбэком |
| 2 | Выручка по барам неделя к неделе | да | SALES: Department (или Store.Name) + OpenDate.Typed, CUSTOM-период 14 дней одним запросом; дельту считает исполнитель |
| 3 | Средний чек / чеки по дням и часам | да | SALES: DayOfWeekOpen + HourOpen, UniqOrderId + DishDiscountSumInt (+ .average для сверки); формула чека = сумма/чеки, показывать в UI |
| 4 | Маржинальность топ-20 позиций | да | SALES: DishName, меры ProductCostBase.ProductCost/.Profit/.Percent рядом с DishDiscountSumInt; топ-20 режется локально (сортировки в API нет); проверить заполненность ProductCostBase на живых данных |
| 5 | Списания за месяц по причинам | да | OLAP TRANSACTIONS: TransactionType=WRITEOFF, группировка Account.Name (= причина); сверка списком документов /v2/documents/writeoff; поле cost в документах read-only и обычно 0 — деньги только из OLAP |
| 6 | Остатки пива на складе / в кегах | да | GET /v2/reports/balance/stores?timestamp=... (быстрый срез); НЕ использовать OLAP StartBalance/FinalBalance (официально «тяжёлый запрос»); «пиво» — классификация по дереву групп локально |
| 7 | ABC-анализ за квартал | да | Один SALES-запрос за квартал (длинные периоды работают; дробить по месяцам — только если ответ окажется медленным); классы A/B/C — локально; модуль ABC уже есть (routes/analysis.py) |
| 8 | Продажи по сотрудникам/сменам | да | SALES: WaiterName (+ AuthUser для продаж через стойку — см. lessons), SessionNum; смены — /v2/cashshifts/list. Связь только по имени, не UUID |
| 9 | Отмены/удаления/сторно и кем | да | SALES с ИНВЕРТИРОВАННЫМ фильтром DeletedWithWriteoff=DELETED_*; RemovalType, AuthUser, DeletionComment; возвраты — Storned=TRUE + DishReturnSum. Точный журнал событий /resto/api/events требует лицензию модуля 2200 (API_EVENTS) |
| 10 | Влияние скидок | да | SALES: OrderDiscount.Type, DiscountSum vs DishSumInt vs DishDiscountSumInt; каузальность («привела ли акция к росту») — честно за рамками |
| 11 | Выручка по часам vs часы персонала | частично | SALES по HourOpen + /resto/api/employees/attendance (XML; явки пересекают интервал, to включающий — резать края); join и норматив «руб./бармен-час» — наши |
| 12 | Гости и время визита | частично | GuestNum, GuestNum.Avg, OrderTime.AverageOrderTime — но качество зависит от кассовой дисциплины; «время визита» = время чека, оговаривать |
| 13 | Почему упала выручка | частично | Каузальности в API нет; детерминированный drill-down (трафик vs чек, часы, категории, смена, оплаты) серией из 5-7 запросов; вне MVP |
| 14 | Прогноз выручки | нет | Forecast-endpoint не существует; отдавать план (/reports/monthlyIncomePlan или data/ проекта) + историю с пометкой «прогноза iiko не даёт» |
| 15 | Что покупают вместе | частично | Поле SoldWithDish (семантика в доке не расписана — проверить на живом сервере); полная корзина — офлайн-джоба, не чат |

MVP — вопросы 1, 2, 3, 4, 8: все на одном SALES-шаблоне и уже работающей инфраструктуре.

## Чего нельзя получить по API (консолидировано)

> Правка 2026-06-13 (фидбек владельца): первая версия раздела выдавала рекомендации iiko
> за жёсткие лимиты. Разделено. Практика подтверждает: OLAP-запросы за 3+ года выполняются,
> наш прод параллелит запросы (ThreadPoolExecutor в `core/olap_reports.py:1388`) без проблем.

Жёсткие лимиты схемы запроса (в теле запроса этих возможностей нет — постобработка всегда на исполнителе):

- **Нет сортировки, top-N, пагинации, HAVING, вычисляемых полей** — в теле v2 только
  reportType/buildSummary/groupBy*/aggregateFields/filters; «топ-10» и «выручка > X» режем локально.
- **Фильтр по дате обязателен с iiko 5.5** — не проблема, а правило исполнителя: для «за всё время»
  подставляется диапазон от даты открытия баров до сегодня.
- **/columns не отдаёт допустимые ENUM-значения** — брать из kody-bazovykh-tipov.md / живых данных.
- **iikoCloud API аналитики не даёт вообще** (в changelog 2022-2025 ни одного OLAP-эндпоинта) — только Server API.

Рекомендации iiko по нагрузке (ogranicheniya-i-rekomendatsii.md — НЕ лимиты; API выполняет и больше,
вопрос только во времени ответа и нагрузке на боевой RMS):

- «Период не длиннее месяца, в идеале день/неделя» — по факту запросы за 3+ года работают;
  дробление держать как опцию для тяжёлых отчётов, не как правило.
- «Запросы последовательно» — наш прод 2 параллельных воркера давно живёт нормально;
  агенту разумно ограничить параллелизм небольшим пулом, а не единицей.
- «build-summary=false, не более 7 полей» — для крупных сетей; для 4 баров запас большой,
  но buildSummary=false всё равно дефолт (итоги считаем сами).

Операционное (из доков; проверить поведение на нашем сервере):

- **OLAP API входит в платную лицензию iikoAPIServer**; авторизация занимает слот лицензии (avtorizatsiya.md),
  при занятых слотах — 403. У нас уже есть рабочий токен (OlapReports) — переиспользовать его пул,
  не плодить auth. Свободные слоты: GET /resto/api/licence/info?moduleId=28008806.
- **Токен по доке живёт ~1 час** и умирает при рестарте сервера — ловить 401 и перелогиниваться
  (retry в core/iiko_api.py уже есть); фактический TTL на нашем сервере проверить.
- **Числовых rate-лимитов в документации нет** («узнавать в техподдержке»).

Дыры в данных:

- Литров как меры нет только в OLAP SALES (DishAmountInt — в единицах блюда, т.е. порции).
  Складской контур литры ДАЁТ, в единицах товара: остатки — balance/stores (наш /stocks уже
  пишет remaining_liters из amount, routes/stocks.py:315); расход за период — OLAP TRANSACTIONS
  (Amount.Out по товару) или /reports/storeOperations с documentTypes=SALES_DOCUMENT
  (наш /stocks так считает outgoing); метаданные объёма — unitCapacity/containers в ProductDto.
  Нюанс: расход по продажам = нормативные литры по техкартам; фактический пролив виден
  только инвентаризацией. Парсинг объёмов из названий блюд — фоллбэк, не основной путь.
- Серверного поиска по имени нет нигде: товары, бары, сотрудники резолвятся только по локальной выгрузке справочников (products/list, corporation/departments, employees — все с revisionFrom для инкрементального кэша).
- /columns не отдаёт допустимые ENUM-значения — брать из kody-bazovykh-tipov.md.
- Проведённые инвентаризации, акты реализации — только загрузка (import), выгрузки нет.
- Техкарты без цен; связка «продажи <-> сотрудник» только по строковому имени; payroll-начислений нет (paymentDetails пуст для окладников).
- Событийный журнал /resto/api/events — отдельная лицензия (модуль 2200) и право B_VTJ.
- P&L, кассовая книга, поштучные чеки как готовые отчёты — нет; только сборка через OLAP.

Качество локальной копии доков: ClickHelp-блоки (XSD, примеры ответов) местами заменены
плейсхолдерами `%%CH%PRE..%%` — точные структуры ответов снимать с живого сервера или портала.
rabota-s-zakupkami.md — про другой продукт (iikoWeb PublicApi закупок, отдельная лицензия).

## Архитектура (предлагаемая)

```
Вопрос пользователя
  -> LLM-классификатор: какой шаблон из библиотеки? (structured output, JSON Schema)
     + параметры: период, бар, категория, метрика, top-N
     + при неоднозначности — уточняющий вопрос («лучше — по выручке или по литрам?»)
  -> Валидатор: поля по живому /columns, значения по локальным справочникам,
     обязательные предохранители (DateRange; NOT_DELETED x2 — кроме вопросов про удаления)
  -> Детерминированный исполнитель: OLAP/balance/cashshifts/attendance,
     кэш по ключу QuerySpec (10 мин TTL + single-flight — уже есть в extensions.py),
     сортировка/top-N/производные метрики по документированным формулам
  -> Ответ: цифры + ОБЯЗАТЕЛЬНО интерпретация («я понял вопрос как...»),
     применённые фильтры и «как проверить в iikoOffice»
```

Принципы (из уроков индустрии и собственных docs/lessons.md):

1. **Числа никогда не из LLM.** LLM выбирает шаблон и параметры; считает исполнитель.
2. **Канонический словарь метрик** фиксируется заранее: «выручка» = DishDiscountSumInt (со скидкой),
   «чеки» = UniqOrderId.OrdersCount, «март» = учётные дни OpenDate.Typed с +1 днём к exclusive-границе to.
3. **Шаблоны, не free-form.** Стартовая библиотека — 16 production-тел из OLAP_REPORTS_COLLECTION.md.
   Free-form генерация — только как explicit-режим «эксперимент» с пометкой.
4. **Показ интерпретации обязателен** — первая невоспроизводимая в iikoOffice цифра = конец доверия.
5. **Латентность**: реальный OLAP-roundtrip на широком периоде 16-17 с (lessons.md) — чат должен
   стримить статус («строю отчёт по марту...»), популярные вопросы отдавать из кэша/витрины
   (ETL уже начат: knowledge_graph/etl/sales_loader.py).
6. **Refusal-дизайн**: «почему», прогнозы, кросс-системные вопросы (ЧЗ) — честный отказ
   с тем, что есть (факторное разложение, план, история).

## Проверить на живом сервере до старта кода

1. Заполненность ProductCostBase.* (себестоимость ведётся корректно?).
2. Семантика SoldWithDish.
3. Есть ли лицензия модуля 2200 (API_EVENTS) и сколько слотов iikoAPIServer
   (GET /resto/api/licence/info?moduleId=28008806).
4. Поведение Account.Name vs Contr-Account.Name в TRANSACTIONS/WRITEOFF на наших данных.
5. Живой /columns vs локальная дока (deprecated-поля; «Месяц» = Mounth — опечатки API реальны).
6. Точное значение TransactionType для расхода по продажам (литры через TRANSACTIONS Amount.Out)
   и структура ответа /reports/productExpense (XSD в локальной копии утерян).

## Основные риски

| Риск | Митигация |
|------|-----------|
| Недетерминированность плана: один вопрос -> разные запросы в разные дни | Библиотека шаблонов + канонический словарь метрик + golden-набор вопросов (100+ с эталонным QuerySpec) как регрессионный тест |
| Потеря доверия после первого расхождения с iikoOffice | Показ интерпретации/фильтров/«как проверить»; сверка канона с iikoOffice до запуска |
| Положить боевой RMS тяжёлым запросом | buildSummary=false, умеренное число полей, небольшой пул параллелизма, кэш, витрина для популярных вопросов; дробление длинных периодов — опция при медленных ответах |
| Слоты лицензии / токен 1 час | Единый пул токенов с существующим синком; ловить 401/403, перелогин |
| Ретроспективные изменения данных (переоткрытые смены, накладные задним числом) | Метка «данные на момент HH:MM»; «замораживать» только закрытые периоды |
| Чат-агент проиграет каталогу с поиском | Гибрид: агент = маршрутизатор к существующим 13 страницам, где они есть («это есть на /discounts») + произвольные срезы там, где страницы нет |
