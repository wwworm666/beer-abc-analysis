Интеграция, логика которой построена на обработке веб-хуков, отправляемых из Cloud Api, должна реализовать сервис, который будет получать и обрабатывать запросы (веб-хуки) и указать url этого сервиса в настройках Cloud Api в личном кабинете iikoWeb.

Сервис, обрабатывающий веб-хуки, должен выполнять следующие требования:

1. обрабатывать входящие запросы достаточно быстро, не дольше 3 секунд,
2. возвращать статус http 200, если запрос был обработан успешно,
3. контролировать появление ошибок при обработке и устранять их причины.

Если обработчик возвращает в Cloud Api ошибочный статус (любой, кроме 200), это свидетельствует об ошибке, произошедшей при обработке веб-хука.
Так же ошибки происходят в случаях, когда данные из Cloud Api приходят в формате, который сервис обработки веб-хуков не ожидает получить, т.е. сервис не был разработан в соответствии с документацией, описывающий формат данных [веб-хуков](https://api-ru.iiko.services/docs#tag/Webhooks)

В случаях, когда отправляемые запросы систематически выполняются долго, с ошибками или не выполняются вообще (падают по таймауту), этот обработчик будет блокироваться, т.е. Cloud Api прекратит отправлять на него запросы:

1. сначала происходит блокировка на 15 минут, т.е. в течение 15 минут веб-хуки на проблемный url не отправляются. После чего отправка веб-хуков возобновляется. Это позволяет избежать полной блокировки обработчика из-за кратковременного сбоя (например, перезапуска сервиса обработки веб-хуков).
2. если после первой блокировки и последующей разблокировки поведение обработчика не меняется, то он блокируется повторно на час, после чего разблокируется.
3. при продолжающихся проблемах происходит следующая блокировка на 12 часов и далее происходит окончательная блокировка.
4. важно понимать, что обработчик будет блокироваться, даже если неуспешными являются не все отправляемые в него запросы, а только существенная их часть (10% и более).
