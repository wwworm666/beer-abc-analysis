Front Web API является плагином фронта (Resto.Front.Api.WebServer), работающим через API и открывающим возможность работать с фронтовыми данными через веб. В SDK в папке SampleHtml публикуется пример работы с web Api.

## Установка 

* Взять Resto.Front.Api.WebServer
* При необходимости поменять дефолтные настройки в файле Resto.Front.Api.WebServer.dll.config.
* | BaseAddress | Адрес развертывания сервера |
    | --- | --- |
    | Port | Порт развертываемого сервера |
    | DefaultExp | Время автоматического логаута по бездействию |
    | HttpsMode | Кроме включения данного флага нужно будет настроить SSL-сертификат |
* Для организации сетевых обращений стоит убедиться, что соответствующий порт не блокируется. Подробнее рассмотрено в п. 4.
* После запуска фронта в логе plugin-Resto.Front.Api.WebServer.log будет записано Web server started. http://127.0.0.1:9042/ с актуальным адресом.

## Авторизация и аутентификация 

* Любой запрос должен сопровождаться токеном в качестве параметра: http://127.0.0.1:9042/api/orders?key=12345678 или в качестве cookie.
* В случае отсутствия токена или его невалидности, будет возвращена соответствующая ошибка.
* Для получения токена (key) работы с web-api нужно вызвать метод http://127.0.0.1:9042/api/login/[moduleId].
* ModuleId должен удовлетворять ряду требований (они описаны ниже).
* В результате выполнения этого метода захватывается один слот лицензии данного moduleId. А вызывающей стороне будет передан токен. Скорее всего это будет некий guid.
В случае неуспешного логина (прислан невалидный moduleId, у пользователя не осталось столов запрашиваемого moduleId и т.д.) будет возвращена соответствующая ошибка.
* Слот лицензии можно освободить явно http://127.0.0.1:9042/api/logout/[key].
* Слот лицензии освобождается при перезапуске фронта, который является хостом, при этом соответствующий токен становится невалидным, нужно заново выполнять логин.
* Слот лицензии освобождается по таймеру, при этом соответствующий токен становится невалидным, нужно заново выполнять логин. Таймер отсчитывается от последнего обращения. *(Пример: Таймаут 30 минут. Получили key в 12:15. Обратились к сервису с этим key в 12:40. Key станет невалидным в 13:10)*

Правила для moduleId.

* Есть общий ключ — 2050. Т.е. если у клиента приобретена лицензия ApiFrontWeb, все желающие смогут брать себе слот этой лицензии запросом http://127.0.0.1:9042/api/login/2050.
* Возможна выдача частных ключей. Ограничение на частный ключ: он должен быть как минимум пятизначным, он должен заканчиваться на 50. Например, под плагин iikoArrivals можно зарегистрировать moduleId 13550. В этом случае запрос http://127.0.0.1:9042/api/login/13550 вернет токен доступа, если у клиента есть свободные слоты соответствующей лицензии.

Соответственно, запрос типа http://127.0.0.1:9042/api/login/100, http://127.0.0.1:9042/api/login/2021100 вернет неуспешный результат, даже если у клиента есть свободные слоты с этой лицензией. Ведь эти ключи не предназначены для front web api. Мы отфильтруем такие запросы на уровне нашего веб-сервиса.

## Доступные справочники 

* Заказы
    * http://127.0.0.1:9042/api/orders
    * http://127.0.0.1:9042/api/orders/[ID-заказа]
* Отделения
    * http://127.0.0.1:9042/api/sections
    * http://127.0.0.1:9042/api/sections/[имя-отделения]
* Столы
    * http://127.0.0.1:9042/api/tables
    * http://127.0.0.1:9042/api/tables/[номер-стола]
* Все продукты
    * http://127.0.0.1:9042/api/products
    * http://127.0.0.1:9042/api/products/[артикул]
* Иерархическое меню
    * http://127.0.0.1:9042/api/productgroups
    * http://127.0.0.1:9042/api/productgroups/[guid-конкретной-группы-продуктов]
* Быстрое меню
    * http://127.0.0.1:9042/api/quickmenu - быстрое меню для стола по умолчанию
    * http://127.0.0.1:9042/api/quickmenu/[guid-конкретного-отделения] - быстрое меню для конкретного отделения
* Пользователи
    * http://127.0.0.1:9042/api/users
    * http://127.0.0.1:9042/api/users/[имя]
* Кухонные заказы
    * http://127.0.0.1:9042/api/kitchenorders
    * http://127.0.0.1:9042/api/kitchenorders/[номер-заказа]
* Доставки
    * http://127.0.0.1:9042/api/deliveries
    * http://127.0.0.1:9042/api/deliveries/[номер-доставки]
* Гости

    * http://127.0.0.1:9042/api/clients - начиная с версии 4.3.5 не поддерживается.
    * http://127.0.0.1:9042/api/clients/[id-гостя]

## Принципы работы 

В результате обращения по одному из указанных выше адресов пользователь получает сериализованное представление запрашиваемых объектов. Формат зависит от запрашиваемого формата. Например, chrome по умолчанию представляет данные в xml.

Можно запрашивать Accept: application/json; charset=utf-8 и т.д.

Можно применять различные фильтры IQueryable (OData Query).

Например:

http://127.0.0.1:9042/api/tables?$top=10&$skip=15 (пропустить 15 и взять 10)
http://127.0.0.1:9042/api/orders?$filter=Number%20lt%20550 (т.е. Number less then 550 т.е. Number &lt; 550)

Для справочников доступны только get-запросы.

Для операций login/logout доступны get, put и post запросы.

Для корректной работы плагина необходимо запускать фронт от имени администратора или выключить UAC на машине, где он запускается, либо дать пользователю, от имени которого запускается фронт, права, как описано [в статье на Stackoverflow](http://stackoverflow.com/questions/5774606/host-wcf-application-without-uac-admin-privilege).

Правильная команда для запуска не из под администратора (звездочка, не плюсик!):

netsh http add urlacl url=http://\*:9042/ user=&lt;user&gt;

На порт 9042 это должно быть единственным разрешением. Надо проверить имеющиеся командой

netsh http show urlacl | findstr ":9042"

и удалить лишнее командой

netsh http delete urlacl url=&lt;url&gt;
