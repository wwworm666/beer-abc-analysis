Облачные системы с открытым API всегда имеют ограничения по использованию, обусловленные необходимостью сделать сервис максимально надежным и стабильным для клиентов.Чтобы гарантировать работу высоконагруженного сервиса, необходимо анализировать трафик и ограничивать случаи ненадлежащей работы с сервисом.В Cloud API реализовано 2 вида ограничений по работе с API:1. Полная блокировка API-логина;
2. Ограничение API-логина по количеству запросов.

### Блокировка API-логина

Такое ограничение применяется для API-логина только в случае, если интеграция может существенно нарушить работу любой из частей нашей системы.
Блокировку можно применить в следующих случаях:* массовая отправка одних и тех же запросов;

    * как правило, это происходит из-за того, что интеграция не проверяет ответы от нашего сервиса;
* большой % ошибочных запросов;
    * ошибки в интеграции не исправляются. Это особенно характерно при отправке запросов на создание заказа. В этом случае проще отключить такую интеграцию, чем потом разбираться с такими заказами на кассе);
* слишком большое количество заказов;
    * если ваша интеграция создает более 1000 заказов в день на один ресторан, то скорее всего вы используете нашу систему неправильно;
* пиковая нагрузка по любым группам методов в API;
    * как правило, это связано с ошибками в логике интеграции;
* тестовая нагрузка и/или нагрузочное тестирование, выполняемое интеграцией;
    * нагрузочное тестирование на продуктовом контуре для любых интеграций запрещено, поэтому, когда наша система выявляет такие интеграции, она может заблокировать их навсегда;
* другие случаи
    * невозможно описать все варианты, ведущие к блокировке. Однако, мы будем регулярно добавлять новые кейсы при недобросовестном использовании API

### Ограничение API-логина по количеству запросов или «Too many requests» (ответ с кодом 429) 
Такое ограничение применяется для API-логина только в случае, если интеграция начинает делать неоправданно большое количество запросов к одной и той же группе запросов.Принцип работы:* все методы в API разбиты на группы запросов на основании возвращаемых данных;
* каждый метод обязательно входят в какую-то одну группу запросов;
* максимально допустимое количество запросов в один временной слот для каждой группы запросов определяется экспертно на основании анализа исторических данных. При этом количество таких запросов:
    * не зависит от числа ресторанов, с которыми работает интеграция — для методов по работе со справочниками сети;
    * зависит от числа ресторанов, с которыми работает интеграция — для методов по работе с заказами, номенклатурой и другими данными, зависящими от ресторана.
* временной слот различен для разных групп методов (обычно от 1 до 10 мин)
    * в большинстве случаев он равен 1 мин. 
(исключение - справочник номенклатуры: временной слот - 1 час для неревизионного запроса. Более подробно см. [Номенклатура](/articles/api-documentations/nomenklatura/a/h2__842385953))
* если интеграция уже выполнила максимально допустимое количество запросов к конкретной группе запросов в текущий временной слот, то до его окончания все запросы к методам группы будут сразу возвращать ответ с кодом 429.
Примеры групп запросов:* справочники (включают все справочники, получаемые на сеть);
* информация по заказам (включает все множество методов, возвращающих заказы);
* и другие.
Примеры, когда может быть применена блокировка:* интеграция не кеширует на своей стороне справочники ресторана и при каждом новом обращении загружает справочники снова, что создает неоправданную нагрузку на API;
* интеграция работает под высокой нагрузкой (создает много заказов), но не использует групповые методы, возвращающие информацию сразу по множеству заказов. Вместо этого она в цикле перебирает список всех заказов и по каждому запрашивает статус;
* Интеграция запрашивает номенклатуру в один момент времени по всем РМС входящим в чейн, что создает неоправданную нагрузку на API. Верным решением будет запрашивать номенклатуру по каждому РМС последовательно;
* другие случаи.

Невозможно описать все варианты, ведущие к ограничению. Однако, мы будем регулярно добавлять новые кейсы при недобросовестном использовании API.

### Ограничение от сервера лояльности по количеству запросов или «Too many requests»
Отдельный вид ограничения от сервера лояльности применяется для сети или одиночного РМС только в случае, если интеграция начинает делать неоправданно большое количество запросов к одной и той же группе запросов лояльности. Ограничения накладываются не Cloud API, а кардом.
Примеры групп запросов:

* запросы по работе с гостями: [create_or_update](https://api-ru.iiko.services/docs#tag/Customers/paths/~1api~11~1loyalty~1iiko~1customer~1create_or_update/post) и [info](https://api-ru.iiko.services/docs#tag/Customers/paths/~1api~11~1loyalty~1iiko~1customer~1info/post) и [запрос на расчет лояльности](https://api-ru.iiko.services/docs#tag/Discounts-and-promotions/paths/~1api~11~1loyalty~1iiko~1calculate/post) — лимит на 100 одновременных запросов на сеть.
* [журнал транзакций](https://api-ru.iiko.services/docs#tag/Report):
    * общий лимит на сеть или одиночный РМС: 2 тяжелых одновременных запроса или до 10-и «легких».

        | ![Information](/resources/Storage/api-documentations/project-iikoweb/info.png) | «Легкие» запросы — это запросы по одному гостю за непродолжительный период. |
        | --- | --- |
    * Дополнительно:
        * до 15 одновременных запросов суммарно от всех;
        * до 5 запросов в секунду от каждой сети.

**Принцип расчета одновременно выполняющихся запросов:**

* **Лимит** — 100 одновременных запросов.
* Запросы поступают и выполняются **одновременно**.
* Когда завершается обработка одного запроса, освобождается одно место для нового.
* Если **общее количество обрабатываемых запросов уже равно или превышает 100**, система отклонит новые запросы с ошибкой.

**Пример:**

1. В системе 98 выполняющихся запросов.
2. Одновременно приходят 4 новых.
3. Первые 2 запроса (99-й и 100-й) займут свободные места и начнут выполняться.
4. Следующие 2 запроса (101-й и 102-й) **превысят лимит в 100** и будут отклонены с ошибкой.
