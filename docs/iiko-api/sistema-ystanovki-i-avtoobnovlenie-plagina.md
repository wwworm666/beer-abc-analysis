Начиная с версии 7.4.6 в iikoFront поддерживается механизм автообновления плагинов. Данный функционал разработан таким образом, чтобы обеспечить максимальную совместимость с существующими плагинами. Это позволит наиболее гладко и безопасно перейти на использование новых возможностей, которые появляются с внедрением данной функции продукта.
 
## Компоненты системы автообновления плагинов

Технически, данный механизм представляет собой два дополнительных плагина в iikoFront, которые предустанавливаются инсталлятором и сервис для распространения обновлений плагинов:

* Плагин **Resto.** **Front.Api.** **Updater** - основной компонент системы автообновления. Выполняет непосредственно планирование, поиск обновлений и их установку.
* Плагин **Resto.** **Front.** **Api.Configurator** - вспомогательный компонент системы автообновления. Предназначен для удаленной настройки установленных в iikoFront плагинов. Предоставляет информацию о соответствии ModuleId и имени плагина.
* Глобальный сервис для размещения обновлений плагинов - портал на базе продукта Nexus Repository Manager, на котором будут размещаться обновления плагинов (в ручном и/или автоматическом режиме). Web-интерфейс портала доступен по адресу https://repository.iiko.online/.

Плагины размещаются на сервере и передаются конечным потребителям в виде единого файла архива – пакета. Такой файл-пакет (по сути zip-архив) имеет расширение «.nupkg» и содержит в себе все необходимые файлы для работы самого плагина.
 
## Лицензирование

Функционирование системы автообновления контролируется через систему лицензирования. Такой подход позволяет централизованно включать и отключать работу автообновления с точностью до CRMID. Дополнительно, работой системы на конкретном терминале можно управлять через конфигурационный файл плагина Resto.Front.Api.Updater (см. Настройки).
 
Для Updater-плагина выделен ModuleId = **0021035101**. Если он включен в лицензию, то плагин начинает свою работу - регулярно проверяет и устанавливает обновления для других плагинов. В противном случае - плагин Resto.Front.Api.Updater переходит в дежурный режим, периодически проверяя наличие/добавление лицензии (начиная с версии iikoFront 7.8.6, до этого – просто отключался до следующего запуска приложения).
 
## Именование плагинов

При разработке системы автообновления плагинов были формализованы правила именования плагинов. Чтобы разрешить разработчикам поддерживать несколько версий Front API для одного и того же плагина было решено, что для каждой из поддерживаемых версий Front API будет создаваться отдельный пакет. Каждый такой пакет будет иметь имя, состоящее из основного имени плагина и замыкающего суффикса. Этот суффикс должен строго соответствовать той версии iikoFront API для которой собран этот пакет.
 
Таким образом имя пакета с плагином должно быть соответствовать правилам:

* &lt;PackageName&gt; -&gt; &lt;PluginName&gt;[.&lt;APISuffix&gt;](.|-)&lt;version&gt;
* &lt;PluginName&gt; -&gt; string (допустимы символы «a-z», «A-Z», «.»)
* &lt; APISuffix &gt; -&gt; V#[Preview#]
* &lt;version&gt; -&gt; #.#[.#[.#]][-[&lt;label&gt;]]
* &lt;label&gt; - string

, где
 
[…] – необязательная часть
 
# - число
 
(a|b) - альтернативные варианты
 
Имя файла пакета с плагином обязательно должно иметь расширение «\*.nupkg».
 
*Примеры*:
 
Resto.Front.Api.ArcusPlugin.V5.1.0.0.3.nupkg – пакет плагина Resto.Front.Api.ArcusPlugin для Front API V5. Версия сборки плагина 1.0.0.3.
 
PrintNumberOnOrderClose.V6Preview4.1.0.0.4.nupkg – пакет плагина PrintNumberOnOrderClose для Front API V6Preview4. Версия плагина 1.0.0.4.
 
В имени пакета всегда присутствует часть, которая отвечает за **версию** пакета. Версия играет важную часть в автообновлении плагина. Обновление **всегда** ведется на версию с большим номером. Невозможно обновление плагина из пакета с меньшей (или равной) версией, чем у уже установленного плагина. При вычислении старшинства версий применяются правила семантического версионирования (подробнее см. [https://semver.org/lang/ru/](https://semver.org/lang/ru/)). При использовании этих правил существуют понятия:

* Стабильная версия – версия продукта, который прошел полное тестирование. Должна соответствовать виду: #.#[.#[.#]]. Т.е. содержит только от 2 до 4 числовых полей.
* Нестабильная версия – тестовая/отладочная версия, которая еще не прошла полного цикла тестирования, может содержать ошибки. Такой версией считается та, в которой кроме основной части (#.#[.#[.#]]) присутствуют одна или более меток (&lt;label&gt;). Метки могут быть как числовыми, так и строковыми полями, разделенными «.». При этом, если у двух версий основная часть одинакова, то далее сравниваются метки версий: числовые метки сравниваются как числа, символьные сравниваются лексикографически. Но версия без меток считается более старшей чем такая-же, но с метками. (т.е., например, версия 1.2.3 &gt; 1.2.3-rc1)

В соответствии с этими понятиями при автообновлении принимается разделение на обновление со стабильного канала (содержит **только** стабильные версии), и с «нестабильного» канала (содержит все версии - стабильные и нестабильные). По умолчанию, выбор пакетов для установки ведется только со **стабильного канала** обновлений. Чтобы разрешить обновление какого-либо плагина с нестабильного канала требуется произвести дополнительную настройку (см. раздел Настройки, параметр *PluginsOnUnstableUpdateChannel*).
 
При поиске обновления плагина выбираются пакеты под самую новую поддерживаемую версию API (это делается для автоматической миграции на новые версии API), затем среди них выбирается пакет с наибольшим номером сборки.
 
## Схема работы обновления

Опуская технические детали, алгоритм работы автообновления выглядит так:

1. Дождаться условий для запуска поиска обновлений (есть лицензия на автообновление и подошло время очередной итерации).
2. Получить список соответствий ModuleId (из текущей лицензии) и имен плагинов из Resto.Front.Api.Configurator.
3. Получить список уже установленных в iikoFront плагинов и их версий.
4. Произвести **поиск** новых версий, установленных или запланированных к установке плагинов. Поиск выполняется одновременно на всех доступных источниках плагинов, при этом делается попытка перевести плагин на более новую версию Front API. Составить план обновления.

*Пример*:
 
Установлен плагин Resto.Front.Api.PluginName v.1.0.0 (FRONT API v5)
 
Найдены версии:

* Resto.Front.Api.PluginName.V6 v1.1.0
* Resto.Front.Api.PluginName.V7Preview3 v0.0.3

Результат:

* Удаление Resto.Front.Api.PluginName v.1.0.0
* Установка Resto.Front.Api.PluginName.V7Preview3 v0.0.3 (т.к. версия Front API v7-Preview3 считается приоритетной).
* Если были найдены пакеты для установки/обновления плагинов – скачать их во временную папку.
* Запланировать следующий поиск обновлений не ранее чем истечет интервал времени, указанный в настройках плагина (по умолчанию - 6 часов). Если обновления для плагинов не найдены, то перейти к п.1.
* Если обновления были найдены, то до наступления времени следующей итерации поиска:
    1. Дождаться **закрытия кассовой смены и блокировки** терминала. Если этого не произошло до наступления времени следующей итерации поиска обновлений, то перейти к п.1.
    2. Выполнить установку плагинов в соответствии с планом из п.4. Для каждого плагина выполнить:
    3. Выгрузить старую версию плагина.
    4. Создать архивную копию.
    5. Удалить старые файлы плагина. Дополнительно, если при удалении строй версии плагина, было обнаружено что config-папке – пусто, то туда делается копия \*.dll.config-файла из папки плагина.
    6. Распаковать плагин в папку в соответствии с правилами именования папок плагинов
    7. Выполнить миграцию данных и настроек со старой версии плагина (при изменении имени пакета, или переходе на другую версию Front API).
    8. Запустить новую версию плагина.
    9. Если на каком-то из этапов c-g произошел сбой, то выполнить восстановление предыдущей версии плагина из резервной копии.
    10. Если на предыдущем шаге был установлен/обновлен плагин Resto.Front.Api.Configurator, то повторить процедуру поиска обновлений (начиная с шага 2)

Во время обновления плагинов система в папке плагинов могут периодически возникать папки с именами вида «\.delete-me.\*». Такие папки являются временными и регулярно удаляются системой. Если при завершении работы iikoFront такая папка, по какой-то причине, осталась не удаленной, то она будет удалена при следующем запуске приложения. Ручное удаление папки, тоже не должно приводить к каким-либо проблемам.
 
## Установка новых плагинов

Первичная установка iikoFront выполняется только со строго ограниченным перечнем плагинов:

* Resto.Front.Api.Updater
* Resto.Front.Api.Configurator
* Resto.Front.Api.iikoTransport
* Resto.Front.Api.Delivery
* Resto.Front.Api.DeliverySms

Дополнительные плагины раньше устанавливались только вручную – распаковкой zip-архива в соответствующую подпапку каталога **%ProgramFiles%\iiko\iikoFront\Plugins**.
 
Разработка системы автообновления плагинов позволяет автоматизировать процесс разворачивания приобретенных клиентом плагинов. Система выполняет поиск установленных плагинов, и сверяет полученный список с прописанными в лицензии идентификаторами. В результате этого сравнения получается список плагинов, которые необходимо развернуть на данном терминале. Для них также выполняется поиск и установка по описанному выше алгоритму.
 
Ручная установка плагинов по-прежнему сохраняется, как одна из возможностей. Система автообновления для таких плагинов так же будет работать. Но стоит учитывать, что если плагин не известен системе, то в качестве имени плагина будет взято имя папки, в которую были распакованы его файлы. Если же ModuleId такого плагина зарегистрирован для какого-то конкретного имени, то тогда будет использовано именно зарегистрированное имя.
 
Система автообновления работает для всех плагинов, установленных в iikoFront. Даже если лицензия на какой-то из них завершилась или еще не приобретена, то он все равно будет обновляться. Это делается для того, чтобы все плагины всегда были в актуальном состоянии и готовы к немедленному использованию после активации лицензии (т.е. находились в «теплом» режиме).
 
## Источники пакетов (плагинов)

Для поиска обновлений используются так называемые **источники пакетов**. Эти источники, как правило, представляют собой специальные http-серверы, на которые периодически выкладываются новые версии плагинов.
 
В общем случае источником может служить и обычная папка на диске и даже сетевая папка.
 
«Из коробки» сервис автообновлений настроен на использование двух предопределенных источников:

1. **iikoCloud** – основной официальный источник загрузки плагинов. Этот источник доступен в браузере по адресу: https://repository.iiko.online/#browse/browse:front-plugins-official (в системе автообновления доступ идет через nuget api по адресу: https://repository.iiko.online/repository/front-plugins-official/)
2. **iikoServer built-in** - встроенное хранилище на сервере iikoRMS. Адрес этого источника вычисляется относительно адреса самого сервера как: http://localhost:8080/resto/update/plugins/. Этот источник обновлений имеет упрощенную функциональность и предназначен для выделенных и изолированных установок iikoRMS. В этот источник обновления плагинов выкладываются администратором сервера iikoRMS вручную. Не следует хранить в этом источнике разные версии плагинов.

## Настройки

Изначально, система автообновления не нуждается в какой-либо настройке. Но существует несколько параметров, которыми можно повлиять на работу системы. Все они, как правило, используются для тестирования и отладки на конкретном терминале. В обычном режиме менять их не требуется.
 
Параметры расположены в файле **%AppData%\ \iiko\CashServer\PluginConfigs\Resto.Front.Api.Updater** :
 
**OperatingMode** – режим работы автообновления. «Enabled» - нормальный режим работы с установкой найденных обновлений, «Disabled» - полностью отключено (даже если есть лицензия), «PreviewChanges» - только поиск обновлений (без установки), запись в лог результатов.
 
**UpdateInvocationPeriod** – интервал проверки наличия обновлений для установленых плагинов в формате hh:mm:ss. По умолчанию – 06:00:00
 
**CustomPluginSources** – список дополнительных источников с обновлениями плагинов. По умолчанию – пустой.
 
**PluginsOnUnstableUpdateChannel** – список плагинов (ModuleId), которым разрешено обновляться с тестового канала обновлений (т.е. обновляться на не стабильные версии). Идентификаторы плагинов перечисляются в виде вложенных элементов (&lt;Id&gt;######&lt;/Id&gt;)
 
## Ручное обновление

Параллельно с автоматической проверкой обновлений для плагинов (по умолчанию производится не чаще через каждые 6 часов) существует возможность запустить такую проверку принудительно – из дополнительного меню в интерфейсе iikoFront. Сделать это можно в любой момент (никак не повлияет на цикл регулярных проверок). При этом, если кассовая смена еще не была закрыта на терминале, то будет выведено сообщение с предупреждением и рекомендацией во время проверки не совершать операций с заказами. Так делается из-за того, что в случае установки обновлений, плагины, связанные с обработкой/оплатой заказов, могут быть кратковременно выгружены, а затем снова загружены.
 
## Разработка и публикация плагина

При разработке плагинов, чтобы обеспечить их готовность к распространению и автообновлению, следует придерживаться следующих рекомендаций:

1. Разработчику рекомендуется использовать стандартный механизм сохранения настроек плагина в классе Properties\Settings.cs с использованием встроенного в VisualStudio редактора настроек. Во время работы плагина эти настройки будут неявно читаться сначала из выделенной для плагина папки (см п.2) файл \*.dll.config, а затем уже (при отсутствии, например) из папки, в которую установлен сам плагин.
2. Настройки плагина хранятся в специально предназначенной для этого папке. Путь к папке можно получить следующим образом:
 var pluginConfigDirPath = PluginContext.Integration.GetConfigsDirectoryPath();
3. Рабочие данные плагина так же следует хранить в отведенной для этого папке. Путь к папке предоставляется:
 var pluginDataDirPath = PluginContext.Integration. GetDataStorageDirectoryPath();
4. При разработке плагинов iikoFront **настоятельно** рекомендуется подключать пакет Resto.Front.Api.PluginPackaging. Он настраивает проект для **правильной** упаковки nuget-пакета с плагином.
5. При подключении в проект iikoFront-плагина дополнительных nuget-пакетов все они (за исключением iikoFront API пакета), должны иметь атрибут PrivateAssets = all
6. Для корректного, автоматического версионирования плагина рекомендуется воспользоваться nuget-пакетом https://www.nuget.org/packages/Nerdbank.GitVersioning (см описание и инструкцию по использованию на https://github.com/dotnet/Nerdbank.GitVersioning ).

Для упрощения разработки и обеспечения совместимости с новейшими версиями VisualStudio рекомендуется использовать «SDK-style» проекты. Для создания проект нужно в VisualStudio выбрать File -&gt; New/Add -&gt; Project -&gt; Visual C# -&gt; .NET Core -&gt; Class Library. Далее в Solution Manager используя контекстное меню созданного проекта выбрать пункт «Edit ClassLibrary1.csproj». В редакторе изменить узел &lt;TargetFramework&gt; следующим образом:
 
&lt;PropertyGroup&gt;
 
&lt;TargetFramework&gt;net472&lt;/TargetFramework&gt;
 
&lt;/PropertyGroup&gt;
 
После этого файл проекта можно закрыть – теперь компиляция проекта будет вестись под указанную версию Net Framework.
 
## Подготовка пакета с плагином

Для установки и автообновления плагинов используется инфраструктура управления пакетами NuGet. Это позволяет разработчикам использовать привычные инструменты для создания и обработки пакетов с плагинами. При создании nuget-пакета с плагином необходимо выполнять все стандартные требования к ним. Дополнительно стандартным, накладываются следующие ограничения:

* Пакет с плагином может содержать только одну внешнюю зависимость (dependency) и только на пакет, соответствующий версии FrontAPI - Resto.Front.Api.V\*. Все остальные внешние зависимости плагина должны быть упакованы в этот же пакет (п.5).
* Файлы плагина должны быть размещены либо в папке /lib внутри пакета, либо во вложенной папке, соответствующей версии .Net Framework (например, /lib/net472). Версия Net Framework должна быть такой же, как и версия в используемом пакете Froте API.
* Папка /lib в пакете
* Имя пакета с плагином должно содержать суффикс, который соотвествует используемой версии Front.API.
* В пакете должен находиться файл manifest.xml, который содержит дополнительные метаданные плагина.
* Другие служебные папки папки nuget-пакета (/build, /Content и др.) в настоящее время никак не обрабатываются и не используются при распаковке плагина.

Для облегчения соблюдения указанных требований предназначен nuget-пакет Resto.Front.Api.PluginPackaging (п. 4). Он производит валидацию настроек проекта, генерирует (если он не указан явно в проекте) или дополняет файл manifest.xml. При использовании этого пакета нет необходимости в явном ручном создании файла manifest.xml в проекте. Если в дополнение к стандартным метаданным nuget-пакета требуется указать дополнительную информацию, то можно воспользоваться либо встроенными свойствами проекта (PackageDescription, Authors и др. см документацию в MSDN) либо создать собственный \*.nuspec-файл в проекте, используя предопределенные плейсхолдеры:

* $id$ - идентификатор пакета вместе с Api-суффиксом (создается на основе имени проекта)
* $author$ - автор пакета
* $version$ - версия пакета
* $configuration$ - конфигурация проекта, в которой ведется сборка
* $apiVersion$ - версия Froте API
* $outdir$ - папка в которой размещены бинарные файлы плагина после сборки

## Публикация новых версий плагина

Для размещения плагина в системе автообновления необходимо **опубликовать** его пакет на каком-либо зарегистрированном для данного сервера RMS источнике пакетов. Обычно пакеты публикуются на общем глобальном источнике (iikoCloud), но в некоторых случаях, используется встроенный (локальный для данного сервера RMS) источник пакетов. Для публикации пакета с плагином в таком источнике, достаточно разместить его файл в папке &lt;iikoRMSServerRoot&gt;\exploded\update\plugins.
 
Для публикации же пакетов в общем для всех клиентов источнике у разработчика плагинов есть две возможности – самостоятельная публикация и автоматизированная публикация для тех плагинов, чей исходный код размещен на сервере [http://gitlab.iiko.ru](http://gitlab.iiko.ru).
 
## Самостоятельная публикация

Для самостоятельной публикации плагина, разработчику необходимо получить, по предварительной заявке, свою учетную запись на сервере [https://repository.iiko.online/](https://repository.iiko.online/) , API-ключ и Url-адрес для публикации пакетов. Далее он самостоятельно производит публикацию пакетов с плагинами либо через Web-UI этого сервера либо используя командную строку nuget.exe:
 
**nuget push foo.nupkg &lt;API-KEY&gt; -src &lt;push-url&gt;**

## Автоматизированная публикация в gitlab.iiko.ru

Те разработчики, которые пользуются возможностью размещать свой код на сервере [http://gitlab.iiko.ru](http://gitlab.iiko.ru) имеют возможность автоматизированной публикации плагинов.
 
Обычно разработка плагина ведется в ветке основной master/main репозитория. В тот момент, когда разработчик убедился в стабильности работы своего проекта и хочет опубликовать плагин на общий сервер iikoCloud, он должен выполнить:

1. Убедиться, что версия собираемого пакета – правильная, т.е. как минимум больше чем предыдущая опубликованная версия.
2. Сделать новую ветку(branch) в текущей стабильной ревизии кода. Имя ветки должно соответствовать шаблону: /release/\*. Вместо «\*» рекомендуется использовать первые два числовых поля из выпускаемой версии, которые не будут изменяться при выпуске хотфиксов (если такие потребуются).

Далее сработает предварительно настроенный конвеер сборки, который соберет и автоматически опубликует зафиксированную версию плагина.
 
Те разработчики, которые используют в проекте пакет Nerdbank.GitVersioning могут воспользоваться утилитой nbgv (подробнее см. [https://github.com/dotnet/Nerdbank.GitVersioning/blob/master/doc/nbgv-cli.md](https://github.com/dotnet/Nerdbank.GitVersioning/blob/master/doc/nbgv-cli.md)):
 
**nbgv prepare-release**
 
Эта утилита сама сделает необходимую новую ветку и нарастит версию и сконфигурирует текущую (ветку).
 
## Действия при обнаружении критических проблем в опубликованной версии плагина

Иногда может возникнуть ситуация, когда опубликованный плагин содержит критическую проблему, нарушающую нормальное функционирование клиентов. Требуется срочно вернуться к одной из предыдущих версий плагина и предотвратить «расползание» сбойной версии по всем пользователям iikoFront.
 
Это не стандартная ситуация. Автообновление подразумевает последовательный переход от более младшей версии к более старшей. В общем случае, невозможно осуществить обратный переход. Для решения создавшейся проблемы разработчику плагина необходимо:

1. Определить те изменения в коде плагина, которые привели к сбою.
2. Сделать откат этих изменений используя git или другую SCM, если быстрая починка ошибки невозможна. Или сделать быстрое исправление проблемы.
3. Сохранить откат/исправление как новый commit в SCM-системе разработчика
4. Собрать новую версию пакета с плагином. Обязательно убедиться, что версия нового пакета больше версии проблемного пакета.
5. Произвести публикацию нового пакета.
6. Сообщить клиентам, которые успели получить сбойную версию провести ручное обновление плагинов. Либо они автоматически его получат при следующей итерации поиска обновлений.
7. Далее разработчик может в спокойном режиме произвести анализ сбоя и реализовать полноценное его исправление и выпустить следующее обновление.
