## Общие правила для размещения на сервере gitlab.iiko.ru исходного кода (v.0.5)

### Общие рекомендации

1. В корне репозитория должен обязательно размещаться файл «.gitignore».
2. В корне репозитория может содержаться скрипт-файл для сборки проекта (build.xml, build.proj) или sln-файл.
3. В корне репозитория может быть также размещен файл Readme.txt с кратким описанием проекта и инструкцией по сборке.
4. Файлы проектов (.csproj, .wixproj, .vcxproj и др.) должны быть размещены в соответствующих подкаталогах. Исходный код проектов не должен размещаться в корне репозитория.
5. Проекты должны использовать Nuget для своих внешних зависимостей. Если не существует официального nuget-пакета, то возможно добавление ссылок на бинарные сборки, не выходящие за пределы структуры репозитория. Ссылка на сборки iikoFront API v5, v6 и тд. должна быть только через nuget-пакет.
6. Проекты не должны содержать ссылок на COM объекты по GUID (COM-reference). Вместо них следует использовать ссылки на Interop-сборки (dll).
7. Если для проекта требуются дополнительные бинарные сборки (не входящие в nuget-пакеты), то их рекомендуется разместить в папке \Binaries в корне репозитория.
8. Проект должен собираться из репозитория и не зависеть напрямую от других проектов в GitLab. То есть, если двум проектам в двух разных репозиториях требуется общая сборка из какого-либо 3-го проекта, то такой проект должен быть выделен в отдельный репозиторий и использоваться либо как git-submodule либо должен собираться отдельно и публиковаться как nuget-пакет и использоваться в таком виде.

### Выделение части кода в отдельный репозиторий вместе с историей изменений

Иногда возникает необходимость выделить часть кода из большого репозитория в отдельный небольшой, желательно сохранив при этом всю историю изменений выделяемой части. При использовании git в качестве системы управления версиями кода такая задача может быть решена. Далее предполагается что:

* команды начинают выполняться в корневой папке исходного репозитория,
* текущий репозиторий обновлен до актуального состояния, выбрана ветка для выделения истории,
* весь код, который нужно выделить в отдельный репозиторий находится в отдельной папке /path/ProjectToExtract,
* имеется созданный репозиторий git`@`gitlab`.`iiko`.`ru`:<`user`_`or`_`organization`>/`new`-`repo`.`git`, `в который необходимо переместить выделяемый код вместе с историей.

Необходимо выполнить ряд команд в интерпретаторе bash:

1. **pushd .**
 запоминаем папку текущего репозитория
2. **git clone --no-hardlinks . ../repo.tmp**
 клонируем текущий репозиторий в соседнюю папку. Далее вся работа ведется в созданной копии.
3. **cd ../repo.tmp**
4. **git remote rm origin**
 отцепляем удаленный репозиторий, чтобы случайно ничего не сломать в нем.
5. **git filter-branch --subdirectory-filter /path/ProjectToExtract -- --all**
 После выполнения этого шага в локальном репозитории останется только контент папки /path/ProjectToExtract, причем самой папки больше нет – все содержащиеся в ней файлы теперь лежат в текущей директории (в repo.tmp). Команда выполняется **очень** долго. На репозиториях с большой историей может выполняться несколько часов.
6. Все файлы проекта сейчас лежат в корне, если их нужно перенести в папку, которая будет лежать в корне. То лучше воспользоваться данной командой (**NEW\_FOLDER\_NAME** - название папки, куда вы всё будете перемещать): 
 **git filter-branch -f --index-filter 'git ls-files -s | sed "s:\t\"\*:&NEW\_FOLDER\_NAME/:" | GIT\_INDEX\_FILE=$GIT\_INDEX\_FILE.new git update-index --index-info && mv "$GIT\_INDEX\_FILE.new" "$GIT\_INDEX\_FILE"' HEAD**
7. **git push git@gitlab.iiko.ru:&lt;user\_or\_organization&gt;/new-repo.git HEAD:extract --force**
8. **popd**
 восстановить исходную папку
9. **rm –r –d -I ../repo.tmp**
 удаляем временный рабочий клон репозитория.

    Теперь с репозиторием git@gitlab.iiko.ru:&lt;user\_or\_organization&gt;/new-repo.git можно работать как обычно – в нем есть ветка «extract», которую можно будет сливать в основную ветку и/или производить в ней дополнительные преобразования кода.

### Перенос кода iikoFront-плагина из общего репозитория в отдельный. Настройка сборки. 

Для выделения сборки iikoFront-плагина в отдельный сборочный процесс с хранением кода в отдельном репозитории необходимо выполнить:

1. Создать новый репозиторий на сервере [https://gitlab.iiko.ru](https://gitlab.iiko.ru) в группе «iikoFront» ([https://gitlab.iiko.ru/iikofront](https://gitlab.iiko.ru/iikofront), кнопка «New Project»)
2. Перенести исходный код плагина в новый репозиторий, используя один из возможных вариантов:
3. Без сохранения истории изменений из исходного репозитория – просто создать в корне репозитория папку с именем плагина (/MyPluginName) и переместить в нее файлы проекта из оригинального репозитория. Сохранить перенесенный код в новом репозитории и удалить в старом.
4. С сохранением истории изменений – используя последовательность действий из раздела «Выделение части кода в отдельный репозиторий вместе с историей изменений». После этого переместить код в репозитории так, чтобы все исходные файлы были в отдельной папке в корне репозитория (например /MyPluginName)
5. Создать в корне нового репозитория solution-файл (\*.sln) и добавить в него проект перенесенного плагина. В solution-файле обязательно настроить конфигурацию «Release|Any CPU» - она, по умолчанию, будет собираться на TeamCity.
6. Сохранить изменения в репозитории.
7. Убедиться, что структура репозитория соответствуют разделу «Общие рекомендации».
8. Убедиться, что структура проекта удовлетворяет требованиям документа «iikoFront - Автообновление плагинов», раздел «Разработка и публикация плагина».
9. Обратиться в iiko для формирования сборки, e-mail **api@iiko.ru**
