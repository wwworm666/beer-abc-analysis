При вызове любых методов Cloud Api интеграция должна обрабатывать полученные ошибки. Ошибкой считается получение http-статуса ответа 4xx или 5xx. В этих случаях тело ответа содержит более подробную информацию о произошедшей ошибке.

Также интеграция должна сигнализировать разработчику интеграции о происходящих ошибках, чтобы он мог выяснить их причины и исправить. Если ошибки происходят постоянно, это свидетельствует о проблеме, которая требует решения. Если интеграция игнорирует ошибки и продолжает выполнять запросы, которые к ним приводят, то она будет заблокирована, т.е. Cloud Api перестанет выполнять любые запросы от данного апи-логина.

**Если и после блокировки апи-логина интеграция по-прежнему пытается выполнять запросы и игнорирует получаемые ошибки, то будут блокироваться все запросы, приходящие с этого ip-адреса. Это может привести к полной блокировке не только проблемной интеграции, но и других интеграций, которые работают корректно, но выполняются на том же хостинге, что и проблемная интеграция.**

Далее перечислены некоторые стандартные коды ошибок, возможные причины их возникновения и способы обработки.

## 500 Internal Server Error

Cloud Api возвращает ошибку 500 при каких-то ошибках на стороне Cloud Api. Интеграция может попробовать повторить запрос через какое-то время; если проблема была временной и была устранена, то запрос выполнится успешно. Если запрос постоянно завершается ошибкой 500, интеграция должна прекратить его выполнять и сигнализировать о проблеме. Разработчик интеграции должен обратиться в тех.поддержку iiko с описанием проблемы.

## 400 Bad Request

Cloud Api возвращает ошибку 400 в случае некорректного запроса (например, в body запроса не указаны какие-то обязательные для его выполнения поля). Как правило, эта ошибка свидетельствует о том, что интеграция написана неверно. Точная причина ошибки как правило содержится в ответе запроса, например:

```
Error converting value {null} to type 'System.Guid'. Path 'productId', line 66, position 25. 
```

Интеграция не должна пытаться повторять такой запрос, т.к. при последующих вызовах она будет получать ту же ошибку. В случае массовых ошибок 400 интеграция должна прекратить выполнять проблемные запросы и сигнализировать о проблеме. Разработчик интеграции должен выяснить и устранить причину проблемы.

Если проблема не будет устранена, интеграция будет заблокирована, т.е. Cloud Api перестанет выполнять любые запросы от данного апи-логина.

## 401 Unauthorized, 402 Payment Required, 423 Locked

Для любого метода Cloud Api, кроме метода /api/1/access\_token, ошибка 401 может возвращаться в следующих случаях: - истекло время жизни токена, который передаётся в заголовке Authorization запроса - апи-логин заблокирован

В случае получения ошибки 401 при вызове любого метода Cloud Api, кроме метода /api/1/access\_token, интеграция должна однократно вызвать метод /api/1/access\_token и получить новые токен. Если метод /api/1/access\_token выполнится успешно и токен будет получен, значит время жизни предыдущего токена истекло, и вместо него интеграция теперь должна передавать в заголовке Authorization новый токен. Это штатная ситуация, срок жизни токенов ограничен, поэтому они должны регулярно обновляться. При этом не следует вызывать метод /api/1/access\_token перед вызовом каждого метода, это избыточно. Следует использовать полученный токен до тех пор, пока он истечёт, т.е. пока при вызове очередного метода с этим токеном не будет получена ошибка 401.

Если метод /api/1/access\_token тоже возвращает ошибку 401, это означает, что апи-логин заблокирован, т.е. Cloud Api не будет выполнять никакие методы от имени этого апи-логина. Возможные причины блокировки апи-логина: - закончилась лицензия на Cloud Api. В этом случае в ответе запроса будет текст "Apilogin`s license for using the Cloud API has expired." Следует обратиться в тех.поддержку iiko для восстановления лицензии. После восстановления лицензии апи-логин будет разблокирован. - интеграция ранее систематически выполняла запросы, которые завершались другими ошибками (400, 429 и т.д.), и игнорировала эти ошибки; проблемы, о которых сигнализируют эти ошибки, не были решены. В случае блокировке в ответе запроса будет текст "Apilogin has been blocked." Следует обратиться в тех.поддержку iiko для выяснения причин блокировки и их устранения.

Также ошибка 401 при вызове метода /api/1/access\_token будет возвращаться, когда сам запрос выполнен неверно: - Login ... is not authorized. - в поле apiLogin передан несуществующий логин - ApiLogin is absent. - поле apiLogin отсутствует - Value cannot be null. (Parameter 'apiLogin') - поле apiLogin пустое

Интеграция не должна пытаться повторять запрос, завершившийся ошибкой 401, т.к. при последующих вызовах она будет получать ту же ошибку. Исключение представляет однократный повторный вызов при истечении срока жизни токена (см. выше). В случае массовых ошибок 401 интеграция должна прекратить выполнять проблемные запросы и сигнализировать о проблеме. Разработчик интеграции должен выяснить и устранить причину проблемы.

**Если проблема не будет устранена, то со стороны Cloud Api будет невозможно заблокировать проблемную интеграцию, т.к. ошибка 401 чаще всего означает, что интеграция уже заблокирована. В этом случае будут блокироваться все запросы, приходящие с этого ip-адреса. Это может привести к полной блокировке не только проблемной интеграции, но и других интеграций, которые работают корректно, но выполняются на том же хостинге, что и проблемная интеграция.**

В некоторых случаях вместо ошибки 401 Cloud Api возвращает ошибку 402 или 423. Интеграция должна обрабатывать эти ошибки одинаковым образом как описано выше.

## 403 Forbidden

Cloud Api возвращает ошибку 403 в случае некорректного запроса (например, в body запроса указан идентификатор организации, которая не привязана к данному апи-логину). Точная причина ошибки как правило содержится в ответе запроса.

Интеграция не должна пытаться повторять такой запрос, т.к. при последующих вызовах она будет получать ту же ошибку. В случае массовых ошибок 403 интеграция должна прекратить выполнять проблемные запросы и сигнализировать о проблеме. Разработчик интеграции должен выяснить и устранить причину проблемы.

Если проблема не будет устранена, интеграция будет заблокирована, т.е. Cloud Api перестанет выполнять любые запросы от данного апи-логина.

## 404 Not Found

Cloud Api возвращает ошибку 404 в случае вызова несуществующего метода Cloud Api. Скорее всего допущена ошибка в url вызываемого метода.

Интеграция не должна пытаться повторять такой запрос, т.к. при последующих вызовах она будет получать ту же ошибку. В случае массовых ошибок 404 интеграция должна прекратить выполнять проблемные запросы и сигнализировать о проблеме. Разработчик интеграции должен выяснить и устранить причину проблемы.

Если проблема не будет устранена, интеграция будет заблокирована, т.е. Cloud Api перестанет выполнять любые запросы от данного апи-логина. **Возможны ситуации, когда по вызываемому методу невозможно идентифицировать интеграцию, которая его вызывает. В этом случае будут блокироваться все запросы, приходящие с этого ip-адреса. Это может привести к полной блокировке не только проблемной интеграции, но и других интеграций, которые работают корректно, но выполняются на том же хостинге, что и проблемная интеграция.**

## 405 Method Not Allowed

Cloud Api возвращает ошибку 405 в случае некорректного метода http при выполнении запроса. Практически все методы Cloud Api являются POST-запросами, поэтому при попытке вызвать их как GET-запрос интеграция будет получать ошибку 405.

Интеграция не должна пытаться повторять такой запрос, т.к. при последующих вызовах она будет получать ту же ошибку. В случае массовых ошибок 405 интеграция должна прекратить выполнять проблемные запросы и сигнализировать о проблеме. Разработчик интеграции должен выяснить и устранить причину проблемы.

Если проблема не будет устранена, интеграция будет заблокирована, т.е. Cloud Api перестанет выполнять любые запросы от данного апи-логина.

## 408 Timeout

Cloud Api возвращает ошибку 408 при ошибках на стороне Cloud Api, обусловленных повышенной нагрузкой. Интеграция может попробовать повторить запрос через какое-то время; если проблема была временной и была устранена, то запрос выполнится успешно. Если запрос постоянно завершается ошибкой 408, интеграция должна прекратить его выполнять и сигнализировать о проблеме, возможно, она связана не с повышенной нагрузкой на Cloud Api, а с характером данных, с которыми работает данная интеграция. Разработчик интеграции должен обратиться в тех.поддержку iiko с описанием проблемы.

## 410 Gone

Cloud Api возвращает ошибку 410 в случае некорректного запроса /api/1/commands/status: в поле CorrelationId передаётся несуществующий или просроченный идентификатор команды. Идентификатор команды является ограниченным только на протяжении короткого промежутка времени после выполнения команды (30 минут), на протяжении которого метод /api/1/commands/status возвращает статус запрошенной команды. По истечении этого промежутка времени метод /api/1/commands/status будет возвращать ошибку 410.

Интеграция не должна пытаться повторять такой запрос, т.к. при последующих вызовах она будет получать ту же ошибку. В случае массовых ошибок 410 интеграция должна прекратить выполнять проблемные запросы и сигнализировать о проблеме. Разработчик интеграции должен выяснить и устранить причину проблемы.

Если проблема не будет устранена, интеграция будет заблокирована, т.е. Cloud Api перестанет выполнять любые запросы от данного апи-логина.

## 422 Unprocessable Content

Cloud Api возвращает ошибку 422 в случае, если запрос может привести к выборке слишком большого объёма данных (например, запрошены заказы за очень большой диапазон дат или для большого количества организаций). Точная причина ошибки как правило содержится в ответе запроса. Интеграция должна быть исправлена таким образом, один проблемный запрос был разделён на несколько возвращающих меньшее количество данных.

Интеграция не должна пытаться повторять такой запрос, т.к. при последующих вызовах она будет получать ту же ошибку. В случае массовых ошибок 422 интеграция должна прекратить выполнять проблемные запросы и сигнализировать о проблеме. Разработчик интеграции должен выяснить и устранить причину проблемы.

Если проблема не будет устранена, интеграция будет заблокирована, т.е. Cloud Api перестанет выполнять любые запросы от данного апи-логина.

## 429 Too Many Requests

Cloud Api возвращает ошибку 429 в случаях, когда какой-либо метод вызывается слишком часто. Для каждого метода установлен свой лимит вызовов в единицу времени, по достижении которого интеграция при вызове этого метода начинает получать ошибку 429. Например, если для метода установлен лимит 10 запросов в час, то первые 10 запросов будут выполнены успешно, а все последующие будут завершаться ошибкой 429. По истечении часа отсчёт начнётся заново, т.е. интеграция сможет выполнить ещё 10 запросов.

Если запрос постоянно завершается ошибкой 429, интеграция должна прекратить его выполнять и сигнализировать о проблеме. Разработчик интеграции должен уменьшить частоту выполнения проблемного запроса. О рекомендованных лимитах запросов можно узнать в тех.поддержке iiko.

Если интеграция систематически получает ошибки 429, и количество таких неуспешных запросов сравнимо с количеством успешных, интеграция будет заблокирована, т.е. Cloud Api перестанет выполнять любые запросы от данного апи-логина.
