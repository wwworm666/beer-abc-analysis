# Цели месяца (/goals)

## Что это

Страница-памятка для персонала: **что нужно, чтобы попасть в цель за месяц**, в
одном месте. Сделана под сценарий «в начале месяца скинуть смене её цифры»,
чтобы люди знали, к чему стремиться, а не смотрели в двух разных экранах.

Собирает два уже существующих набора данных:

1. **План выручки по дням** — по каждой точке, из конвейера планов
   (месячный план -> веса дней; пт и сб = ×2, праздники/закрытия — override).
2. **KPI-цели месяца** — цель и минимум по каждому показателю и точке
   (те же, что использует расчёт премии на `/salary`).

Ничего не считает заново — только читает и красиво показывает готовые данные.
Разбивка **по каждой точке отдельно**: переключатель `Все / Крем / Варш / ВО / Лиг`.
Смене одной точки не нужны цифры другой.

Отдача — экраном: открыл, показал/сфоткал с телефона, либо «Печать / PDF»
(`window.print`, печатная вёрстка прячет сайдбар/шапку).

## Файлы

- [`templates/goals.html`](../templates/goals.html) — вся страница (разметка + стили + JS).
  Данных с бэкенда не добавляет: тянет три существующих эндпоинта на фронте.
- [`routes/pages.py`](../routes/pages.py) — маршрут `GET /goals` (рендер шаблона).
- [`templates/shared/nav.html`](../templates/shared/nav.html) — пункт «Цели месяца»
  в сайдбаре (секция «Команда», после «График»).

Источники данных (эндпоинты не менялись):

- `GET /api/schedule/plans/<year>/<month>` -> `{days, plan_formula}` — план/факт по
  дням; берём только `plan` каждой точки (факт из iiko игнорируем — он тут не нужен).
  См. [schedule.md](schedule.md), `core/schedule_plans.py:build_month_plans`.
- `GET /api/schedule/locations` -> `[{id, name, venue_key, short_name}]` — мост
  `location_id <-> venue_key` для чтения планов.
- `GET /api/kpi-targets` -> `{months, defaults, available_metrics}` — KPI-цели,
  метрики/единицы/знаки после запятой. См. [employee.md](employee.md), `core/kpi_calculator.py`.

## Как работает

### Разбивка по точкам

Фикс-список точек на фронте (`BARS`) связывает три представления имени:

| venue_key (планы) | Русское имя (UI, KPI-цели) | Короткое (вкладка) |
|---|---|---|
| `kremenchugskaya` | Кременчугская | Крем |
| `varshavskaya` | Варшавская | Варш |
| `bolshoy` | Большой пр. В.О | ВО |
| `ligovskiy` | Лиговский | Лиг |

- **Планы** читаются по `location_id`: точку находим по `venue_key`
  (`locations.find(l => l.venue_key === bar.key)`), затем берём `days[date].locations[loc_id].plan`.
- **KPI-цели** лежат под русским именем точки в `months["YYYY-MM"]`. Имя в реестре
  KPI и имя точки в графике различаются пунктуацией (**`Большой пр В.О.`** в
  `kpi_targets.json` vs **`Большой пр. В.О`** в `locations`), поэтому имя матчится
  устойчиво: точное совпадение -> совпадение множеств слов без точек
  (`kpiKeyForBar` / `namesMatch`). Проверено на живых данных: все 4 точки резолвятся.

### План выручки по дням (календарь)

Календарь-сетка 7 колонок (Пн–Вс). Для каждого дня месяца:

```
план дня точки = days["YYYY-MM-DD"].locations[location_id].plan   (null = плана нет)
```

- Пт и Сб помечены `×2` и рамкой — вес выходного дня уже учтён в самом значении
  плана (формула на бэке), метка лишь объясняет, почему цифра больше.
- Сумма по всем дням = месячный план точки (сходимость проверена: Крем = 1 200 000 и т.д.).
- Строка «ритма»: средний план буднего дня и средний план пт/сб — быстрый ориентир
  «сколько делать в обычный день / в пятницу-субботу».
- Значения в ячейках сокращены (`fmtK`: `30769 -> 31k`), точное — в `title` ячейки.

### KPI-цели месяца (карточки)

Карточка на каждый показатель из `kpi_config` месяца:

```
Цель  = months["YYYY-MM"][РусскоеИмяТочки][kpiN].target   (крупно, зелёным)
Мин   = months["YYYY-MM"][РусскоеИмяТочки][kpiN].min       (ниже минимума премии нет)
```

Единица (`%`, `₽`, `шт`, `ч`) и число знаков после запятой берутся из
`available_metrics[metric]`. Если цель/минимум = 0 (не заполнено) — карточка
показывает «не задано». Если на месяц нет KPI-целей вовсе — подсказка завести их
на `/salary` -> «Настройка KPI-целей».

### Период по умолчанию

При открытии — **текущий месяц** (`YYYY-MM` в селекторе `input[type=month]`).
Переключается вручную; смена месяца или точки перерисовывает без нового расчёта
(планы/цели перезапрашиваются только при смене месяца).

## Формулы (для тултипов и футера страницы)

```
План дня = месячный план точки / сумма весов месяца × вес дня
           вес дня: пт и сб = 2.0, остальные = 1.0; праздники/закрытия — override
KPI      = цель и минимум за месяц по точке; премия растёт от минимума к цели
```

## Зависимости

### От каких модулей зависит
- `core/schedule_plans.py` (`build_month_plans`) -> план по дням.
- `core/daily_plans_generator.py` -> `daily_plans.json` (материализованные дневные планы).
- `core/kpi_calculator.py` (`KpiTargetsReader`) -> KPI-цели, `available_metrics`.
- `core/shifts_manager.py` (`get_locations`) -> мост `location_id <-> venue_key`.

### Кто использует
- Владелец/менеджер — анонс целей персоналу в начале месяца.

## Ограничения

- Только показ (экран + печать). Экспорта в картинку/Excel и авто-отправки в
  Telegram нет — по решению владельца пока не нужно.
- Читает те же файлы, что дашборд и `/salary`: если месячный план точки не задан
  (дашборд, План/Факт) — план по дням пуст; если KPI-цели на месяц не заведены
  (`/salary`) — блок KPI пуст. Страница честно это показывает, а не молчит.

## Changelog

### 2026-07-05 — Создан модуль «Цели месяца» (/goals)

**Добавлено:** новая страница-памятка для персонала — план выручки по дням + KPI-цели
месяца по каждой точке в одном месте (разбивка `Все / Крем / Варш / ВО / Лиг`),
печать/PDF. Собирается на фронте из трёх существующих эндпоинтов
(`/api/schedule/plans`, `/api/schedule/locations`, `/api/kpi-targets`), без нового
бэкенд-расчёта. Матчинг имени точки для KPI устойчив к пунктуации
(`Большой пр. В.О` vs `Большой пр В.О.`).

**Почему:** в начале месяца планы по дням лежали на `/schedule`, а KPI-цели — на
`/salary`; чтобы скинуть смене её цифры, приходилось смотреть в двух местах.

**Файлы:** `templates/goals.html` (новый), `routes/pages.py` (маршрут `/goals`),
`templates/shared/nav.html` (пункт сайдбара).

**Проверка:** данные проверены на живых файлах — планы по дням суммируются в
месячный план каждой точки, вес пт/сб = ×2 (средний выходной = 2× среднего буднего),
все 4 точки резолвятся к своим KPI-целям; JS-логика рендера прогнана headless над
реальными payload'ами эндпоинтов.
