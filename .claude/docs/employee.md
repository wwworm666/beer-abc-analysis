# Employee Dashboard

## Что это

Рентген для барменов. Выбираешь сотрудника, период — и видишь 12 метрик: от выручки до количества новых карт лояльности. Два режима: «Анализ» одного человека и «Сравнение» нескольких. Живёт на `/employee`.

## Файлы

| Файл | Описание |
|------|----------|
| `templates/employee.html` | Вся страница: 12 карточек + круговая диаграмма + радар сравнения |
| `core/employee_analysis.py` | `EmployeeMetricsCalculator` — мозг, считает все 12 метрик |
| `core/employee_plans.py` | Расчёт планов: какой план у бармена, если он работал в 3 разных точках |
| `core/olap_reports.py` | OLAP-запросы к iiko: выручка, чеки, карты лояльности |
| `core/iiko_api.py` | Кассовые смены: кто, где, когда работал, сколько часов |
| `app.py` | Два endpoint'а: `/api/employee-analytics` и `/api/employee-compare` |
| `data/bar_plans.json` | JSON с планами каждой точки на каждый день |

## Как работает

### Откуда берутся данные

Дашборд собирает данные из 4 источников, причём **6 OLAP-запросов идут параллельно** (иначе ждать 15+ секунд):

```
Cash Shifts API  →  Смены, часы, опоздания, где работал
OLAP Reports     →  Выручка, чеки, скидки, наценка, отмены, карты лояльности
Groups API       →  pointOfSaleId → «Варшавская» (перевод ID в человеческое название)
bar_plans.json   →  Планы по точкам (разные для каждого дня недели!)
```

### 12 метрик — что и откуда

| # | Метрика | Откуда | Зачем |
|---|---------|--------|-------|
| 1 | Выручка | OLAP: `DishDiscountSumInt` | Базовый показатель, но без контекста бесполезен |
| 2 | План/Факт | Cash Shifts + bar_plans.json | Справедливая оценка: учитывает ГДЕ и КОГДА работал |
| 3 | Смены/Часы | Cash Shifts API | Контекст: 500К за 20 смен ≠ 500К за 8 смен |
| 4 | Выручка/смена | Расчётное | Продуктивность смены, независимо от графика |
| 5 | Выручка/час | Расчётное | «Скорость генерации денег» |
| 6 | Кол-во чеков | OLAP: `OrderNum` | Активность: не залипает ли на одном столе |
| 7 | Средний чек | Расчётное | Навык допродаж: рекомендует ли снеки, второй бокал |
| 8 | Наценка | OLAP: `ProductCostBase.MarkUp` | ЧТО продаёт: маржинальный крафт или акции в ноль |
| 9 | % скидок | OLAP: `DiscountSum` | Контроль злоупотреблений: не дарит ли скидки друзьям |
| 10 | Отмены | OLAP: `DeletedWithWriteoff` | Ошибки, возвраты, или потенциальные махинации |
| 11 | Опоздания | Cash Shifts: открытие позже 14:35 | Дисциплина: бар открылся позже = потерянная выручка |
| 12 | Новые карты | OLAP: `Delivery.CustomerPhone` | Развитие лояльности: активно ли предлагает регистрацию |

### Как определяется место работы

Раньше использовали `get_attendance()`, но он для ВСЕХ возвращал "Пивная культура" — бесполезно. Переключились на кассовые смены:

```
Cash Shift → pointOfSaleId → Groups API → «Варшавская»
```

Теперь мы точно знаем, что Никита 15 января работал именно на Варшавской, а не «где-то в сети».

### Ловушка с именами

OLAP пишет «Васильев Никита», а iiko employees — «Никита Васильев». Если сравнивать строки напрямую, employee_id не найдётся и все метрики из кассовых смен будут нули. Решение — сравниваем множества слов:

```python
def normalize_name(name):
    return set(name.lower().strip().split())
# {"васильев", "никита"} == {"никита", "васильев"} → совпадает!
```

### Карты лояльности — не перепутай поля!

Считаем уникальные телефоны (`Delivery.CustomerPhone`), у которых дата создания клиента (`Delivery.CustomerCreatedDateTyped`) попадает в период. Группируем по `WaiterName`.

**Граблями прошлись**: сначала использовали `CardNumber`, а оно оказалось полем для банковских карт (замаскированные Visa/Mastercard). Карты лояльности в iiko — это телефоны, а не номера карт. Подробнее в [lessons.md](lessons.md).

### Расчёт плана — не просто «сумма за месяц»

План сотрудника — это **не** фиксированная цифра. Это сумма дневных планов тех точек, где он реально работал:

```
План = Σ(план_ТТ[дата]) для каждого дня из кассовых смен
```

Планы разные по дням недели (пятница > понедельника):

```json
"2026-01-06": {"Кременчугская": 58354}  // Понедельник
"2026-01-10": {"Кременчугская": 39556}  // Пятница
```

Пример: Артемий отработал 12 смен на Кременчугской. Код суммирует план за каждый конкретный день → 525 238 ₽. Если бы он работал по пятницам вместо понедельников, план был бы другим.

### Маппинг названий точек

iiko и планы называют точки по-разному. Маппинг спасает:

```python
BAR_NAME_MAPPING = {
    "Пивная культура": "Кременчугская",  # главная ловушка
    "Большой пр. В.О": "Большой пр В.О.",  # обрати внимание на точку!
}
```

## API

### POST /api/employee-analytics

```json
{
  "employee_name": "Артемий Новаев",
  "date_from": "2026-01-01",
  "date_to": "2026-01-22"
}
```

Response: `total_revenue`, `total_checks`, `avg_check`, `shifts_count`, `work_hours`, `plan_fact_percent`, `draft_share`, `bottles_share`, `kitchen_share`, `avg_markup`, `discount_percent`, `cancelled_count`, `late_count`, `loyalty_cards_count`

### POST /api/employee-compare

Сравнение 2+ сотрудников. Все OLAP-запросы делаются один раз, потом результат нарезается по каждому имени — не 6×N запросов, а всего 6.
