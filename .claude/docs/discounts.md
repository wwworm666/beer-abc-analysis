# Анализ скидок

## Что это

Инструмент маркетолога для анализа скидок и акций. Показывает ROI каждой скидки (глубина, выручка с/без скидки), сегментацию гостей по лояльности (новички → возвращающиеся → постоянные), и детализацию по каждому гостю: что брал, сколько раз, на какую сумму. Два режима: сводка по всем барам и drill-down в конкретный бар.

## Файлы

| Файл | Назначение |
|------|-----------|
| `core/olap_reports.py` | `get_discount_report()` — детальный OLAP, `get_discount_names()` — лёгкий OLAP для dropdown |
| `app.py` | Роуты: `GET /discounts`, `GET /api/discount-names`, `POST /api/discount-analyze` |
| `templates/discounts.html` | Весь фронтенд: CSS, карточки, таблицы, сегментация, Chart.js doughnut |

## Как работает

### Архитектура: один OLAP — вся логика в Python

Вместо трёх отдельных запросов — **один OLAP с 6 groupBy-полями**, возвращающий строки на уровне блюд. Вся агрегация — в Python.

Почему так? Потому что iiko OLAP API не умеет делать сложные агрегации (unique count, nested grouping). Проще получить "сырые" строки и сгруппировать самим.

```
OLAP запрос (get_discount_report):
  groupByRowFields:
    Store.Name                    → точка продажи
    Delivery.CustomerCardNumber   → идентификатор гостя (карта/телефон)
    Delivery.CustomerName         → ФИО
    OrderNum                      → номер чека (для подсчёта визитов)
    DishName                      → блюдо (для детализации)
    ItemSaleEventDiscountType     → название скидки

  aggregateFields:
    DishDiscountSumInt            → сумма со скидкой (выручка)
    DiscountSum                   → сумма скидки
```

Второй лёгкий запрос (`get_discount_names`) — только `ItemSaleEventDiscountType` + `DishDiscountSumInt`, вызывается при загрузке страницы для заполнения dropdown. Период — последние 365 дней.

### Бэкенд: два прохода по данным

`POST /api/discount-analyze` делает два прохода по OLAP-строкам:

**Проход 1 — гости** (`discounts_data`):
```
discount_name → card_number → {
    visits: count unique OrderNum,
    sum_with_discount: sum DishDiscountSumInt,
    discount_sum: sum DiscountSum,
    dishes: [список блюд с деталями],
    stores: set(уникальные точки)
}
```

**Проход 2 — сводка по барам** (`stores_summary`):
```
discount_name → store_name → {
    orders_count: count unique OrderNum,
    guests_count: count unique cards,
    sum_with_discount, discount_sum
}
```

JSON-ответ содержит оба среза — фронтенд решает, какой показать.

### Фронтенд: два режима отображения

**Режим "Все бары"** (бар не выбран):
1. ROI-карточки: выручка без скидки, со скидкой, отдали скидками, глубина %, средний чек, чеков, гостей
2. Таблица по барам с колонками глубины — строки кликабельные, переключают на этот бар
3. Сегментация гостей (doughnut-chart + карточки)

**Режим конкретного бара:**
1. Те же ROI-карточки, но по одному бару
2. Таблица гостей — клик открывает модалку с детализацией по блюдам
3. Сегментация гостей

### ROI скидок

Ключевые метрики, которые видит маркетолог:

| Метрика | Формула | Зачем |
|---------|---------|-------|
| Выручка без скидки | `DishDiscountSumInt + DiscountSum` | Сколько бы заплатили без скидки |
| Выручка со скидкой | `DishDiscountSumInt` | Сколько реально заработали |
| Отдали скидками | `DiscountSum` | Стоимость скидки |
| Глубина скидки | `DiscountSum / (DishDiscountSumInt + DiscountSum) * 100%` | Насколько "щедрая" скидка |
| Средний чек | `DishDiscountSumInt / кол-во чеков` | Тратит ли гость больше со скидкой |

### Сегментация гостей

Гости делятся на три группы по количеству визитов:

| Сегмент | Визитов | Цвет | Смысл |
|---------|---------|------|-------|
| Новички | 1 | фиолетовый `#6366f1` | Привлечение — пришёл один раз |
| Возвращающиеся | 2–3 | жёлтый `#f59e0b` | Конверсия — начинает возвращаться |
| Постоянные | 4+ | зелёный `#10b981` | Лояльность — ходит регулярно |

Для каждого сегмента считаются: количество гостей, % от общего, выручка, средний чек.

Визуализация — Chart.js doughnut-chart с cutout 55%.

### Идентификация гостей

Гости идентифицируются по `Delivery.CustomerCardNumber` — номер карты лояльности. В iiko это часто телефон формата `+7XXXXXXXXXX`. Гости без карты группируются как "Без карты".

### UX-флоу

```
Загрузка страницы → лёгкий OLAP → dropdown скидок заполнен
         ↓
[Период] + [Бар] + "Анализировать" → тяжёлый OLAP (5-15 сек)
         ↓
ROI-карточки + таблица + сегментация
         ↓
"Все бары": клик по бару → переключение в режим бара → авто-запрос
"Бар": клик по гостю → модалка с блюдами
```

### OLAP-поля (справка)

| Поле | Назначение | Тип |
|------|-----------|-----|
| `ItemSaleEventDiscountType` | Название скидки/надбавки | group + filter |
| `Delivery.CustomerCardNumber` | Номер карты клиента | group + filter |
| `Delivery.CustomerName` | ФИО клиента | group |
| `OrderNum` | Номер чека (уникальный) | group |
| `DishName` | Блюдо | group |
| `Store.Name` | Точка продажи | group + filter |
| `DishDiscountSumInt` | Сумма со скидкой (выручка) | aggregate |
| `DiscountSum` | Сумма скидки | aggregate |
